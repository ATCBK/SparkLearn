# 学而思｜功能层级与闭环问题总览

## 1. 文档目的

本文档用于总览 SparkLearn 当前功能层级和学习闭环存在的问题，为第二阶段功能树重构、页面归属调整和演示主线设计提供问题依据。本文档是在当前项目已具备较多页面和接口能力，但用户路径、页面层级和数据回流关系不够清晰的背景下形成的分析文档。

本文档聚焦五个问题：

第一，当前一级导航为什么会混淆主链路功能、支撑工具、资源类型和展示页面。

第二，学习画像为什么必须成为个性化学习闭环的起点，而不是普通个人资料页。

第三，学习路径如何从路线展示页升级为承接资源、练习、辅导和报告的主控台。

第四，资源生成、资源中心、资源推送和视频播放为什么必须统一归入资源体系。

第五，学习报告、错题、掌握度和资源推荐之间为什么必须形成可解释的数据回流。

本文档是问题分析文档，不是最终开发方案。后续开发不能直接照本文改代码，而应结合 `学而思｜第二阶段功能层级与学习闭环改造方案.md` 和具体页面设计文档执行。本文指出的一级导航收敛、画像入口强化、资源推送下沉、视频/动画归入资源体系、报告回流和错题补弱推荐等内容仍需在代码中实现或验证。

## 2. 总体结论

当前 SparkLearn 已经具备较多功能页面和后端接口基础，但产品功能层级尚未围绕赛题形成清晰主线。

目前最核心的问题不是功能数量不足，而是：

```text
功能页面很多，但缺少以赛题为核心的主功能层级；
资源、推荐、视频、报告等能力散落在平级导航里；
用户和评委看不到“画像驱动学习”的完整闭环。
```

按照赛题要求，产品主闭环应当明确为：

```text
对话建档 -> 生成学习画像 -> 规划学习路径 -> 推荐/生成资源 -> 学习练习 -> 效果评估 -> 更新画像 -> 重新推荐
```

当前项目中，这条闭环的各个能力大多已经有雏形，但产品层级、页面归属、用户路径和数据回流关系还不够清楚。第二阶段需要先把这些问题沉淀清楚，再进入功能树重构和页面调整。

## 3. 问题一：当前导航把主链路功能和支撑工具混在一起

### 3.1 当前现状

当前侧边栏一级入口是：

```text
首页总览
学习路径
资源中心
练习与错题
资源推送
资源生成
智能辅导
学习报告
视频播放
```

这些入口不是同一层级。

其中：

- `学习路径` 属于主链路功能。
- `资源生成` 属于赛题核心功能。
- `练习与错题` 属于学习效果反馈入口。
- `智能辅导` 属于加分能力和学习过程辅助能力。
- `学习报告` 属于评估与闭环反馈能力。
- `资源中心` 更像资源生成后的产物库。
- `资源推送` 更像学习路径和画像驱动下的推荐结果。
- `视频播放` 更像资源类型之一。
- `首页总览` 更像聚合工作台，而不是独立学习功能。

### 3.2 问题本质

当前导航把以下几类内容放在了同一个层级：

| 类型 | 当前入口 | 判断 |
|---|---|---|
| 主链路功能 | 学习路径、资源生成、练习与错题、学习报告 | 应作为核心能力 |
| 加分增强能力 | 智能辅导 | 可保留高权重，但不能抢主线 |
| 支撑工具 | 资源中心、资源推送 | 应归属资源体系或路径体系 |
| 资源类型 | 视频播放 | 应归属多模态资源 |
| 聚合入口 | 首页总览 | 应承担主线引导 |

一级导航应该展示同一抽象层级的产品能力。当前将主功能、工具、资源类型和产物库并列，导致用户难以判断主线和辅助关系。

### 3.3 造成的影响

1. 用户进入系统后不知道先做什么。
2. 用户容易把资源中心、资源推送、资源生成理解为重复功能。
3. 视频播放被误认为和学习路径、资源生成同等级。
4. 画像建档不在主导航中，赛题第一核心被弱化。
5. 演示时容易变成功能页面巡览，而不是完整学习闭环展示。

## 4. 问题二：画像建档是赛题第一核心，但产品层级存在感偏弱

### 4.1 赛题要求

赛题第一条核心功能是：

```text
对话式学习画像自主构建
```

要求包括：

1. 摒弃传统繁琐表单。
2. 支持自然语言对话。
3. 结合专业、学习目标、学习历史等信息自动抽取特征。
4. 构建不少于 6 个维度的动态学生画像。
5. 支持画像随学随新。

这说明画像不是普通个人资料，而是整个系统智能化的起点。

### 4.2 当前现状

当前项目已有：

- `/onboarding`：问卷式建档。
- `/profile`：个人画像查看和编辑。
- `/api/profile/onboarding`：问卷建档接口。
- `/api/profile/initiate`：对话式建档初始化接口。
- `/api/profile/chat`：对话式建档聊天接口。

从技术基础看，画像能力已经开始建设；但从产品结构看，画像没有成为主闭环起点。

### 4.3 问题本质

当前画像更像：

```text
注册后的表单 + 个人信息页
```

而不是：

```text
先理解学生 -> 再规划路径 -> 再生成资源
```

这会导致用户和评委无法直观看到系统为什么能够个性化。

画像应该承担以下产品职责：

1. 解释系统如何理解学生。
2. 作为学习路径规划的输入。
3. 作为资源生成和推荐的依据。
4. 作为练习评估后的更新对象。
5. 作为整个学习闭环的状态中枢。

### 4.4 造成的影响

1. 赛题第一核心能力不突出。
2. 用户不知道画像会影响什么。
3. 路径、资源、练习、报告之间缺少统一驱动因子。
4. 对话式建档和问卷式建档的关系不清楚。
5. `/profile` 容易被理解成普通账号资料，而不是学习智能体的记忆中心。

## 5. 问题三：学习路径像主线，但和资源、练习、报告的闭环不够显性

### 5.1 当前现状

当前 `/path` 已经有学习路径和知识图谱概念，后端也有：

- `/api/learning-path`
- `/api/learning-path/node-advice`
- `/api/mastery`

这部分很适合作为产品主舞台，因为它天然可以承接画像、资源、练习和评估。

### 5.2 当前缺口

现在学习路径更像“展示学习路线”，还没有在功能树层面明确承接后续动作：

1. 点某个知识点后，应该生成什么资源？
2. 推荐哪些资源？
3. 做哪些练习？
4. 完成后如何更新掌握度？
5. 掌握度变化后路径如何调整？
6. 路径调整后如何重新推荐资源？

### 5.3 问题本质

学习路径目前具备主功能潜力，但闭环连接不够清楚。

理想情况下，学习路径不只是“路线图”，而应该是学习主控台：

```text
当前阶段
-> 当前知识点
-> 推荐学习动作
-> 推荐/生成资源
-> 练习验证
-> 掌握度更新
-> 下一步路径调整
```

当前产品还没有把这个关系明确表达出来。

### 5.4 造成的影响

1. 学习路径容易停留在展示层。
2. 资源生成和路径之间缺少强绑定。
3. 练习结果对路径的反向影响不明显。
4. 学习报告像独立页面，而不是路径调整依据。
5. 用户看不到“动态个性化”的过程。

---

## 6. 问题四：资源相关功能被拆得太散

### 6.1 当前现状

现在资源相关至少有三个一级入口：

```text
资源中心
资源推送
资源生成
```

再加上相关能力：

```text
视频播放
PPT 生成
PDF 下载
资源预览
Coze 链接预览
```

这些其实都属于“个性化学习资源体系”的不同子能力。

### 6.2 问题本质

当前问题不是资源能力太少，而是资源能力被拆成多个平级页面。

按赛题看，资源主功能应该是：

```text
多智能体资源生成
```

其下再包含：

- 推荐资源
- 已生成资源库
- 文档
- PPT
- 思维导图
- 题库
- 拓展阅读
- 视频/动画
- 代码案例
- 预览/下载

也就是说，`资源中心` 和 `资源推送` 不应该抢一级位置。它们更适合作为资源生成或学习路径下的子模块。

### 6.3 资源体系的合理语义

资源相关能力应被统一理解为：

```text
个性化资源体系
├── 多智能体资源生成
├── 基于画像的资源推荐
├── 基于路径节点的资源推荐
├── 已生成资源库
├── 多模态资源预览
├── 资源下载与导出
└── 资源使用反馈
```

当前项目把这些拆成多个入口后，用户无法判断它们之间的关系。

### 6.4 造成的影响

1. 用户不知道应该先进 `资源推送` 还是 `资源生成`。
2. 资源中心像独立仓库，和学习路径关系不清。
3. 资源推送像独立信息流，和画像驱动关系不清。
4. 视频播放像独立功能，和多模态资源关系不清。
5. 资源生成的赛题核心地位被多个辅助入口稀释。

---

## 7. 问题五：智能辅导功能较多，但产品定位不够清楚

### 7.1 当前现状

当前 Tutor 已经有比较丰富的能力：

- 导师角色
- 会话
- 文件上传
- 流式聊天
- 多角色 workshop 事件
- 历史记录
- 评估报告接口也放在 `tutor_eval` 模块里

从功能实现看，Tutor 已经不是简单聊天框，而是具备多角色、多会话和过程事件的基础。

### 7.2 当前混乱点

产品上有两个核心问题：

第一，智能辅导到底是一级主功能，还是嵌入每个学习节点的辅助能力？

第二，多智能体展示到底应该放在 Tutor，还是资源生成，还是两边都有？

### 7.3 赛题视角判断

赛题中智能辅导是可选加分项，不是前三个必做主链路。

因此，智能辅导可以是重要功能，但不能抢过：

```text
画像 -> 路径 -> 资源 -> 评估
```

这条主线。

智能辅导更合理的定位可能是：

1. 学习路径节点上的即时讲解能力。
2. 资源学习过程中的答疑能力。
3. 错题讲解和追问能力。
4. 多智能体协作过程的展示窗口之一。

### 7.4 造成的影响

1. Tutor 容易变成独立聊天产品。
2. 用户不知道什么时候应该进入智能辅导。
3. 多智能体能力可能被分散展示，缺少统一叙事。
4. Tutor 内部能力多，但和主学习闭环关系不明确。
5. 评估报告接口放在 tutor_eval 模块中，也会加重模块边界不清的问题。

## 8. 问题六：学习报告/评估是闭环末端，但目前更像展示页

### 8.1 赛题要求

赛题第五条强调：

```text
学习行为 + 练习测试 + 资源使用反馈
-> 学习效果评估
-> 动态调整资源推送策略和学习计划
```

这说明学习报告不是普通统计页面，而是闭环回流的关键节点。

### 8.2 当前现状

当前项目已有：

- `/report`
- `/api/evaluation/report`
- `/api/evaluation/refresh`
- `learning_events.jsonl`
- `quiz_records.json`
- `resource_usage.jsonl`
- `mastery_records`

数据基础是有的。

### 8.3 问题本质

产品层级上，报告现在更像独立展示页，没有明确告诉用户：

```text
评估结果会反过来影响画像、路径、资源推荐
```

也就是说，“闭环最后一步”存在，但闭环回流不明显。

理想情况下，学习报告应该承担：

1. 汇总学习行为。
2. 解释掌握度变化。
3. 识别薄弱知识点。
4. 更新画像特征。
5. 调整后续学习路径。
6. 改变资源推荐策略。

当前产品还没有在功能层级和页面表达上把这些关系讲清楚。

### 8.4 造成的影响

1. 学习报告容易被理解为静态统计。
2. 练习、资源使用、学习行为的价值没有被充分表达。
3. 用户看不到“评估后系统变聪明”的效果。
4. 赛题要求的“持续优化学习方案”不够明显。
5. 闭环缺少最后的回流说明。

## 9. 问题七：视频播放目前不适合作为一级功能

### 9.1 赛题要求

赛题确实要求多模态资源，并提到视频/动画。

但这并不意味着视频播放应该成为一级功能。视频/动画的产品定位应该是：

```text
多模态学习资源的一种类型
```

而不是独立业务模块。

### 9.2 当前现状

当前导航中有独立一级入口：

```text
视频播放
```

这会让用户误以为产品中存在一个和学习路径、资源生成并列的视频学习模块。

### 9.3 问题本质

如果视频放一级，会改变用户对产品的理解。

用户可能会认为这是：

```text
视频学习平台
```

而不是：

```text
个性化资源生成与学习多智能体系统
```

更合理的定位是：

```text
资源生成 -> 多模态视频/动画资源 -> 预览/播放
```

而不是：

```text
视频播放 与 学习路径 并列
```

### 9.4 造成的影响

1. 视频能力被过度放大。
2. 多模态资源体系被拆散。
3. 资源生成主能力被削弱。
4. 导航层级进一步混乱。
5. 评委可能误解产品重点。

---

## 10. 综合问题归纳

当前问题可以归纳为五类：

### 10.1 功能层级问题

主功能、支撑功能、资源类型和工具能力被并列展示。

### 10.2 产品主线问题

画像驱动学习的主闭环没有被清晰表达。

### 10.3 资源体系问题

资源生成、资源中心、资源推送、视频播放、预览下载等能力没有统一归口。

### 10.4 数据闭环问题

画像、路径、资源、练习、报告之间存在数据基础，但回流关系不显性。

### 10.5 赛题表达问题

赛题要求的三大必做能力和两个加分能力没有被清晰组织成演示主线。

---

## 11. 建议采用的三层问题框架

为后续功能树重构，当前可先按三层重新理解功能：

### 第一层：赛题主链路

```text
学习画像 -> 学习路径 -> 资源生成 -> 练习评估 -> 学习报告
```

这一层必须直接支撑赛题三大必做能力和学习闭环。

### 第二层：加分与增强能力

```text
智能辅导 -> 多智能体过程可视化 -> 多模态资源
```

这一层用于增强创新价值、演示效果和加分项表现。

### 第三层：支撑能力

```text
资源中心
资源推送
视频播放
预览下载
PPT/PDF
语音
管理端
```

这一层不应抢一级主线位置，应作为主链路的支撑能力出现。

---

## 12. 后续处理顺序建议

这些问题都需要解决，但顺序应当是：

1. 先解决一级导航混乱问题。
2. 再明确学习画像作为产品起点。
3. 再重组资源体系，把资源中心、资源推送、视频播放下沉。
4. 再强化学习路径与资源、练习、报告之间的闭环。
5. 再明确智能辅导的产品定位。
6. 最后设计多智能体可视化和演示叙事。

原因是：如果一级功能层级不先定清楚，后续每个功能都会继续出现归属混乱。

---

## 13. 当前阶段结论

SparkLearn 第一阶段已经完成较多页面和接口能力，但第二阶段必须从“页面清单思维”切换到“赛题闭环思维”。

当前最重要的产品判断是：

```text
不是继续增加页面，而是把已有功能重新组织成一条清晰的个性化学习闭环。
```

后续所有功能树、导航、页面合并、入口调整、演示脚本和技术文档，都应围绕这条闭环展开。
