# 学而思｜AI辅助开发规范

> 生成日期：2026-05-06  
> 适用范围：SparkLearn 项目中由 AI 或开发者参与的代码、文档、配置、联调与交付工作

---

## 1. 基准原则

本项目所有开发、文档整理、功能树梳理、接口设计和交付材料，均以第二阶段赛题文件为最高业务基准：

`Dorc/第二阶段项目开发文档/学而思｜赛题.md`

AI 辅助开发时必须先理解赛题目标，再判断功能是否应该保留、合并、下沉或重构。不得只根据当前页面数量或已有代码惯性继续扩展功能。

---

## 2. 中文文档编码规范

### 2.1 统一编码

项目内所有中文 Markdown、JSON、TypeScript、Python、配置说明文件统一使用 UTF-8。

禁止使用以下编码作为项目文件保存格式：

- GBK
- ANSI
- UTF-16
- 带 BOM 的混合编码

### 2.2 PowerShell 读取规范

在 Windows PowerShell 中读取中文 Markdown，必须显式指定 UTF-8：

```powershell
Get-Content -LiteralPath 'D:\Project_building\SparkLearn\Dorc\第二阶段项目开发文档\学而思｜赛题.md' -Encoding UTF8 -Raw
```

禁止直接使用：

```powershell
Get-Content -LiteralPath 'xxx.md' -Raw
```

原因：PowerShell 默认编码可能受系统代码页影响，容易把 UTF-8 中文显示成 `鍩轰簬...` 这类乱码。

### 2.3 AI 读取文档规范

AI 在读取中文文档时必须遵守：

1. 优先使用显式 UTF-8 读取。
2. 如果输出中出现 `鍩`、`绗`、`璧`、`鎬`、`€` 等明显乱码符号，应立即停止基于该内容做判断。
3. 先重新用 UTF-8 读取原文件，再继续分析。
4. 不得把乱码内容复制、总结或沉淀到新文档中。

### 2.4 写入规范

新增或修改中文文档时必须保证：

1. 文件保存为 UTF-8。
2. 中文标点正常显示。
3. 文件名使用中文时保持原样，不做拼音化或乱码化。
4. 写入后至少抽查文档前 10 行，确认没有乱码。

## 3. 文件命名规范

### 3.1 Markdown 命名

阶段文档统一使用：

```text
学而思｜文档主题.md
```

说明：

1. 使用全角竖线 `｜`，不要使用 Windows 不支持的半角 `|`。
2. 阶段归属通过目录区分，不强制在文件名中重复写“第一阶段”“第二阶段”。
3. 文件名必须能直接表达用途，例如 `学而思｜AI辅助开发规范.md`。

### 3.2 目录命名

阶段目录使用：

```text
Dorc/学而思｜第一阶段项目开发文档/
Dorc/第二阶段项目开发文档/
```

若后续新增阶段，按同类格式延续。

## 4. 赛题基准规范

AI 在提出功能建议、功能树、页面结构或技术方案时，必须优先对齐以下赛题要求：

1. 对话式学习画像自主构建。
2. 多智能体协同的资源生成。
3. 个性化学习路径规划与资源推荐。
4. 智能辅导。
5. 学习效果评估。
6. 防幻觉与内容安全过滤。
7. 流式输出或生成进度追踪。
8. 完整可运行的多智能体相关文件、数据集、配置文件与配套文档。

如果当前功能不能服务以上要求，应标记为辅助功能、演示功能或待下沉功能，不应放在一级主功能树中。

## 5. 功能设计规范

### 5.1 功能必须形成闭环

功能设计不能只停留在“页面存在”或“按钮存在”。每个核心功能必须说明：

1. 用户输入是什么。
2. 系统处理过程是什么。
3. 输出结果是什么。
4. 结果是否被保存。
5. 结果如何影响下一步学习。

推荐主闭环：

```text
对话建档 -> 学习路径 -> 资源生成 -> 学习/练习 -> 效果评估 -> 画像更新 -> 路径与资源重新推荐
```

### 5.2 功能层级判断

一级功能只保留能直接支撑赛题主链路的能力：

1. 学习画像
2. 学习路径
3. 资源生成
4. 智能辅导
5. 练习评估
6. 学习报告

以下能力默认不作为一级功能，除非有明确演示需要：

1. 资源预览
2. 下载导出
3. 视频播放
4. 管理端
5. 语音接口
6. 工具配置

这些能力应作为主链路的支撑能力或二级能力。

## 6. 技术开发规范

### 6.1 前端规范

1. 页面层只处理展示与交互。
2. API 请求统一放在 `frontend/src/lib/api/real.ts`。
3. 字段映射不得散落在页面组件中。
4. 页面必须具备 loading、empty、error 三类状态。
5. SSE 页面只消费解析后的事件，不直接处理原始 chunk。

### 6.2 后端规范

1. 路由层只处理 HTTP 入参和响应。
2. 复杂业务逻辑逐步下沉到 service、adapter、orchestrator 或 tool 层。
3. 接口返回统一结构：`success`、`data`、`error`。
4. 长任务使用 SSE，并提供 `done` 或 `error` 终态。
5. 外部工具调用必须有失败降级与可读错误。

### 6.3 字段规范

1. 后端、数据库、接口原始字段使用 snake_case。
2. 前端页面消费 camelCase。
3. 转换逻辑统一放在 API 适配层。
4. 时间字段统一使用 ISO 8601 字符串。

## 7. 敏感信息规范

AI 或开发者不得把以下内容写入 Git：

1. `.env`
2. `.env.local`
3. API Key
4. Secret
5. Token
6. Cookie
7. 私有服务地址和账号密码

允许提交：

```text
.env.example
```

但只能写变量名和示例占位值，不能写真实密钥。

提交前必须检查：

```powershell
git status --short
git ls-files .env
```

若 `.env` 已被 Git 跟踪，应执行：

```powershell
git rm --cached .env
```

## 8. AI 输出校验清单

AI 每次生成或修改文档后，必须检查：

1. 是否以赛题为依据。
2. 是否存在乱码。
3. 是否误写敏感信息。
4. 是否把辅助功能误放到一级功能。
5. 是否说明功能闭环。
6. 是否明确前后端边界。
7. 是否能被开发人员直接执行。

## 9. 本次乱码问题处理结论

`Dorc/第二阶段项目开发文档/学而思｜赛题.md` 文件本身为可正常读取的 UTF-8 中文文档。

此前出现乱码的原因是读取命令未显式指定 UTF-8，导致 PowerShell 按错误编码解释中文内容。后续所有 AI 读取中文 Markdown 时，必须使用 `-Encoding UTF8`，并在发现乱码迹象时立即重新读取，不得基于乱码内容继续分析。
