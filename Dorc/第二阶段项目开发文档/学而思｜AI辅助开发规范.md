# 学而思｜AI辅助开发规范

## 1. 文档目的

本文档用于约束 SparkLearn 第二阶段中由 AI 或开发者参与的文档、代码、配置、接口、联调与交付工作。本文档是在项目已出现中文文档乱码、功能层级混乱、辅助功能误放一级导航等问题后形成的统一开发规范。

本文档聚焦五个问题：

第一，AI 和开发者读取、写入中文文档时如何避免乱码。

第二，新增文档、功能方案和页面设计时应遵守什么命名、结构和状态说明规则。

第三，功能层级如何对齐赛题主闭环，避免把资源中心、资源推送、视频播放等支撑能力误放到一级导航。

第四，前端、后端、字段、SSE、敏感信息和测试验收应遵守哪些工程边界。

第五，后续所有文档如何在开头明确文档目的、当前状态、落地检查和未实现内容，便于团队快速判断是否可以直接执行。

本文档是当前第二阶段 AI 辅助开发和协作的强制规范。规范本身不是功能实现，后续仍需在代码中检查一级导航是否已收敛、AI 精灵助手是否已落地、资源推送是否已下沉、错题补弱推荐是否真实回流。

## 2. 基准原则

本项目所有开发、文档整理、功能树梳理、接口设计和交付材料，均以第二阶段赛题文件为最高业务基准：

`Dorc/第二阶段项目开发文档/学而思｜赛题.md`

AI 辅助开发时必须先理解赛题目标，再判断功能是否应该保留、合并、下沉或重构。不得只根据当前页面数量或已有代码惯性继续扩展功能。

## 3. 中文文档编码规范

### 3.1 统一编码

项目内所有中文 Markdown、JSON、TypeScript、Python、配置说明文件统一使用 UTF-8。

禁止使用以下编码作为项目文件保存格式：

- GBK
- ANSI
- UTF-16
- 带 BOM 的混合编码

### 3.2 PowerShell 读取规范

在 Windows PowerShell 中读取中文 Markdown，必须显式指定 UTF-8：

```powershell
Get-Content -LiteralPath 'D:\Project_building\SparkLearn\Dorc\第二阶段项目开发文档\学而思｜赛题.md' -Encoding UTF8 -Raw
```

禁止直接使用：

```powershell
Get-Content -LiteralPath 'xxx.md' -Raw
```

原因：PowerShell 默认编码可能受系统代码页影响，容易把 UTF-8 中文显示成 `鍩轰簬...` 这类乱码。

### 3.3 AI 读取文档规范

AI 在读取中文文档时必须遵守：

1. 优先使用显式 UTF-8 读取。
2. 如果输出中出现 `鍩`、`绗`、`璧`、`鎬`、`€` 等明显乱码符号，应立即停止基于该内容做判断。
3. 先重新用 UTF-8 读取原文件，再继续分析。
4. 不得把乱码内容复制、总结或沉淀到新文档中。

### 3.4 写入规范

新增或修改中文文档时必须保证：

1. 文件保存为 UTF-8。
2. 中文标点正常显示。
3. 文件名使用中文时保持原样，不做拼音化或乱码化。
4. 写入后至少抽查文档前 10 行，确认没有乱码。

## 6. 功能设计规范

### 6.1 功能必须形成闭环

功能设计不能只停留在“页面存在”或“按钮存在”。每个核心功能必须说明：

1. 用户输入是什么。
2. 系统处理过程是什么。
3. 输出结果是什么。
4. 结果是否被保存。
5. 结果如何影响下一步学习。

推荐主闭环：

```text
对话建档 -> 学习路径 -> 资源生成 -> 学习/练习 -> 效果评估 -> 画像更新 -> 路径与资源重新推荐
```

### 6.2 功能层级判断

一级功能只保留能直接支撑赛题主链路的能力。第二阶段默认一级导航为：

1. 学习工作台
2. 学习画像
3. 学习路径
4. 资源生成
5. 练习评估
6. 学习报告

其中，`学习画像` 是高优先级一级功能，不得弱化为普通个人资料页。它是对话式建档、路径规划、资源推荐、练习生成和评估回流的起点。

`智能辅导` 默认不再作为一级导航。它应改造成全局页面精灵助手，出现在学习画像、学习路径、资源生成、练习评估、学习报告等核心页面中，并根据页面上下文提供解释、追问、讲解和建议。原 `/tutor` 页面可保留为助手会话中心、历史记录和多智能体过程查看页，但不应抢占主链路一级位置。

以下能力默认不作为一级功能，除非有明确演示需要：

1. 资源预览
2. 下载导出
3. 视频播放
4. 管理端
5. 语音接口
6. 工具配置
7. 资源中心
8. 资源推送

这些能力应作为主链路的支撑能力或二级能力。

`资源推送` 不应作为独立一级主功能，但必须保留推荐结果可查看页面或等价入口。推荐资源应出现在学习工作台、学习路径节点、练习错题补弱、学习报告后续建议和资源生成页中。

`资源中心`、`视频播放`、`预览下载` 默认下沉到资源生成体系。`视频播放` 应被理解为多模态资源类型之一，不应与学习路径、资源生成、练习评估并列。

新增页面或入口前必须先判断其归属：

```text
主链路功能
页面精灵助手
资源体系
推荐结果页
支撑工具
资源类型
```

不允许因为已有路由或已有页面就自动放入一级导航。

### 6.3 错题与资源推送闭环

练习错题、薄弱点、掌握度和资源推送必须形成补弱闭环：

```text
练习做错
-> 记录错题
-> 识别相关知识点
-> 更新掌握度和薄弱点
-> 触发补弱资源推荐
-> 用户查看推荐资源或一键生成针对性资源
```

后续涉及练习评估、资源推荐、学习报告的页面或接口改造时，必须说明错题和薄弱点是否影响了后续推荐。如果暂时不能真实回流，也必须在页面或文档中标明当前为待实现能力，不得把静态推荐包装成动态闭环。

## 7. 技术开发规范

### 7.1 前端规范

1. 页面层只处理展示与交互。
2. API 请求统一放在 `frontend/src/lib/api/real.ts`。
3. 字段映射不得散落在页面组件中。
4. 页面必须具备 loading、empty、error 三类状态。
5. SSE 页面只消费解析后的事件，不直接处理原始 chunk。

### 7.2 后端规范

1. 路由层只处理 HTTP 入参和响应。
2. 复杂业务逻辑逐步下沉到 service、adapter、orchestrator 或 tool 层。
3. 接口返回统一结构：`success`、`data`、`error`。
4. 长任务使用 SSE，并提供 `done` 或 `error` 终态。
5. 外部工具调用必须有失败降级与可读错误。

### 7.3 字段规范

1. 后端、数据库、接口原始字段使用 snake_case。
2. 前端页面消费 camelCase。
3. 转换逻辑统一放在 API 适配层。
4. 时间字段统一使用 ISO 8601 字符串。

## 8. 敏感信息规范

AI 或开发者不得把以下内容写入 Git：

1. `.env`
2. `.env.local`
3. API Key
4. Secret
5. Token
6. Cookie
7. 私有服务地址和账号密码

允许提交：

```text
.env.example
```

但只能写变量名和示例占位值，不能写真实密钥。

提交前必须检查：

```powershell
git status --short
git ls-files .env
```

若 `.env` 已被 Git 跟踪，应执行：

```powershell
git rm --cached .env
```

## 9. AI 输出校验清单

AI 每次生成或修改文档后，必须检查：

1. 是否以赛题为依据。
2. 是否存在乱码。
3. 是否误写敏感信息。
4. 是否把辅助功能误放到一级功能。
5. 是否说明功能闭环。
6. 是否明确前后端边界。
7. 是否能被开发人员直接执行。

## 10. 本次乱码问题处理结论

`Dorc/第二阶段项目开发文档/学而思｜赛题.md` 文件本身为可正常读取的 UTF-8 中文文档。

此前出现乱码的原因是读取命令未显式指定 UTF-8，导致 PowerShell 按错误编码解释中文内容。后续所有 AI 读取中文 Markdown 时，必须使用 `-Encoding UTF8`，并在发现乱码迹象时立即重新读取，不得基于乱码内容继续分析。
