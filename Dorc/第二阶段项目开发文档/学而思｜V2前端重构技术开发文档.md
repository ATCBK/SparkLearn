# 学而思｜V2前端重构技术开发文档

## 文档入口说明

文档用途：

本文档用于指导 SparkLearn 第二阶段前端重构工作。目标是以 `ui-prototypes/phase2-complete-ui/学而思 SparkLearn 全页面学习闭环原型.html` 为视觉和交互基准，重构 `frontend` 目录中的 Next.js 页面，并迁移、接入当前后端已有能力，形成可演示、可运行、可回流的学习闭环。

当前状态：

1. 已有高保真 HTML 原型，覆盖学习工作台、学习画像、学习路径、资源生成、资源库、个人知识库、练习评测、错题本、收藏题、学习报表、复习计划和全局 AI 助手。
2. 当前前端为 Next.js App Router 项目，已存在 `/`、`/path`、`/generate`、`/resources`、`/practice`、`/profile`、`/report`、`/feed`、`/tutor`、`/video` 等路由。
3. 当前后端已具备画像、学习路径、资源生成、资源库、练习、推荐、智能辅导、评估报告、语音和管理端等接口基础。
4. 当前前端视觉系统与原型不一致，页面布局、侧边栏分组、顶部面包屑、全局 AI 助手、知识库、错题本独立页等能力仍需改造。

落地检查：

1. 文档命名符合 `学而思｜文档主题.md` 规范。
2. 本文档以第二阶段赛题和 AI 辅助开发规范为基准。
3. 本文档明确区分已实现能力、待改造能力和新增能力。
4. 本文档要求前端 API 统一经过 `frontend/src/lib/api/real.ts`，后端新增复杂逻辑下沉到 service / adapter / orchestrator / tool 层。
5. 本文档不包含 `.env`、API Key、Token、Cookie 等敏感信息。

未实现内容：

1. 本文档是技术实施说明，不代表代码已经完成。
2. 个人知识库真实文本提取、切片、摘要和标签生成尚未落地。
3. 全局 AI 精灵助手尚未接入 `/api/tutor/chat` 的页面上下文 SSE 对话。
4. `/practice/mistakes`、`/practice/favorites`、`/loop`、`/profile/settings` 等目标路由尚未在当前前端中建立。
5. 学习报告的日/周/月维度、报告回流、资源使用统计仍需补强。

## 4. 重构目标

### 4.1 产品目标

本次重构要让前端从“功能页面集合”变成“学习闭环产品”：

```text
学习工作台
-> 学习画像
-> 学习路径
-> 资源生成 / 资源库 / 知识库
-> 练习评测 / 错题本 / 收藏题
-> 学习报表 / 复习计划
-> 更新画像、路径和推荐
```

每个核心页面必须回答五个问题：

1. 用户当前输入是什么。
2. 系统如何处理。
3. 页面输出什么结果。
4. 结果保存在哪里。
5. 结果如何影响下一步学习。

### 4.2 技术目标

1. 保留 Next.js App Router、React、Tailwind CSS、现有 API 适配层和后端路由基础。
2. 以原型 CSS 变量替换当前视觉 token，统一蓝白浅色风格。
3. 重写 shell：侧边栏、顶部栏、主内容宽度、全局 AI 悬浮助手。
4. 按原型逐页改造页面，但不破坏已有后端功能。
5. 新增个人知识库后端接口，并接入资源生成流程。
6. 将资源推送做小，接入首页推荐资源，不保留独立主页面。
7. 将错题本和收藏题从 `/practice` tab 拆成独立路由。

### 7.2 Sidebar

当前 `Sidebar.tsx` 是扁平导航，需要改为原型分组：

1. 品牌区：使用“学而思 SparkLearn”。
2. 分组标题：学习中心、资源与练习、个人知识库、分析与反馈、工具。

### 7.3 Topbar

新增 `Topbar.tsx`：

1. 左侧显示面包屑，例如 `学习中心 / 学习工作台`。
2. 右侧显示用户 chip 和必要图标按钮。
3. 面包屑由 `pathname` 映射，不在页面中重复写。

### 7.4 全局 AI 助手

新增 `AIAssistant.tsx`：

1. 所有核心页面右下角常驻。
2. 根据当前 `pathname` 显示上下文说明和快捷问题。
3. 调用 `POST /api/tutor/chat`，使用 SSE 流式回复。
4. 请求中传入 `page_context`，至少包含：

```json
{
  "pathname": "/practice",
  "page_title": "练习评测",
  "current_node_id": "optional",
  "resource_id": "optional",
  "quiz_id": "optional",
  "report_period": "optional"
```

5. 在 `/tutor` 页面隐藏全局悬浮助手，避免两个聊天入口冲突。
6. AI 回答后的可执行动作包括：更新画像、调整路径、优化生成要求、讲解题目、生成补弱资源。

## 8. 页面改造规格

### 8.1 `/` 学习工作台

目标：回答“今天先学什么，为什么，下一步去哪”。

数据来源：

| 模块 | 接口 |
|---|---|
| 今日任务 | `GET /api/tasks/today` |
| 工作台统计 | `GET /api/dashboard/stats` |
| 推荐资源 | `GET /api/resources/recommendations/list` |
| 活跃数据 | `GET /api/contribution` |
| 薄弱点 | `GET /api/mastery` |

页面结构：

1. `PageHead`：学习中心、今日学习工作台、状态摘要。
2. 英雄决策卡：展示当前优先薄弱点、建议时长、推荐动作。
3. `MetricStrip`：掌握度、待练题、连续学习、路径进度。
4. 左侧资源回顾：最近相关资源。
5. 右侧今日新资源推荐：承接原 `/feed` 的资源推送能力，功能做小。

闭环：

```text
薄弱点和任务
-> 首页推荐资源
-> 用户进入资源库或资源生成
-> 学习和练习结果回写画像、路径、报告
```

### 8.2 `/profile` 学习画像

目标：把当前 `/profile` 从个人信息编辑页改为只读画像展示页。

数据来源：

| 模块 | 接口 |
|---|---|
| 画像基础 | `GET /api/profile` |
| 掌握度 | `GET /api/mastery` |
| 练习统计 | `GET /api/dashboard/stats` |

页面结构：

1. `PageHead`：学习画像、画像版本、当前稳定度、偏好标签。
2. 左侧人物画像卡：头像/插画、浮动标签、画像说明。
3. 右侧四卡网格：当前焦点、学习偏好、目标路径、今日计划。
4. 下方双栏：能力雷达图、最近学习资源。
5. 右上角“编辑画像”跳转 `/profile/settings`。

闭环：

```text
资源学习 / 练习结果 / AI 对话
-> 更新画像字段
-> 影响路径、资源推荐、练习难度
```

### 8.3 `/profile/settings` 个人信息编辑

目标：迁移当前 `/profile` 的表单编辑能力。

要求：

1. 表单字段、保存逻辑复用现有实现。
2. 保存后调用 `PUT /api/profile`。
3. 保存成功后返回 `/profile`，画像展示页显示最新数据。
4. 不在侧边栏主导航中重复展示，只作为底部个人信息入口。

### 8.4 `/path` 个性化学习路径

数据来源：

| 模块 | 接口 |
|---|---|
| 学习路径 | `GET /api/learning-path` |

### 8.5 `/generate` 资源生成

目标：还原原型六步流程。

六步流程：

| 步骤 | 前端行为 | 后端 |
|---|---|---|
| 01 确认上下文 | 展示画像、路径、薄弱点 | `GET /api/profile`、`GET /api/learning-path` |
| 02 选择类型 | 选择文档、PPT、思维导图、练习、阅读、代码等 | 前端状态 |
| 03 配置要求 | 输入 prompt、选择知识库资料 | `GET /api/knowledge/files?status=indexed` |
| 04 开始生成 | 展示多智能体进度 | `POST /api/resources/generate` SSE |
| 05 预览结果 | iframe 或内容预览 | `GET /api/resources/{id}/preview/html` |
| 06 保存学习 | 确认保存、跳资源库或练习 | 生成时已入库，前端确认状态 |

要求：

1. 现有快速生成能力保留为“极简模式”。
2. SSE 解析逻辑封装在 API 层或 hook，不在页面处理原始 chunk。
3. 生成进度必须有 `done` 或 `error` 终态。
4. 选择知识库资料后，请求体增加 `knowledge_file_ids`。

闭环：

```text
路径节点 / 薄弱点 / 知识库资料
-> 多智能体生成资源
-> 保存资源库
-> 学习资源
-> 进入练习
```

### 8.7 `/knowledge` 我的资料库

目标：新增真实个人知识库能力，作为资源生成和 AI 辅导的资料来源。

页面结构：

1. `PageHead`：我的资料库、资料状态摘要。
2. 四列统计：文件数、已整理数、知识片段、引用次数。
3. 左侧上传卡：拖拽上传、上传进度、支持格式说明。
4. 右侧资料表：文件、类型、标签、更新时间、状态、操作。
5. 底部：AI 结构化提取、资料可用于、隐私说明。

新增数据库表：

```sql
CREATE TABLE knowledge_files (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id TEXT NOT NULL,
  filename TEXT NOT NULL,
  stored_path TEXT NOT NULL,
  mime_type TEXT,
  size_bytes INTEGER,
  status TEXT DEFAULT 'pending',
  tags TEXT DEFAULT '[]',
  summary TEXT,
  chunk_count INTEGER DEFAULT 0,
  reference_count INTEGER DEFAULT 0,
  created_at TEXT,
  updated_at TEXT
);

CREATE TABLE knowledge_chunks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_id INTEGER NOT NULL,
  chunk_index INTEGER NOT NULL,
  content TEXT NOT NULL,
  created_at TEXT
);
```

新增接口：

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/knowledge/files` | 文件列表，支持 `status` 和 `search` |
| `POST` | `/api/knowledge/files` | 上传文件 |
| `GET` | `/api/knowledge/files/{id}` | 文件详情 |
| `DELETE` | `/api/knowledge/files/{id}` | 删除文件和切片 |
| `PUT` | `/api/knowledge/files/{id}/index` | 触发整理 |
| `GET` | `/api/knowledge/stats` | 统计 |
| `GET` | `/api/knowledge/chunks?file_id=` | 切片列表 |

整理流程：

```text
读取文件
-> 按 mime_type 提取文本
-> 800 字左右切片
-> 存入 knowledge_chunks
-> 调用 LLM 生成摘要和 3-5 个标签
-> 更新 knowledge_files.status = indexed
```

文本提取要求：

1. PDF 使用 PyMuPDF。
2. DOCX 使用 python-docx。
3. TXT / MD 直接按 UTF-8 读取。
4. 提取失败时状态写为 `failed`，返回可读错误。

与资源生成打通：

```text
/generate 第 03 步选择资料
-> 请求 /api/knowledge/files?status=indexed
-> 用户勾选资料
-> /api/resources/generate 请求携带 knowledge_file_ids
-> 后端将对应 chunks 拼入生成上下文
```

## 9. 前后端边界

### 9.1 前端边界

1. 页面只负责展示、交互和路由跳转。
2. API 请求统一放在 `frontend/src/lib/api/real.ts`。
3. snake_case 到 camelCase 的转换统一在 API 适配层完成。
4. 页面必须具备 loading、empty、error 状态。
5. SSE 页面只消费解析后的事件对象。

### 9.2 后端边界

1. 路由层只处理 HTTP 入参、状态码和统一响应。
2. 知识库整理、文本提取、切片、摘要、标签生成下沉到 service。
3. 资源生成上下文组装下沉到 orchestrator。
4. 所有接口返回统一结构：

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

5. 长任务必须使用 SSE，并提供 `done` 或 `error` 终态。

## 12. 风险与约束

1. 设计系统切换会影响已有页面，应采用 `components/proto/` 渐进迁移。
2. 知识库真实整理依赖 PyMuPDF、python-docx 和 LLM，必须有失败状态和降级文案。
3. AI 助手接入 SSE 后，要避免在页面中散写 chunk 解析逻辑。
4. `/profile` 语义从编辑页改为画像页，必须通过 `/profile/settings` 保留编辑入口。
5. `/feed` 合并后仍要保证推荐资源可见，不能让资源推送能力消失。
6. 原型优先导致“个人知识库”保留为独立入口，应在答辩口径中解释其承担赛题知识库输入职责。

## 13. 最终交付口径

本次前端重构完成后，SparkLearn 第二阶段应能演示以下完整链路：

```text
进入学习工作台

```

该链路需要同时在 UI、接口、数据库记录和文档中保持一致，不能只在静态页面中表达。
