# 学而思｜功能层级问题分析-一级导航混乱

> 生成日期：2026-05-06  
> 适用范围：SparkLearn 第二阶段功能树重构、页面层级梳理、赛题主链路对齐

---

## 1. 问题结论

当前 SparkLearn 的一级导航存在明显层级混乱：主链路功能、资源支撑能力、展示工具和辅助页面被并列放在同一个导航层级中，导致用户无法快速理解产品的核心使用路径，也削弱了项目对赛题要求的表达。

赛题要求的核心不是“页面数量多”，而是构建一个围绕个性化学习的多智能体闭环系统。当前导航虽然覆盖了较多页面，但没有清晰呈现：

```text
对话建档 -> 学习画像 -> 学习路径 -> 资源生成/推荐 -> 学习练习 -> 效果评估 -> 画像更新 -> 再推荐
```

因此，一级导航混乱是第二阶段首先需要处理的问题。它会影响后续所有功能树、演示流程、答辩叙事和页面重构工作。

---

## 2. 当前一级导航现状

当前前端侧边栏一级入口包括：

```text
首页总览
学习路径
资源中心
练习与错题
资源推送
资源生成
智能辅导
学习报告
视频播放
```

从页面数量上看，产品能力较完整；但从功能层级看，这些入口并不是同一抽象层级。

其中：

- `学习路径` 是主链路能力。
- `资源生成` 是赛题核心能力。
- `练习与错题` 是学习效果反馈入口。
- `智能辅导` 是加分能力和学习过程辅助能力。
- `学习报告` 是评估与闭环反馈能力。
- `资源中心` 是资源产物管理能力。
- `资源推送` 是推荐结果展示能力。
- `视频播放` 是某一类多模态资源的消费方式。
- `首页总览` 是聚合看板，不是独立学习功能。

问题在于：这些能力被放在同一层级，用户很难判断哪些是主流程，哪些是支撑功能，哪些只是某个资源类型的展示页。

---

## 3. 与赛题要求的偏差

### 3.1 赛题强调的是系统闭环，不是页面并列

赛题的必做核心功能包括：

1. 对话式学习画像自主构建。
2. 多智能体协同的资源生成。
3. 个性化学习路径规划和资源推送。

可选加分功能包括：

1. 智能辅导。
2. 学习效果评估。

这些功能之间天然存在先后关系：

```text
先了解学生 -> 再规划路径 -> 再生成资源 -> 再学习练习 -> 再评估反馈
```

当前导航把相关能力拆成并列入口，弱化了这种先后关系。

### 3.2 画像建档没有出现在主导航结构中

赛题第一条明确要求“对话式学习画像自主构建”，这是整个系统智能化的起点。

当前项目虽然有：

- `/onboarding`
- `/profile`
- `/api/profile/onboarding`
- `/api/profile/initiate`
- `/api/profile/chat`

但在一级导航中，画像能力并没有作为核心入口出现。用户进入系统后，更容易看到“资源中心、资源生成、视频播放”等功能，却不容易理解：系统应该先通过对话理解学生，再驱动后续学习路径和资源推荐。

这会导致产品叙事偏离赛题重点。

### 3.3 资源能力被拆成多个平级入口

当前资源相关入口至少包括：

```text
资源中心
资源推送
资源生成
视频播放
```

但按赛题逻辑，它们都应该归属于“个性化学习资源体系”：

```text
资源体系
├── 多智能体资源生成
├── 个性化资源推荐
├── 已生成资源管理
├── 多模态资源预览
├── 视频/动画资源
├── PPT/文档/题库/代码案例
└── 下载与导出
```

现在拆成多个平级导航后，会给用户造成两个问题：

1. 不知道应该先进 `资源推送` 还是 `资源生成`。
2. 不知道 `视频播放` 和 `资源中心` 是什么关系。

这会让资源能力显得多而散，而不是形成一个强主线能力。

### 3.4 视频播放被放大为一级功能

赛题确实要求多模态资源，视频/动画可以作为重要加分点。

但视频在产品层级上应该是资源类型之一，而不是与“学习路径、资源生成、学习报告”并列的一级功能。

如果 `视频播放` 保持一级导航，用户会误以为它是独立业务模块，甚至会把产品理解成“视频学习平台”。这与赛题真正强调的“多智能体生成个性化资源”不一致。

更合理的产品语义应该是：

```text
资源生成 -> 多模态视频/动画 -> 预览/播放
```

而不是：

```text
学习路径 / 资源生成 / 视频播放
```

### 3.5 首页总览缺少主线引导作用

`首页总览` 当前承担了数据聚合、今日任务、近期资源、打卡、掌握度等内容。

这些信息本身有价值，但首页没有明显表达“下一步该做什么”。它更像仪表盘，而不是学习闭环入口。

对于赛题演示来说，首页应帮助用户理解：

1. 当前画像状态是什么。
2. 当前学习路径走到哪里。
3. 今天应该学什么。
4. 推荐资源为什么出现。
5. 学完后如何评估并更新画像。

如果首页只展示分散数据，会进一步加重“功能多但主线不清”的问题。

## 4. 具体问题拆解

### 4.1 抽象层级不一致

当前一级导航混合了四类对象：

| 类型 | 当前入口 | 问题 |
|---|---|---|
| 主链路功能 | 学习路径、资源生成、练习与错题、学习报告 | 应保留为核心能力 |
| 资源支撑能力 | 资源中心、资源推送 | 更适合作为资源或路径下的子模块 |
| 资源类型页面 | 视频播放 | 应归属多模态资源，不应独立为一级 |
| 聚合看板 | 首页总览 | 应作为主线入口，而不是普通功能页 |

一级导航应该放同一抽象层级的内容。当前把功能、工具、产物、资源类型都混在一起，是造成理解困难的根本原因。

### 4.2 用户路径不明确

用户进入系统后会看到多个入口，但缺少明确顺序：

```text
我应该先看学习路径？
还是先生成资源？
资源推送和资源中心有什么区别？
视频播放是不是必须先用？
学习报告什么时候看？
智能辅导是全局助手还是某个知识点的助手？
```

这些疑问说明当前导航没有回答“用户下一步该做什么”。

### 4.3 主功能与辅助功能抢位置

`资源中心`、`资源推送`、`视频播放` 都有存在价值，但它们不应与主链路功能抢一级位置。

这些辅助入口占据一级导航后，会稀释真正的核心功能：

- 学习画像
- 学习路径
- 资源生成
- 练习评估
- 学习报告
- 智能辅导

结果是用户看到的不是一条学习闭环，而是一组并列工具。

### 4.4 赛题展示叙事被削弱

比赛评委通常不会长时间自由探索系统，而是通过演示流程理解项目价值。

当前导航如果直接用于演示，会出现叙事断裂：

1. 从哪里体现“对话式画像”？
2. 从哪里体现“画像驱动路径”？
3. 从哪里体现“多智能体协同生成资源”？
4. 从哪里体现“资源学习后反馈到评估”？
5. 从哪里体现“评估结果继续调整推荐”？

这些能力可能在代码和页面中部分存在，但导航结构没有帮助表达它们之间的关系。

## 5. 对当前页面的初步归类

### 5.1 应作为主链路能力的页面

| 页面 | 当前定位 | 问题判断 |
|---|---|---|
| `/onboarding` | 首次建档 | 应升级为画像主链路的一部分 |
| `/profile` | 画像查看与编辑 | 应归属学习画像，不应只是个人信息 |
| `/path` | 学习路径 | 应作为核心主页面 |
| `/generate` | 资源生成 | 应作为资源体系核心页面 |
| `/practice` | 练习与错题 | 应作为练习评估入口 |
| `/report` | 学习报告 | 应作为闭环反馈入口 |
| `/tutor` | 智能辅导 | 可作为一级增强功能，但应明确服务学习过程 |

### 5.2 应下沉为二级或支撑能力的页面

| 页面 | 当前定位 | 建议归属 |
|---|---|---|
| `/resources` | 资源中心 | 归属资源体系 |
| `/feed` | 资源推送 | 归属学习路径或资源体系 |
| `/video` | 视频播放 | 归属多模态资源 |
| `/auth` | 登录注册 | 系统入口，不属于学习功能树 |
| `/` | 首页总览 | 应作为学习工作台，而不是独立功能 |

## 6. 造成混乱的深层原因

### 6.1 早期页面驱动开发导致导航按页面堆叠

项目第一阶段主要先完成页面覆盖，因此导航更像“页面清单”：

```text
有什么页面，就放什么入口
```

这在原型阶段可以接受，但进入第二阶段后，导航应从“页面清单”切换为“功能树和用户闭环”。

### 6.2 资源体系没有统一归口

资源是本赛题最重要的能力之一，但当前资源相关能力分散在多个页面：

- 资源列表
- 推荐资源
- 生成资源
- 视频资源
- 预览下载
- PPT/PDF 生成

缺少一个统一的资源体系定义，导致每个资源能力都像独立功能。

### 6.3 学习画像没有被产品化表达

画像是后续路径、资源、练习、评估的驱动因子，但当前它更多以数据字段和个人资料形式存在。

这会导致用户和评委无法直观看到：

```text
系统为什么知道我适合什么路径？
系统为什么推荐这个资源？
系统为什么生成这种练习？
系统为什么调整学习计划？
```

### 6.4 闭环回流没有在导航中体现

练习、报告、资源使用记录已经有数据基础，但导航没有表达它们会反向影响画像和路径。

如果闭环回流不明显，系统就容易被看成：

```text
路径展示 + 资源生成 + 聊天助手 + 报告页面
```

而不是：

```text
动态个性化学习智能体系统
```

## 7. 当前问题的影响

### 7.1 对用户体验的影响

1. 用户不知道从哪里开始。
2. 用户不知道各功能之间的先后关系。
3. 用户容易把资源中心、资源推送、资源生成理解成重复功能。
4. 用户容易忽略画像建档的重要性。
5. 用户无法感知学习行为会影响后续推荐。

### 7.2 对开发的影响

1. 页面职责不清，后续容易继续堆功能。
2. 新功能不知道应该放在哪个入口下。
3. 资源相关接口和状态容易重复设计。
4. 学习路径、练习、报告之间的数据边界容易继续割裂。
5. 多智能体能力可能被分散到多个页面，难以形成统一展示。

### 7.3 对比赛表达的影响

1. 赛题三大必做能力没有被清晰组织成主流程。
2. 多智能体和个性化闭环的亮点不够突出。
3. 演示视频容易变成页面功能巡览，而不是完整业务闭环展示。
4. PPT 和开发文档需要额外解释页面之间的关系，增加答辩成本。

---

## 8. 问题判断

当前一级导航不是简单的“名字不好听”，而是功能树抽象层级有问题。

它反映出项目第一阶段以页面覆盖为目标，第二阶段必须切换到以赛题闭环为目标。

需要先解决：

```text
哪些能力是一层？
哪些能力是二层？
哪些能力只是资源类型？
哪些能力只是工具和支撑？
```

只有先把这个问题说清楚，后续才能继续处理：

- 资源功能拆散问题
- 画像起点弱化问题
- 学习路径闭环不足问题
- 多智能体展示不足问题

---

## 9. 下一步讨论方向

本文档只沉淀问题，不直接给出最终改法。

下一步应继续围绕以下问题讨论：

1. 一级导航最终保留哪些入口？
2. 首页是否改为“学习工作台”？
3. 学习画像是否应作为独立一级功能？
4. 资源中心、资源推送、资源生成如何归并？
5. 视频播放是否下沉到资源详情页？
6. 智能辅导作为一级功能，还是嵌入路径和练习？
7. 学习报告如何表达“评估反哺推荐”的闭环？
