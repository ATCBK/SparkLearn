## 3. 当前项目代码现状

### 3.1 前端工程

前端主工程位于 `frontend/`，技术栈为：

- Next.js 16.2.4
- React 19.2.4
- TypeScript
- Tailwind CSS 4
- lucide-react
- react-markdown / remark-gfm
- Playwright

主要页面已经覆盖：

- `/auth` 登录页
- `/onboarding` 建档页
- `/` 学习总览
- `/path` 学习路径
- `/resources` 资源中心
- `/practice` 练习与错题
- `/feed` 推荐流
- `/generate` 资源生成
- `/tutor` 智能辅导
- `/report` 学习报告
- `/video` 视频页
- `/profile` 个人信息

当前前端已经切到 `frontend/src/lib/api/real.ts` 作为真实接口适配层，页面层通过 `frontend/src/lib/api/index.ts` 统一消费 API。这个方向正确，第二阶段应继续保持：页面不直接散写 `fetch`，字段映射集中放在 API 适配层。

### 3.2 后端工程

后端主工程位于 `backend/`，技术栈为：

- FastAPI
- Uvicorn
- Pydantic
- SQLite
- JSON / JSONL 文件存储
- httpx
- Playwright
- pytest

后端入口为 `backend/app/main.py`，已注册以下路由模块：

- `profile`：学习画像、问卷建档、对话建档接口基础。
- `learning`：学习路径、任务、掌握度、打卡、视频、首页统计。
- `resources`：资源列表、资源生成、预览、下载、PDF 导出、推荐。
- `quiz`：题目生成、服务端判题、错题本、收藏。
- `tutor_eval`：智能辅导、导师角色、会话、文件上传、评估报告。
- `ppt`：PPT schema 生成。
- `voice_admin`：语音接口占位与管理端数据。

数据层已经具备：

- SQLite 数据库：`backend/data/db/sparklearn.db`
- 单用户数据目录：`backend/data/users/single_user/`
- 画像快照、错题、收藏、资源索引、资源使用记录、学习事件、导师历史、上传文件等 JSON/JSONL 数据。

### 3.3 当前工程成熟度判断

已经完成：

- 前端主要页面闭环。
- 后端基础服务与 CORS。
- SQLite 初始化与默认数据种子。
- 单用户 `single_user` 跑通。
- 练习服务端判题基础。
- 资源生成 SSE 基础。

## 4. 第一阶段核心业务链路沉淀

### 4.1 学习画像链路

第一阶段统一过两种入口：

- 问卷式建档：`POST /api/profile/onboarding`
- 对话式建档：`POST /api/profile/initiate` + `POST /api/profile/chat`

当前更适合第二阶段的策略是双入口并存。问卷式入口用于稳定跑通现有前端，对话式入口用于赛题答辩和亮点演示。

画像字段至少包含：

- `goal`
- `knowledge_level`
- `weak_points`
- `learning_preference`
- `cognitive_style`
- `daily_time`
- `practical_ability`
- `current_stage`

### 4.2 学习路径链路

学习路径已从普通阶段列表升级为知识图谱思路：

- 节点表示知识点。
- 边表示前置依赖。
- 节点状态表示未学、学习中、已掌握。
- 掌握度、错题频次、学习偏好驱动推荐。

第二阶段应把路径推荐从静态规则推进到“画像 + 掌握度 + 事件日志 + 知识库”的动态计算。

### 4.3 练习与评估链路

第一阶段已经明确：真实模式下不能在前端暴露答案。统一方案为：

- `GET /api/quiz` 返回题面。
- `POST /api/quiz/submit` 作为唯一判题入口。
- 错题本、掌握度、学习事件均以后端提交结果为准。

当前代码已具备服务端判题和错题收藏基础。第二阶段应完善题型扩展、LLM 判题兜底、变式题生成和评估报告回流。

### 4.4 Tutor / Generate 流式链路

第一阶段统一了 SSE 协议：

```text
data: {"type":"...","payload":{...}}
```

Tutor 事件应覆盖：

- `message_start`
- `text`
- `sources`
- `warning`
- `done`
- `error`

Generate 事件应覆盖：

- `job_start`
- `progress`
- `preview`
- `done`
- `error`

当前代码已经开始消费 SSE，但第二阶段仍应把事件类型、错误处理、进度状态、页面渲染和测试标准进一步固化。

### 4.5 多智能体与 Tool 化链路

第一阶段 V3 文档确定了多智能体落地方向：

- `IntentAgent`
- `ProfileAgent`
- `PlannerAgent`
- `TutorAgent`
- `ResourceAgent`
- `ReviewerAgent`

资源生成 Tool 清单：

- `doc_tool`
- `ppt_tool`
- `quiz_tool`
- `reading_tool`
- `code_tool`
- `video_tool`

第二阶段要避免“只在文档里多智能体”。每次请求应输出可回放证据：节点、状态、耗时、中间摘要、最终产物、trace_id。

## 5. 功能执行规范

### 5.1 功能边界原则

1. 第二阶段优先补强已有主链路，不新增与赛题无关的零散页面。
2. 功能验收以“页面可操作、接口可调用、数据可持久化、异常可解释”为准。
3. 所有核心功能必须能对应到赛题要求或答辩亮点。
4. 当前阶段继续保持 `single_user` 单用户模式，暂不引入复杂多用户权限体系。
5. 问卷式建档用于稳定跑通现有页面，对话式建档用于赛题演示与答辩证明。
6. 生成类功能必须保留产物记录，不能只在页面临时展示。

### 5.2 核心功能优先级

第二阶段建议按以下顺序推进：

1. 固化后端接口契约与 API contract test。
2. 完成对话式画像建档演示链路。
3. 补齐知识库解析、检索、引用和 RAG 上下文。
4. 完成 Orchestrator、Agent、Tool 执行日志。
5. 强化 Tutor 流式问答与来源引用。
6. 强化资源生成 Tool 化、预览、下载、审核和降级。
7. 完成至少 5 类资源真实生成。
8. 视时间补齐视频/动画类多模态资源。
9. 完成答辩材料、演示脚本和运行手册。

### 5.3 画像建档功能规范

1. 问卷式建档必须能提交到后端并更新画像。
2. 对话式建档必须支持自然语言多轮收集画像信息。
3. 画像字段至少覆盖 `goal`、`knowledge_level`、`weak_points`、`learning_preference`、`cognitive_style`、`daily_time`、`practical_ability`、`current_stage`。
4. 画像更新后应能影响学习路径、资源推荐、练习生成或评估报告中的至少一条链路。

### 5.4 学习路径功能规范

1. 学习路径必须包含知识点、阶段、掌握度和状态。
2. 图谱节点应能展示前置依赖、推荐动作和推荐资源。
3. 错题、掌握度、画像变化应逐步回流到路径推荐。
4. 路径页不应只展示静态 mock，应尽量由后端接口返回结构化数据。

### 5.5 资源生成功能规范

1. 至少稳定支持 5 类资源：文档、PPT、练习题、拓展阅读、代码案例。
2. 每类资源必须有统一状态：`generating`、`completed`、`failed`，可扩展 `review_failed`、`fallback`。
3. 资源生成必须具备进度、预览、下载和失败提示。
4. 外部工具返回链接时，前端不直接消费外链，应优先通过后端代理预览和下载。
5. 资源产物必须写入资源索引，便于资源中心查看和复用。

### 5.6 智能辅导功能规范

1. Tutor 必须支持流式输出，不能只等待整段响应。
2. 辅导回答应尽量包含知识来源、引用知识点或上下文说明。
3. 多角色导师、会话、历史消息、文件上传属于当前已有能力，应继续保留。
4. 多智能体模式需要展示执行过程或日志证据，避免只表现为普通聊天。

### 5.7 练习与评估功能规范

1. 真实练习模式下，题目接口不应向前端暴露正确答案。
2. 判题统一走 `POST /api/quiz/submit`。
3. 判题结果应回写错题本、掌握度和学习事件。
4. 学习报告应逐步由真实学习事件、练习记录、资源使用记录生成。

## 6. 技术执行规范

### 6.1 前端技术规范

1. 页面层只处理交互状态和展示逻辑。
2. API 请求、字段映射、错误归一化放在 `src/lib/api/real.ts`。
3. 新增通用 UI 优先放入 `src/components/ui/`。
4. 新增业务组件优先按页面或模块局部组织，不把复杂逻辑塞进单页大文件。
5. SSE 页面只消费解析后的事件，不直接处理原始 chunk。
6. 页面空态、错误态、加载态必须齐全。

### 6.2 后端技术规范

1. 路由层只做 HTTP 入参、出参和状态码处理。
2. 复杂业务逻辑应逐步下沉到 service / adapter / orchestrator。
3. 所有接口返回统一结构：`success`、`data`、`error`。
4. 长任务统一走 SSE，并提供 `done` 与 `error` 终态。
5. 所有生成类任务必须记录 trace_id、状态、产物引用和错误信息。
6. 与外部平台对接时不得在文档或日志中暴露密钥。

### 6.3 数据与字段规范

1. 后端内部、数据库和接口原始字段使用 snake_case。
2. 前端页面消费 camelCase。
3. 字段转换统一放在 `frontend/src/lib/api/real.ts` 或共享适配层。
4. 时间字段统一返回 ISO 8601 字符串。
5. 资源、练习、画像、评估等核心业务数据必须能持久化，不依赖页面内存。

### 6.4 SSE 技术规范

统一事件格式：

```text
data: {"type":"...","payload":{...}}
```

要求：

1. 每个流式接口必须发送 `done` 或 `error` 终态事件。
2. 资源生成必须发送 `progress` 或同等进度事件。
3. Tutor 必须发送 `text` 增量事件。
4. 前端应统一解析事件，不在页面中重复写 reader 解析逻辑。

### 6.5 外部工具接入规范

1. Coze、讯飞、PPT、TTS 等外部能力统一封装在后端 adapter 或 tool 层。
2. 外部接口失败必须有可读错误和降级方案。
3. 外部产物链接必须进入后端资源索引。
4. 密钥只能放在环境变量或本地配置中，不能写入 Markdown 或代码常量。

### 6.6 测试与验收规范

1. 后端接口新增或修改后必须补 API 测试。
2. SSE 链路必须覆盖成功、失败、超时或降级场景。
3. 前端关键页面至少保留 smoke test。
4. 练习、资源生成、智能辅导、建档四条链路应有端到端验收脚本。
5. 每个第二阶段里程碑结束时必须更新文档中的“已完成 / 未完成 / 风险”。

## 7. Markdown 文件撰写规范

### 7.1 文件命名规范

所有阶段文档统一使用：

`学而思｜文档主题.md`

要求：

1. 使用全角竖线 `｜`，不要使用 Windows 不支持的半角 `|`。
2. 文件名应表达文档主题，不使用 `新建文档`、`最终版2`、`副本`。
3. 阶段文档放入对应阶段目录，不跨目录散放。
4. 涉及日期的上下文留存可使用 `YYYY-MM-DD`，例如 `学而思｜后端联调上下文留存-2026-04-20.md`。

### 7.2 文档结构规范

建议统一结构：

```markdown
# 学而思｜文档标题

> 生成日期：YYYY-MM-DD
> 适用范围：说明本文档覆盖什么
---

## 1. 背景

## 2. 目标

## 3. 当前现状

## 4. 方案或规范

## 5. 执行清单

## 6. 验收标准

## 7. 风险与待确认问题
```

### 7.3 内容撰写规范

1. 一份文档只解决一个主题。
2. 结论前置，先写判断，再写依据。
3. 接口、字段、枚举值必须使用代码块或表格。
4. 所有“必须”“禁止”“建议”要明确适用范围。
5. 不写无法执行的空泛表述，例如“优化体验”“加强智能化”，必须说明怎么验收。
6. 如果方案已被新文档替代，应在旧文档顶部标注“已废弃，参考 xxx”。

### 7.4 接口文档规范

接口文档必须包含：

- 方法与路径
- 请求参数
- 响应结构
- 错误结构
- 字段说明
- 示例
- 前端适配说明
- 测试要点

示例：

```markdown
### POST /api/resources/generate

请求：

```json
{
  "type": "document",
  "prompt": "Python 函数入门"
}
```

SSE 事件：

```text
data: {"type":"progress","payload":{"stage":"内容生成","percent":40}}
```
```

### 7.5 变更记录规范

重要文档底部建议保留：

```markdown
## 变更记录

| 日期 | 修改人 | 变更内容 | 影响范围 |
|---|---|---|---|
| 2026-05-06 | 项目组 | 新增第一阶段总结 | 第二阶段启动 |
```

## 9. 当前风险

1. 赛题强调“对话式画像自主构建”，若只展示问卷建档，答辩说服力不足。
2. 多智能体如果没有执行日志和节点证据，容易被认为是普通单模型调用。
3. 多模态资源如果没有真实产物，资源生成能力会被削弱。
4. 知识库若只停留在原始 Excel/PDF，没有检索与引用，RAG 亮点不足。
5. SSE 链路若缺少错误与超时处理，演示中容易出现不可解释的等待。
6. 部分 README 或早期文档存在过期描述，第二阶段不要直接照旧执行。

## 10. 待确认问题

1. 答辩演示是否必须以“对话式画像建档”为第一入口？
2. 第二阶段是否要求真实生成 MP4 视频，还是允许用 HTML-PPT + TTS + 字幕链路作为阶段成果？
3. 资源生成是否优先接 Coze，还是优先做本地 FastAPI Tool 化兜底？
4. 最终演示重点更偏“多智能体过程可视化”，还是更偏“资源产出质量”？
5. 是否接受继续保持 `single_user` 单用户模式到比赛结束？
6. 是否需要把所有第一阶段旧文档逐个补充状态标记，例如“已实现 / 部分过期 / 二阶段参考”？

## 11. 交接结论

第一阶段已经完成项目从想法到可执行工程的基础建设。当前最重要的不是继续新增零散页面，而是把已有页面背后的真实链路做硬：画像真实、路径真实、资源真实、辅导真实、评估真实、日志可追踪。

第二阶段应围绕“赛题可证明”推进：每个核心能力都要能在代码、接口、页面、日志、文档和演示脚本中找到对应证据。
