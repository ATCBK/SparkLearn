# 学而思｜第二阶段功能层级与学习闭环改造方案

## 1. 文档目的

本文档用于定义 SparkLearn 第二阶段的目标功能层级、学习闭环、页面归属和分阶段执行顺序。本文档是在功能层级问题总览、当前功能层级现状梳理和团队讨论结论基础上形成的整体改造方案。

本文档聚焦五个问题：

第一，第二阶段一级导航应保留哪些主链路功能。

第二，学习画像、学习路径、资源生成、练习评估和学习报告如何组成完整闭环。

第三，智能辅导为什么应改为全局页面精灵助手，而不是独立一级导航。

第四，资源推送、资源中心、视频播放、个人信息等页面应如何下沉但保留可访问能力。

第五，前端、后端、数据回流和演示验收应按什么顺序推进。

本文档是第二阶段整体目标方案，不代表代码已经完成。学习工作台改造、学习画像高优先级入口、AI 精灵助手、推荐资源页下沉、资源库下沉、报告回流、错题补弱推荐等内容仍需按前端和后端实现逐项验证。

## 2. 结论

第二阶段不继续新增零散一级页面，而是把已有页面和能力重组为一条可演示、可解释、可回流的个性化学习闭环。

目标主闭环为：

```text
对话建档
-> 生成学习画像
-> 基于画像规划学习路径
-> 基于路径节点推荐或生成资源
-> 学习资源并完成练习
-> 评估学习效果
-> 更新掌握度与画像
-> 调整路径和资源推荐
```

目标一级导航收敛为：

```text
学习工作台
学习画像
学习路径
资源生成
练习评估
学习报告
```

`智能辅导` 不再作为一级导航，而是改造成全局页面精灵助手，出现在学习画像、学习路径、资源生成、练习评估、学习报告等核心页面上。

`资源推送`、`资源中心`、`视频播放` 保留可访问页面和查看能力，但不占一级导航。它们下沉为资源体系或路径节点中的二级能力。

## 3. 改造目标

### 3.1 产品目标

1. 让用户进入系统后明确知道当前应先建画像，再走路径，再学资源，再练习评估。
2. 让评委能看到画像如何驱动路径、资源、练习和报告。
3. 让资源推送不再是独立信息流，而是来自画像、路径、错题和掌握度的推荐结果。
4. 让智能辅导不再像独立聊天产品，而是贴合每个学习场景的随身助手。
5. 让学习报告不只展示统计，而是成为闭环回流入口。

### 3.2 工程目标

1. 尽量复用当前已有页面和接口，不做大规模推倒重写。
2. 优先调整导航、页面归属、入口关系和页面文案。
3. 后续再逐步补强真实数据回流、推荐理由、资源索引和日志证据。
4. 所有新增和调整都遵守前端 API 适配层、后端 service 分层、SSE 终态事件等既有规范。

## 4. 当前主要问题

当前一级导航包括：

```text
首页总览
学习路径
资源中心
练习与错题
资源推送
资源生成
智能辅导
学习报告
视频播放
```

主要问题是这些入口不在同一抽象层级：

| 当前入口 | 问题 | 改造判断 |
|---|---|---|
| 首页总览 | 像数据卡片页，闭环引导不足 | 改为学习工作台 |
| 学习路径 | 具备主控台潜力，但与资源、练习、报告连接不强 | 保留一级并强化闭环动作 |
| 资源中心 | 是产物库，不是主功能 | 下沉到资源生成 |
| 练习与错题 | 是效果反馈入口 | 保留一级，命名为练习评估 |
| 资源推送 | 是推荐结果，不是独立主功能 | 下沉为推荐资源页和快捷入口 |
| 资源生成 | 是赛题核心能力 | 保留一级 |
| 智能辅导 | 当前像独立聊天产品 | 改为全局精灵助手，页面下沉 |
| 学习报告 | 是闭环回流入口 | 保留一级并强化回流说明 |
| 视频播放 | 是多模态资源类型 | 下沉到资源生成 |

## 5. 目标功能层级

### 5.1 一级导航

```text
学习工作台
学习画像
学习路径
资源生成
练习评估
学习报告
```

### 5.2 全局能力

```text
AI 精灵助手
```

AI 精灵助手是页面级浮动能力，不进入一级导航。它承接原智能辅导能力，并根据当前页面上下文提供不同帮助。

### 5.3 下沉页面

```text
推荐资源页 /feed
资源中心页 /resources
视频预览页 /video
助手会话页 /tutor
个人信息页 /profile
```

这些页面可以保留路由和查看能力，但它们不是一级导航项。它们应通过主链路页面进入。

## 7. 主闭环设计

### 7.1 学习画像到学习路径

```text
用户通过对话或问卷建档
-> 系统生成画像字段
-> 学习路径根据目标、基础、薄弱点、偏好和时间生成阶段规划
```

画像至少应影响以下内容之一：

1. 当前学习阶段。
2. 优先学习知识点。
3. 推荐资源类型。
4. 练习题难度。
5. 学习报告中的建议。

### 7.2 学习路径到资源生成

```text
用户选择路径节点
-> 系统展示节点掌握度和推荐动作
-> 用户查看推荐资源或生成新资源
-> 资源产物写入资源索引
```

路径节点详情中应出现：

1. 推荐学习资源。
2. 一键生成资源。
3. 开始练习。
4. 呼出 AI 精灵助手讲解。

### 7.3 资源学习到练习评估

```text
用户学习资源
-> 系统记录资源使用
-> 用户进入练习
-> 服务端判题
-> 回写错题、掌握度和学习事件
```

资源使用不应只停留在页面展示。至少应记录：

1. 资源 ID。
2. 资源类型。
3. 关联知识点。
4. 使用时间。
5. 是否触发后续练习。

### 7.4 错题到资源推送

错题和资源推送必须建立直接关系：

```text
练习做错
-> 记录错题
-> 识别相关知识点
-> 更新掌握度和薄弱点
-> 触发补弱资源推荐
-> 用户查看推荐资源或一键生成针对性资源
```

练习评估页应包含：

1. 错题列表。
2. 错因分析。
3. 相关知识点。
4. 推荐补弱资源。
5. 一键生成针对性资源。

### 7.5 学习报告到闭环回流

```text
汇总学习事件、练习记录、资源使用记录和掌握度变化
-> 生成学习报告
-> 输出薄弱点和调整建议
-> 更新画像或路径推荐依据
-> 重新生成资源推荐
```

学习报告页必须明确告诉用户：

```text
本次评估将影响后续学习路径和资源推荐。
```

## 8. 各一级功能职责

### 8.1 学习工作台

职责：

1. 展示当前学习状态。
2. 聚合画像摘要、当前路径节点、今日任务、推荐资源和待完成练习。
3. 引导用户进入下一步动作。

不应承担：

1. 独立复杂配置。
2. 与主闭环无关的数据堆叠。

建议模块：

```text
画像摘要
当前路径节点
今日学习动作
推荐资源
待完成练习
最近评估反馈
```

### 8.2 学习画像

职责：

1. 对话式画像建档。
2. 问卷式建档兜底。
3. 画像查看与编辑。
4. 展示画像如何影响路径、资源和练习。

建议入口：

```text
对话建档
画像维度
画像快照
画像更新记录
个人资料
```

### 8.3 学习路径

职责：

1. 展示阶段、知识图谱、知识点状态和掌握度。
2. 承接路径节点上的推荐资源、生成资源、练习和精灵助手讲解。
3. 展示练习和报告对路径的影响。

建议模块：

```text
阶段路径
知识图谱
节点详情
推荐动作
推荐资源
练习入口
路径调整记录
```

### 8.4 资源生成

职责：

1. 多智能体资源生成。
2. 基于画像和路径节点生成个性化资源。
3. 管理已生成资源。
4. 展示推荐资源和多模态资源。

建议模块：

```text
生成新资源
推荐资源
已生成资源
多模态资源
预览下载
生成记录
```

### 8.5 练习评估

职责：

1. 个性化练习。
2. 服务端判题。
3. 错题沉淀。
4. 掌握度回写。
5. 触发补弱资源推荐。

建议模块：

```text
当前练习
判题结果
错题本
薄弱知识点
推荐补弱资源
生成变式练习
```

### 8.6 学习报告

职责：

1. 汇总学习行为。
2. 汇总练习结果。
3. 汇总资源使用。
4. 输出 AI 总结和调整建议。
5. 解释报告如何影响画像、路径和资源推荐。

建议模块：

```text
学习统计
掌握度变化
薄弱点分析
资源使用反馈
AI 总结
下一步调整
```

## 9. AI 精灵助手改造

### 9.1 定位

AI 精灵助手是全局浮动助手，不是一级主功能。

它的产品定位是：

```text
随页面上下文变化的学习辅助智能体
```

### 9.2 页面能力

| 页面 | 精灵助手职责 |
|---|---|
| 学习工作台 | 解释今日任务，建议下一步 |
| 学习画像 | 引导对话建档，解释画像维度 |
| 学习路径 | 解释知识点，说明为什么推荐当前路径 |
| 资源生成 | 辅助完善生成需求，解释生成结果 |
| 练习评估 | 讲解错题，生成变式题，推荐补弱资源 |
| 学习报告 | 解读评估结果，说明后续调整 |

### 9.3 `/tutor` 页面归属

`/tutor` 可保留，但改为二级页面：

```text
助手会话中心
历史记录
多智能体过程查看
文件上传与上下文管理
```

不再作为一级导航入口。

## 10. 资源推送改造

### 10.1 定位

资源推送不是独立一级功能，而是推荐结果查看能力。

推荐来源包括：

1. 学习画像。
2. 当前路径节点。
3. 错题和薄弱点。
4. 掌握度变化。
5. 学习报告调整建议。

### 10.2 出现位置

资源推送应出现在多个场景：

```text
学习工作台：今日推荐
学习路径：节点推荐资源
练习评估：错题补弱资源
学习报告：后续建议资源
资源生成：推荐资源 Tab
```

### 10.3 `/feed` 页面归属

`/feed` 保留为可查看页面：

```text
推荐资源详情页
```

它负责集中展示推荐结果、推荐理由和推荐来源，但不进入一级导航。

## 11. 数据回流关系

目标数据回流如下：

```text
profiles
-> learning_path
-> resources_index
-> resource_usage
-> quiz_records
-> mastery_records
-> evaluation_report
-> profiles / learning_path / recommendations
```

对应关系：

| 数据 | 来源 | 影响 |
|---|---|---|
| 学习画像 | 对话建档、问卷建档、报告回流 | 路径、资源、练习 |
| 学习路径 | 画像、掌握度、学习事件 | 推荐动作、资源生成 |
| 资源索引 | 资源生成 | 资源中心、报告、使用反馈 |
| 资源使用记录 | 资源预览、下载、学习 | 报告、推荐策略 |
| 练习记录 | 判题提交 | 错题、掌握度、报告 |
| 掌握度 | 练习结果、学习事件 | 路径调整、补弱推荐 |
| 学习报告 | 行为、练习、资源、掌握度 | 画像更新、路径调整、资源推荐 |

## 12. 前端改造建议

```text
学习工作台
学习画像
学习路径
资源生成
练习评估
学习报告
```

移出一级导航：

```text
资源中心
```

### 12.2 页面入口改造

1. 学习路径节点详情增加资源、练习、精灵助手入口。
2. 练习评估页增加错题补弱资源入口。
3. 资源生成页增加推荐资源和已生成资源 Tab。
4. 学习报告页增加“本次评估影响了什么”的反馈区域。

### 12.3 全局精灵助手

新增全局浮动入口，读取当前路由和上下文。

前端页面只负责传入上下文，不直接写复杂业务判断：

```text
route
current_node_id
resource_id
quiz_record_id
report_id
profile_snapshot_id
```

具体字段映射和 API 请求继续放在 `frontend/src/lib/api/real.ts`。

## 13. 后端改造建议

### 13.1 推荐能力

建议统一形成推荐服务：

```text
recommendation_service
```

输入：

```text
profile
current_node
mastery_records
quiz_records
resource_usage
```

输出：

```text
recommended_resources
recommended_actions
recommendation_reason
source_type
```

### 13.2 精灵助手能力

原 Tutor 能力保留，但增加页面上下文：

```text
page_context
learning_node
resource_context
quiz_context
report_context
```

路由层仍只处理 HTTP 或 SSE 入参，具体上下文组织下沉到 service 或 orchestrator。

### 13.3 报告回流能力

评估刷新后应输出：

```text
mastery_changes
weak_points
profile_update_suggestions
path_adjustment_suggestions
resource_recommendation_changes
```

这些结果至少应能在页面上展示，后续再逐步写入画像和路径数据。

## 14. 分阶段执行清单

### 14.1 第一阶段：层级和入口调整

1. 调整一级导航。
2. 将 `/feed`、`/resources`、`/video`、`/tutor` 从一级导航移出。
3. 新增或调整学习画像一级入口。
4. 首页文案和结构改为学习工作台。
5. 保留旧路由可直接访问。

验收标准：

```text
用户从一级导航看到的是主闭环，而不是页面清单。
```

### 14.2 第二阶段：页面内闭环入口

1. 学习路径节点增加推荐资源、生成资源、练习、精灵助手入口。
2. 练习评估页增加错题补弱资源区域。
3. 资源生成页整合推荐资源、已生成资源、多模态资源。
4. 学习报告页增加回流说明。

验收标准：

```text
任一核心页面都能引导用户进入闭环下一步。
```

### 14.3 第三阶段：数据回流补强

1. 练习结果回写掌握度和学习事件。
2. 错题触发补弱资源推荐。
3. 资源使用记录进入报告。
4. 报告输出路径和资源调整建议。

验收标准：

```text
用户做完练习后，能看到推荐资源和路径建议发生变化。
```

## 15. 验收标准

### 15.1 产品验收

1. 一级导航只保留主闭环功能。
2. 学习画像作为高优先级一级入口。
3. 智能辅导以全局精灵助手形式出现。
4. 资源推送保留可查看页面，但不作为一级入口。
5. 错题、薄弱点和资源推送之间存在明确入口关系。
6. 学习报告说明评估结果如何影响后续学习。

### 15.2 技术验收

1. 页面不直接散写新的 `fetch`。
2. 新增 API 适配逻辑放在 `frontend/src/lib/api/real.ts`。
3. 后端复杂推荐和评估逻辑下沉到 service。
4. SSE 能力保留 `done` 或 `error` 终态。
5. 核心业务数据能持久化，不依赖页面内存。

### 15.3 演示验收

演示链路应能完整走通：

```text
对话建档
-> 查看画像
-> 进入学习路径
-> 选择知识点
-> 查看推荐资源
-> 生成资源
-> 完成练习
-> 看到错题补弱推荐
-> 查看学习报告
-> 看到后续路径和资源调整建议
```

## 16. 需要写入 AI 开发规范的新规则

以下规则应同步进入 `学而思｜AI辅助开发规范.md`：

1. 一级导航默认只保留 `学习工作台`、`学习画像`、`学习路径`、`资源生成`、`练习评估`、`学习报告`。
2. `学习画像` 是高优先级一级功能，不得弱化为普通个人资料页。
3. `智能辅导` 默认不作为一级导航，应作为全局页面精灵助手存在。
4. `资源推送` 不作为一级主功能，但必须保留推荐结果可查看页面或等价入口。
5. `资源中心`、`视频播放`、`预览下载` 默认下沉到资源体系。
6. 错题、薄弱点、掌握度必须能触发补弱资源推荐或至少展示推荐入口。
7. 新增页面前必须先判断其归属：主链路、页面助手、资源体系、推荐结果、支撑工具。

## 17. 风险与待确认问题

1. 如果只改导航、不补页面内入口，用户仍可能感知不到闭环。
2. 如果资源推送完全隐藏，评委看不到推荐理由，因此必须保留可查看页面。
3. 如果智能辅导只做浮窗但没有页面上下文，会退化成普通聊天。
4. 如果学习报告只展示统计，不输出调整建议，闭环末端仍然不成立。
5. 后续需要确认全局精灵助手的视觉形态、默认展开策略和移动端表现。
