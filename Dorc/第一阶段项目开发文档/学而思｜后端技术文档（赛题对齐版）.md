## 1. 赛题解读结论

先给结论。

如果 SparkLearn 按本方案完整实现，能够覆盖赛题的大部分核心要求，并具备较强的答辩说服力。

但如果要严格满足赛题原文，必须补齐以下 2 个关键点：

1. 必须实现“对话式学习画像自主构建”主链路。
   赛题明确写了“摒弃传统繁琐表单，支持通过自然语言对话自动抽取特征”。当前前端主链路已统一为“问卷式 onboarding 优先”，这适合先落地，但不足以完全匹配赛题表达。要想严格达标，必须补上 `POST /api/profile/initiate` + `POST /api/profile/chat` 的真实对话式建档能力，并在演示中以它作为主链路。

2. 必须真正完成至少 5 种个性化资源生成，其中至少一种要体现多模态资源。
   当前产品设计里已经覆盖文档、PPT、思维导图、练习题、拓展阅读、代码案例，类型数量足够。
   但若视频/动画只是占位页面，没有真实生成或真实接入，则对“多模态资源生成”这一条的说服力会下降。

因此，严格表述为：

- 以当前架构和产品方向，赛题可达成。
- 但要“完全满足赛题原文并拿高分”，必须补齐“对话式画像主链路”与“真实多模态资源产出”。

## 2. 赛题要求逐条对齐

### 2.1 功能要求 1：对话式学习画像自主构建

赛题要求：

- 通过自然语言对话构建画像
- 不少于 6 个维度
- 支持动态更新

SparkLearn 当前能力：

- 已有画像模型字段：
  - `knowledge_level`
  - `goal`
  - `weak_points`
  - `learning_preference`
  - `cognitive_style`
  - `daily_time`
  - `practical_ability`
  - `current_stage`
- 维度数量已超过 6 个
- 数据模型支持版本递增和随学随新

当前风险：

- 第一阶段统一方案里，主链路是问卷式 onboarding，而不是对话式主链路

赛题达标判断：

- 部分满足
- 若实现旧文档中的对话式画像接口，并将其作为正式演示链路，则可完全满足

最终技术决策：

- 后端同时支持两种入口：
  - `POST /api/profile/onboarding`：前端当前问卷版入口，便于先落地
  - `POST /api/profile/initiate` + `POST /api/profile/chat`：赛题答辩主入口，满足对话式画像要求

### 2.2 功能要求 2：多智能体协同的资源生成

赛题要求：

- 必须体现多智能体架构

## 3. 赛题对齐后的后端目标

后端最终不只是“给前端供数”，而是承担 6 个赛题级职责：

1. 学生画像构建与动态更新
2. 多智能体编排
3. 多类型学习资源生成
4. 个性化路径规划与推荐
5. 答疑与评估
6. 防幻觉、安全过滤与进度追踪

因此，后端必须采用“分层 + Agent 编排 + 知识库 + 流式协议”的结构，而不是简单 CRUD 服务。

## 4. 后端总体架构

目录结构采用如下版本：

```text
backend/
├── main.py
├── config.py
├── requirements.txt
├── api/
├── schemas/
├── services/
├── agents/
├── knowledge/
├── storage/
├── llm/
├── voice/
├── utils/
└── tests/
```

分层规则：

- `api/`
  只负责 HTTP 协议与响应包装
- `services/`
  编排业务流程，是后端主控层
- `agents/`
  定义多智能体职责与调用方式
- `knowledge/`
  知识库构建、embedding、检索
- `storage/`
  SQLite + JSON/JSONL + 文件索引封装
- `llm/`
  讯飞星火与流式封装
- `voice/`
  讯飞 ASR/TTS 封装

核心原则：

- API 层不直连 Agent
- Agent 不直连数据库
- 所有状态变更统一经 Service 层

## 5. 赛题对齐后的核心资源范围

为确保满足“至少 5 种个性化资源”要求，后端正式支持以下 6 类资源：

| 类型 | 资源说明 | 产出形式 | 赛题意义 |
|------|------|------|------|
| `document` | 专业讲解文档 | Markdown / HTML | 核心文本资源 |
| `ppt` | 课程讲解 PPT | JSON 结构 + 导出文件 | 满足教学课件场景 |
| `mindmap` | 知识点思维导图 | Markdown 树 / Mermaid / JSON | 满足知识结构化 |
| `quiz` | 练习题集 | JSON | 满足题库场景 |
| `reading` | 拓展阅读 | Markdown | 满足延伸学习 |
| `code` | 代码实操案例 | Markdown + code block / py file | 满足实践材料 |

为冲击更高分，建议补充第 7 类：

| 类型 | 资源说明 | 产出形式 |
|------|------|------|
| `video` | 多模态讲解视频/动画 | 外部生成 URL + metadata |

注意：

- 赛题要求中“多模态”更有利于体现竞争力
- 即便视频能力不能在第一阶段完全自产，也应在架构上预留 `video` 资源类型与生成管线

## 6. 赛题对齐后的画像系统设计

为兼容“当前前端问卷版”与“赛题要求的对话式建档”，画像系统采用双入口策略。

### 6.1 问卷式入口

接口：

- `POST /api/profile/onboarding`

作用：

- 对接当前前端 `/onboarding`
- 快速生成初始画像

### 6.2 对话式入口

接口：

- `POST /api/profile/initiate`
- `POST /api/profile/chat`

作用：

- 满足赛题“自然语言对话建档”的硬要求
- 作为答辩与演示主链路

### 6.3 画像维度

画像至少包含以下 8 个字段：

- `goal`
- `knowledge_level`
- `weak_points`
- `learning_preference`
- `cognitive_style`
- `daily_time`
- `practical_ability`
- `current_stage`

这满足赛题“不少于 6 个维度”的要求。

### 6.4 动态更新机制

画像更新来源包括：

- 初次建档
- 手动修正
- 学习事件累积
- 练习结果反馈
- 评估 Agent 输出

每次更新都：

- 自增 `version`
- 写入 `learning_events.jsonl`

## 7. 多智能体架构

为满足赛题“多智能体协同”要求，后端明确采用以下 Agent 分工：

| Agent | 职责 |
|------|------|
| `ProfileAgent` | 从对话或问卷中抽取学生画像 |
| `PathAgent` | 规划个性化学习路径与阶段顺序 |
| `ResourceAgent` | 生成文档、PPT、阅读、代码等资源 |
| `QuizAgent` | 生成练习题、变式题和题库内容 |
| `EvaluationAgent` | 对学习效果进行分析与快照生成 |
| `TutorAgent` | 结合知识库进行答疑 |
| `ReviewAgent` | 事实校验、安全过滤、低置信拦截 |

编排器采用：

- 串行编排：适合路径规划、评估、审核
- 并行编排：适合资源与练习共同生成

关键编排链路：

### 资源生成链路

`ProfileAgent -> PathAgent -> [ResourceAgent + QuizAgent] -> ReviewAgent`

### 答疑链路

`HybridRetriever -> TutorAgent -> ReviewAgent`

### 评估链路

`EvaluationAgent -> ReviewAgent`

## 10. 数据存储设计

后端采用三层存储：

### 10.1 SQLite

存放强结构化、需要更新和聚合的数据：

- `students`
- `profiles`
- `mastery_records`
- `contribution_days`

### 10.2 JSON / JSONL

存放事件流和灵活结构数据：

- `learning_events.jsonl`
- `quiz_records.json`
- `resource_usage.json`
- `task_progress.json`
- `evaluation_snapshot.json`
- `report_view_model.json`

### 10.3 ChromaDB

存放知识向量：

- 知识库 Excel 条目
- PDF 切片

这种组合的意义：

- SQLite 负责事务与聚合
- JSONL 负责事件追踪与审计
- ChromaDB 负责检索与答疑

## 11. 知识库与课程输入

赛题要求团队自建至少一门完整课程知识库。

SparkLearn 当前以 Python 课程为切入点，知识库输入来自：

- `KnowledgeRepo/Python基础知识库.xlsx`
- `KnowledgeRepo/Python进阶知识库.xlsx`
- `KnowledgeRepo/Python面试知识库.xlsx`
- `KnowledgeRepo/python从入门到实践.pdf`

初始化流程：

1. 解析 Excel
2. 提取知识点与标签
3. 生成 embedding
4. 存入 ChromaDB
5. 解析 PDF 并切片
6. 建立关键词索引

检索采用混合方案：

- 关键词精确匹配
- ChromaDB 语义召回
- 结合画像进行过滤

这为以下能力提供基础：

- 个性化资源生成
- 个性化路径规划
- 智能辅导
- 防幻觉审核参考

## 12. 防幻觉与内容安全机制

赛题明确要求防幻觉与安全过滤，因此后端必须将这部分做成明确模块，而不是“提示词里顺带约束”。

### 12.1 核心模块

- `ReviewAgent`

### 12.2 职责

- 事实校验
- 与知识库上下文交叉检查
- 格式标准化
- 敏感/不合规内容拦截
- 低置信内容标记

### 12.3 审核结果

审核输出字段必须包括：

- `approved`
- `issues`
- `corrected_content`
- `review_status`

### 12.4 降级策略

- 审核失败：资源标记 `review_failed`
- 审核超时：标记 `review_status: "skipped"`
- 低置信：返回模板兜底或提示学生核对教材

这将成为答辩中的重要技术亮点。

## 13. 响应性能与超时策略

赛题要求核心功能响应时间合理，因此后端统一采用超时与降级控制。

### 13.1 Agent 超时

默认超时：

- 大多数 Agent：30 秒
- 资源生成整链路：120 秒

### 13.2 超时降级

- 画像 Agent 超时：使用缓存画像
- 路径规划 Agent 超时：使用默认知识顺序
- 资源生成 Agent 超时：返回 `fallback`
- 练习 Agent 超时：返回预设题库
- 评估 Agent 超时：返回上次快照
- 审核 Agent 超时：放行并标记未审核
- 答疑 Agent 超时：返回统一重试提示

### 13.3 用户体验保障

- 所有长时任务都要有进度事件
- 所有失败都要有可读文案
- 所有慢链路都要有兜底结果或降级结果

## 14. 与当前前端的统一策略

为减少返工，后端赛题对齐版设计仍遵循此前统一规范：

- `quiz` 采用服务端判题
- `tutor/generate` 统一 SSE
- `onboarding` 先保留问卷版前端入口

但为了赛题完全达标，后端需额外完成：

- 对话式画像主链路
- 视频/动画资源的真实多模态接入能力

因此，前后端统一策略是：

### 第一阶段

- 先接通问卷 onboarding
- 先接通 quiz / tutor / generate / evaluation 等主链路

### 第二阶段

- 补对话式 onboarding
- 补 `video` 资源类型
- 在演示中使用对话式画像 + 多模态资源生成作为亮点链路

## 15. 开发优先级

为最快达到赛题可演示状态，推荐后端开发顺序如下：

1. 后端骨架、配置、数据库初始化
2. 画像、路径、任务、资源列表等非流式读接口
3. `quiz` 服务端判题链路
4. 统一 SSE 基础设施
5. `tutor` 流式答疑
6. `generate` 流式资源生成
7. `evaluation` 评估快照与报告
8. `profile` 对话式建档
9. `voice` 语音接口
10. `video` 多模态资源接入
