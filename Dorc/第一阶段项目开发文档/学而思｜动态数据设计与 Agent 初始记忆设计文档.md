## 1. 文档目的

《学而思》是一套面向高校课程学习场景的个性化学习智能体系统。系统围绕赛题要求，需要实现对话式学习画像构建、多智能体协同生成个性化学习资源、个性化学习路径规划与资源推送，以及学习效果评估与动态优化。赛题特别强调学生画像需要“随学随新”，系统还需体现多智能体协同、学习效果评估和动态调整机制。

因此，本设计文档聚焦两个核心问题：

第一，学而思如何构建一套可信、可追踪、可解释的**动态数据机制**，使学生学习过程中的每一次行为都能被系统记录并转化为后续状态变化。
第二，学而思如何为各类 Agent 提供统一、清晰、可扩展的**初始记忆读取机制**，使智能体能够在统一上下文下完成画像更新、路径规划、资源生成、学习评估与推荐决策。

本设计文档的目标不是仅说明数据文件如何存储，而是从系统运行逻辑出发，说明“数据如何流动、如何更新、如何被智能体读取并转化为系统能力”。

***

## 2. 设计总原则

学而思的动态数据与 Agent 记忆设计遵循以下四项总原则。

### 2.1 事实记录与智能理解解耦

系统不将动态数据记录工作完全交给 Agent。

所有学习行为、资源使用行为、练习作答行为和管理操作行为，首先通过统一接口进行标准化记录，形成客观事实数据。

Agent 不负责底层客观数据的采集，而负责在规范化事实基础上完成解释、判断和决策。

### 2.2 事件驱动状态演化

系统中的动态变化不是“哪里需要变就直接改哪里”，而是遵循统一链路：

**行为发生 → 事件记录 → 状态更新 → 评估判断 → 页面渲染**

这意味着系统先记录过程，再更新状态，再形成结论，最终为学习端和管理端输出可展示结果。

### 2.3 Agent 读取的是“记忆上下文”，不是离散文件

底层虽然以状态文件、事件文件和索引文件承载数据，但 Agent 不应直接无序扫描文件。

系统通过统一的“记忆加载层”读取与任务相关的用户状态和学习过程数据，组装成结构化记忆上下文，再提供给不同类型的 Agent 使用。

### 2.4 动态数据必须服务于闭环优化

动态数据不是为了堆积日志，而是为了支撑系统真正做到：

* 画像随学随新

* 路径随评估调整

* 资源随短板变化

* 报告随行为生成

只有当动态数据能驱动学习画像、资源推荐、学习效果分析和管理端操作时，它才真正成为系统能力的一部分。

***

## 3. 动态数据设计目标

学而思的动态数据设计要达成五个目标。

### 3.1 准确记录学生发生过什么

系统必须能够客观记录学生在平台上的关键行为，包括：

* 首次建档对话

* 学习资源打开与完成

* 练习提交与答题结果

* 学习任务推进

* 学习报告生成

* 管理员或教师干预

### 3.2 真实反映学生当前状态

系统必须将客观行为进一步转化为学生当前状态，包括：

* 当前学习画像

* 当前知识点掌握度

* 当前学习阶段

* 当前任务进度

* 当前学习偏好趋势

### 3.3 让系统能解释“为什么”

系统不只是要给出结论，还要能解释：

* 为什么当前将“函数”判断为薄弱点

* 为什么推荐某类资源

* 为什么学习阶段被推进或回退

* 为什么学习效果报告给出当前建议

### 3.4 支撑多智能体协同

不同 Agent 读取的数据口径必须一致，避免出现：

* 路径 Agent 看到的状态和资源 Agent 不一致

* 评估 Agent 的结论与画像 Agent 的更新依据冲突

* 管理端看到的信息与学生端展示结果不一致

### 3.5 支撑网页与多模态结果渲染

动态数据最终必须能够支撑：

* 学生个人画像页

* 学习路径页

* 历史资源页

* 学习效果分析网页

* 管理端学生详情页与操作页

# 第一部分：动态数据设计

## 4. 动态数据的定义与范围

在学而思中，动态数据不是简单指“会变化的数据”，而是指：

**在学生学习、作答、生成资源、接受评估和被管理干预的过程中，会不断被记录、更新、解释并驱动系统后续决策的数据。**

动态数据主要分为五类。

### 4.1 画像动态数据

用于描述学生整体学习状态的变化，包括：

* 学习目标变化

* 当前薄弱点变化

* 学习偏好变化

* 学习阶段变化

* 实操能力判断变化

### 4.2 掌握度动态数据

用于描述学生对不同知识点掌握程度的变化，包括：

* 某知识点掌握分数提升或下降

* 知识点从“薄弱”变为“已掌握”

* 阶段性章节掌握情况变化

### 4.3 学习过程动态数据

用于记录学生学习过程中的行为变化，包括：

* 打开资源

* 完成资源

* 做题提交

* 错题分布

* 连续学习天数

* 阶段任务完成度

### 4.4 评估动态数据

用于描述系统对学生学习效果的阶段性判断变化，包括：

* 总学习时长

* 正确率趋势

* 完成率变化

* 知识掌握趋势

* 当前阶段评估结论

### 4.5 管理动态数据

用于记录教师或管理员对学生、资源和任务的干预，包括：

* 修改学习目标

* 标记重点关注

* 重新生成资源

* 重新生成报告

* 调整当前阶段

***

## 5. 动态数据总体机制

学而思采用“**接口驱动记录、规则驱动更新、智能体驱动解读**”的动态数据实现机制。

它不是把所有变化都交给 Agent 来写，而是分成三个层次。

### 5.1 接口记录层

负责将平台中发生的客观行为标准化记录成事件。

这一层只解决一个问题：

**到底发生了什么。**

它不做复杂推理，只保证：

* 事件被准确记录

* 字段结构统一

* 时间顺序清晰

* 数据来源明确

### 5.2 状态更新层

负责根据已记录事件，更新系统中的客观状态。

这一层解决的问题是：

**发生这些行为后，系统中的哪些状态需要变化。**

例如：

* 完成资源后，更新资源完成率

* 提交练习后，更新答题记录

* 完成任务后，更新任务进度

* 多次学习同类资源后，更新资源使用统计

这一层优先采用确定性规则，而不是调用 Agent 做基础事实落库。

### 5.3 智能解读层

负责在事件和状态基础上，由 Agent 完成解释、判断和策略输出。

这一层解决的问题是：

**这些变化意味着什么，接下来应该怎么办。**

例如：

* 近三次函数题作答不佳，是否应将“函数”提升为核心薄弱点

* 最近学习行为是否表明该学生更偏好代码案例

* 当前是否适合推进到下一阶段

* 是否应该重新推荐补强资源

***

## 6. 动态数据的实现主线

学而思的动态数据实现遵循统一主线：

### 6.1 行为发生

学生或管理员在平台内完成某项操作，例如：

* 首次建档对话完成

* 打开并完成一份讲义

* 提交一套练习

* 重新生成学习报告

* 管理员调整目标

### 6.2 事件记录

系统通过统一事件接口将该行为记录下来，形成规范化事件数据。

### 6.3 状态更新

系统根据事件类型更新画像、掌握度、任务进度、资源使用情况等客观状态。

### 6.4 智能解读

Agent 在统一上下文下读取这些事件和状态，对当前学习情况进行解释与判断。

### 6.5 页面渲染

系统将判断结果转换为视图模型，用于渲染学生页面、学习效果分析页和管理端页面。

这一主线决定了系统不是“直接改结果”，而是先记录过程，再输出结论。这是系统可信性和可解释性的关键。

***

## 7. 标准事件接口设计

为了保证动态数据记录的规范性，学而思将所有关键行为统一抽象为事件，并通过统一事件接口落地。

每条事件建议至少包含以下字段：

* `event_id`：事件唯一标识

* `event_type`：事件类型

* `user_id`：用户标识

* `course_id`：课程标识

* `timestamp`：事件发生时间

* `source`：事件来源

* `payload`：事件内容载荷

## 8. 客观状态更新设计

事件记录完成后，系统进入客观状态更新阶段。这里的关键思想是：

**能规则化更新的状态，不依赖 Agent。**

### 8.1 资源完成率更新

当发生 `resource_completed` 事件时：

* 资源使用记录更新

* 对应学习时长累计

* 对应任务如绑定当前节点，则同步推进任务进度

### 8.2 练习记录更新

当发生 `quiz_submitted` 或 `question_answered` 事件时：

* 写入练习记录

* 统计正确率

* 标记相关知识点的错误次数

* 更新该阶段的作答情况摘要

### 8.3 任务进度更新

当发生 `task_completed` 事件时：

* 当前节点状态改为已完成

* 阶段完成率更新

* 若已完成条件满足，标记可进入下一阶段

### 8.4 资源使用统计更新

当资源被重复访问、完成或收藏时：

* 更新对应资源使用情况

* 统计用户偏好倾向

* 为后续推荐和偏好识别提供依据

### 8.5 行为历史追加

无论是否触发状态变化，学习过程类事件都需要被追加保存，用于后续评估和追踪。

## 9. 需要 Agent 解读的动态变化

并不是所有动态变化都能通过规则直接完成。学而思将以下变化交由 Agent 进行解读。

### 9.1 薄弱点识别

例如：

* 同一知识点连续多次作答失败

* 某类资源学习后效果仍无明显提升

* 练习与任务表现共同表明某项能力薄弱

这时需要 Agent 结合事件、作答记录和掌握度状态进行综合判断。

### 9.2 学习偏好识别

例如：

* 学生连续选择代码案例而非长篇讲义

* 视频资源完成率显著高于文字资源

* 图解资源重复访问率更高

这类结论不适合用单一规则武断下结论，更适合由 Agent 在一定周期内做趋势判断。

### 9.3 学习阶段判断

例如：

* 是否可以从“基础巩固”推进到“进阶应用”

* 是否应从“项目实战”回退到“概念补强”

* 当前阶段任务完成度与掌握度是否匹配

### 9.4 推荐策略调整

例如：

* 当前更适合推习题、图解、讲义还是代码案例

* 是否需要立即推送补强资源

* 是否应该压缩任务颗粒度

### 9.5 学习效果结论生成

例如：

* 当前学习是否真正产生提升

* 哪些进步值得强调

* 哪些薄弱点仍需重点关注

* 下一步最合理的学习建议是什么

动态数据最终必须服务于页面展示，而不是停留在系统内部。

### 10.1 学生端页面

动态数据支撑以下模块：

* 当前薄弱点

管理端通过索引文件和视图数据查看：

* 学生当前状态

* 历史资源

* 最新学习报告

* 可执行的管理操作

# 第二部分：Agent 初始记忆设计

## 12. Agent 初始记忆设计目标

学而思的多智能体系统需要围绕统一学生状态运行，因此必须解决一个核心问题：

**每个 Agent 在开始执行任务前，如何读取正确、完整、适度的用户记忆。**

这里的“记忆”不是指某一个文件，而是指：

**系统基于学生状态文件、学习过程文件和任务信息，为当前任务组装出来的一份结构化上下文。**

Agent 初始记忆设计的目标有四个：

### 12.1 保证不同 Agent 读取口径一致

避免路径规划 Agent、资源生成 Agent、评估 Agent 对同一个学生状态产生不同理解。

### 12.2 保证不同 Agent 只读取自己需要的记忆切片

避免每个 Agent 都读取所有信息，导致冗余、混乱和成本上升。

### 12.3 保证底层存储可替换

当前可由文件承载，但未来即使升级到数据库，Agent 层接口也不需要被推翻重写。

### 12.4 保证 Agent 初始上下文可解释、可调试

能够明确说明：

* 某个 Agent 启动时到底看到了哪些用户状态

* 为什么生成了当前结果

* 后续如何复现其决策过程

***

## 13. Agent 读取记忆的基本原则

### 13.1 Agent 不直接无序读取底层文件

虽然底层状态由 JSON 和索引文件承载，但不建议每个 Agent 直接到目录中自行查找和拼接。

这会导致：

* 读取口径不一致

* 状态边界不清晰

* 后期迁移成本高

### 13.2 统一通过记忆加载层读取

系统应提供统一的记忆加载机制，由其完成：

* 读取相关状态文件

* 按任务类型抽取关键字段

* 组装成标准化记忆上下文

* 提供给对应 Agent

### 13.3 初始记忆按任务类型裁剪

不同 Agent 读取的不是“同一份全量记忆”，而是“面向该任务的任务上下文”。

### 13.4 初始记忆优先包含摘要信息

Agent 优先读取结构化摘要，而不是原始大段流水。

只有在需要时，才补充最近事件、关键资源或特定记录。

***

## 14. Agent 初始记忆总体结构

学而思建议将 Agent 初始记忆统一组织为以下结构：

各部分含义如下。

### 14.1 `user_identity`

用于说明用户基础背景，例如：

* 用户 ID

* 课程

* 专业

* 年级

### 14.2 `learning_profile`

用于说明当前画像核心字段，例如：

* 学习目标

* 知识基础

* 当前薄弱点

* 学习偏好

* 当前阶段

### 14.3 `mastery_summary`

用于说明当前关键知识点掌握情况，例如：

* 函数：0.41

* 面向对象：0.28

### 14.4 `task_context`

用于说明当前任务节点，例如：

* 当前学习阶段

* 当前节点

* 当前待完成任务

* 当前任务完成率

### 14.5 `recent_activity`

用于说明最近发生过的重要学习行为，例如：

* 最近完成了哪些资源

* 最近提交了哪些练习

* 最近出现了哪些错误知识点

### 14.6 `resource_context`

用于说明与当前任务相关的资源上下文，例如：

* 已生成过哪些同类资源

* 当前资源生成需求

* 当前知识点关联资源类型

### 14.7 `evaluation_context`

用于说明最近评估结论，例如：

* 最近一次学习效果结论

* 最近重点关注的薄弱点

* 最近阶段建议

## 15. 面向不同 Agent 的初始记忆切片设计

不同 Agent 的职责不同，因此其初始记忆也应不同。

## 15.1 画像 Agent 的初始记忆

画像 Agent 的任务是理解学生当前状态并更新画像，因此它重点读取：

* 当前 `profile.json`

* 最近对话内容

* 最近行为摘要

* 最近一次评估结论

* 画像历史变更摘要

它关注的是：

* 画像是否需要修正

* 薄弱点是否变化

* 目标和偏好是否发生迁移

## 15.2 路径规划 Agent 的初始记忆

路径 Agent 的任务是决定下一步学什么，因此它重点读取：

* 当前学习目标

* 当前学习阶段

* 当前知识掌握度

* 当前任务完成情况

* 最近阶段评估结果

它关注的是：

* 当前应该优先补哪部分

* 是推进、维持还是回退

* 下一阶段节点如何排列

## 15.3 资源生成 Agent 的初始记忆

资源 Agent 的任务是生成个性化学习资源，因此它重点读取：

* 当前薄弱点

* 当前知识点掌握度

* 学习偏好

* 当前阶段

* 当前任务目标

* 相关章节上下文

它关注的是：

* 生成何种资源

* 资源难度如何设定

* 内容形式应偏文字、图解还是代码案例

* 是否应生成补强型还是进阶型资源

## 15.4 评估 Agent 的初始记忆

评估 Agent 的任务是输出学习效果分析，因此它重点读取：

* 学习事件摘要

* 练习结果

* 资源使用情况

* 掌握度变化

* 阶段任务完成情况

它关注的是：

* 当前学习是否有效

* 哪些知识点有提升

* 哪些短板仍需关注

* 下一阶段建议是什么

## 15.5 推荐 Agent 的初始记忆

推荐 Agent 的任务是推送下一批学习内容，因此它重点读取：

* 当前画像

* 当前薄弱点

* 当前阶段

* 最近使用过的资源

* 资源偏好趋势

* 最近推荐接受情况

它关注的是：

* 现在推什么最合适

* 推送形式是否需要调整

* 是否应避免重复推荐

***

## 16. 记忆加载层设计思路

为了让 Agent 初始记忆组织清晰，学而思建议引入统一的“记忆加载层”。

### 16.1 记忆加载层的职责

* 读取用户当前状态文件

* 读取最近过程事件

* 按任务类型裁剪上下文

* 输出结构化记忆对象

### 16.2 记忆加载层的价值

其价值不在于增加技术复杂度，而在于形成职责边界：

* 底层文件负责承载状态

* 记忆加载层负责组装上下文

* Agent 负责解释与决策

这样设计后，底层即使从文件升级为数据库，也只需替换加载层，而不必重写各类 Agent。

### 16.3 记忆加载层的输出原则

* 输出尽量结构化

* 输出优先摘要而不是流水全文

* 输出只包含当前任务所需字段

* 输出内容需可追踪、可调试、可复现

***

## 17. Agent 初始记忆与动态数据的关系

动态数据和 Agent 记忆不是两套独立体系，而是前后衔接关系。

### 17.1 动态数据是 Agent 记忆的来源

学生的一切学习行为、状态变化和评估结果，首先以事件、状态和索引的形式被记录下来，随后才被记忆加载层组装成 Agent 初始记忆。

### 17.2 Agent 记忆是动态数据的“任务化表达”

同样一组底层数据，在不同任务中会被组织成不同上下文。

这意味着 Agent 记忆不是对原始文件的简单复制，而是面向当前任务的上下文表达。

### 17.3 Agent 解读结果会反哺动态数据

Agent 在完成判断后，会进一步触发：

* 画像修正

* 阶段调整

* 推荐结果更新

* 报告生成

这些结果最终仍然会沉淀回状态文件、评估文件和索引文件中，形成新的动态数据。

***

## 18. 典型链路说明

为了便于评审理解，学而思可以用以下典型链路说明动态数据与 Agent 记忆的协同。

### 18.1 学生完成首次建档

* 学生通过对话完成建档

* 系统记录 `profile_initialized` 事件

* 画像 Agent 读取建档信息并生成初始画像

* 写入 `profile.json`

* 路径 Agent 读取初始记忆并生成首轮学习路径

### 18.2 学生完成一次资源学习

* 记录 `resource_opened` 和 `resource_completed` 事件

* 更新资源使用和学习时长

* 若资源绑定当前节点，则更新任务进度

* 推荐 Agent 读取更新后的记忆，调整后续资源推荐

### 18.3 学生提交一套练习

* 记录 `quiz_submitted` 事件

* 更新 `quiz_records.json`

* 更新掌握度与错误知识点摘要

* 评估 Agent 读取相关记忆，判断是否应提升薄弱点权重

* 路径 Agent 决定是否需要调整当前学习路径

### 18.4 管理员重生成学习报告

* 记录 `admin_action` 事件

* 触发评估模块重新汇总最新学习数据

* 生成新的 `evaluation_snapshot.json` 与 `report_view_model.json`

* 学习效果分析网页重新渲染

***

## 19. 设计总结

学而思的动态数据设计与 Agent 初始记忆设计，不是简单的数据存储方案，而是一套围绕个性化学习闭环建立的系统化机制。

其核心思想可以概括为三句话：

### 第一，动态数据不是让 Agent 去“记流水账”

系统先通过接口标准化记录客观事实，再通过规则更新基础状态，最后由 Agent 在规范化数据基础上进行智能解读。

### 第二，Agent 读取的不是离散文件，而是结构化记忆上下文

底层文件只承担数据承载职能，真正提供给 Agent 使用的是面向任务裁剪后的统一记忆对象。

### 第三，行为、状态、评估、页面形成完整闭环

学生每一次学习行为都会被记录、被计算、被解释，并最终影响画像、路径、资源推荐和学习报告，从而实现真正意义上的动态个性化学习。

***

## 20. 总结性表述

学而思采用“接口驱动记录、规则驱动更新、智能体驱动解读”的动态数据机制。系统将学生建档、资源学习、练习作答、任务推进和管理干预等行为统一抽象为标准化事件，通过接口完成客观记录，并由状态更新模块对学习时长、资源完成率、任务进度、作答结果和掌握度等客观状态进行维护；在此基础上，智能体通过统一记忆加载层读取任务所需的学生上下文，对薄弱点、学习偏好、阶段推进、资源推荐和学习效果进行解释与推理，最终形成学习端、学习效果分析页和管理端可直接使用的视图数据。该设计既保证了系统运行的可信性、可解释性和一致性，也为多智能体协同与后续平台扩展提供了清晰的数据基础。
