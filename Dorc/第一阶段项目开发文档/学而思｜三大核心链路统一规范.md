# SparkLearn 三大核心链路统一规范

> 适用范围：`quiz`、`tutor/generate`、`onboarding`
> 目的：在后续后端开发前，先把最容易导致联调返工的协议、字段、流式方案、页面改造边界和降级策略一次性统一。
> 生效原则：后续实现以本文件为执行规范，若与旧文档或当前前端 mock 冲突，以本文件为准。

---

## 1. 为什么先统一这三块

这三条链路是当前项目里最容易“写完又推翻”的部分。

原因不是代码复杂，而是它们都同时踩中了下面几类风险：

- 前端现状和后端设计文档不一致
- 涉及流式协议，不能只看普通 REST 思维
- 涉及状态机，不是单个请求一来一回就结束
- 一旦字段或协议定错，会波及页面、API、存储和测试四层

本文件的目标就是先消灭这些隐患，再开始正式开发。

---

## 2. 总体统一原则

### 2.1 协议分层原则

- 页面组件不直接面向后端原始响应。
- 前端真实请求统一经 `frontend/src/lib/api/real.ts`。
- 后端对外保持 snake_case。
- 前端页面继续消费 camelCase。
- 字段映射和事件映射统一在 API 适配层完成。

### 2.2 流式统一原则

`onboarding`、`tutor`、`generate` 三条链路统一使用 SSE。

禁止：

- 一部分用普通 `fetch`，一部分用伪流式定时器模拟
- 页面自己解析原始字节块
- 页面自己猜测流结束时机

必须：

- 统一定义事件格式
- 统一定义 `done` / `error` / `progress` / `message` 类事件
- 前端只消费“已解析事件”

### 2.3 第一阶段范围原则

第一阶段全部按单用户 `single_user` 跑通，不提前引入登录态复杂度。

这意味着：

- 后端内部保留 `user_id`
- 第一阶段从配置或依赖里固定注入 `single_user`
- 不在这三条链路里掺入 JWT、session storage、复杂权限

### 2.4 页面最小改造原则

只有以下场景允许页面结构调整：

- `quiz` 从前端判题改为服务端判题
- `tutor/generate` 从伪流式改为真实 SSE
- `onboarding` 从当前卡片问卷接入真正的建档提交链路

不允许因为后端字段命名不同，就在页面里散改字段名。

## 3. 链路一：Quiz 统一规范

## 3.1 当前隐患

当前前端 `QuizQuestion` 类型中包含：

- `correctAnswer`
- `explanation`

并且页面直接在浏览器内判题。

这会带来 5 个问题：

1. 正确答案提前暴露给前端，真实练习场景不成立。
2. 判题逻辑分散在页面，后续掌握度更新和错题本统计容易不一致。
3. 题型一旦扩展，前端和后端会出现双份判题逻辑。
4. 错题本与掌握度依赖后端记录，但当前前端是“本地结果”。
5. 当前 mock 数据结构会误导后端实现。

## 3.2 统一决策

从现在开始，`quiz` 采用“服务端判题”方案。

### 决策 A

`GET /api/quiz` 不返回答案。

### 决策 B

`POST /api/quiz/submit` 是唯一判题入口。

### 决策 C

错题本、掌握度、学习事件、打卡全部以后端提交结果为准。

### 决策 D

前端保留当前练习页面结构，但改掉本地判题逻辑。

## 3.3 统一后的题目数据结构

后端原始响应：

```json
{
  "success": true,
  "data": [
    {
      "id": "q1",
      "type": "single",
      "content": "以下哪个是 Python 的可变数据类型？",
      "options": ["tuple", "list", "string", "int"],
      "knowledge_point_id": "1.2",
      "knowledge_point_name": "数据类型"
    }
  ]
}
```

前端适配后类型应改为：

```ts
export interface QuizQuestion {
  id: string
  type: 'single' | 'multiple' | 'fill_blank'
  content: string
  options?: string[]
  knowledgePointId?: string
  knowledgePointName?: string
}
```

删除：

- `correctAnswer`
- `explanation`

## 3.4 提交接口统一结构

请求：

```json
{
  "quiz_id": "q1",
  "answer": "list"
}
```

多选题：

```json
{
  "quiz_id": "q3",
  "answer": ["list", "dict"]
}
```

响应：

```json
{
  "success": true,
  "data": {
    "quiz_id": "q1",
    "correct": true,
    "correct_answer": "list",
    "explanation": "list 是可变类型，可以修改其元素。",
    "knowledge_point_id": "1.2",
    "mastery_delta": 0.05,
    "current_mastery": 0.85
  }
}
```

## 3.5 页面改造边界

前端练习页需要做以下最小改造：

1. 保留题目展示逻辑。
2. 保留单选、多选、填空的选项交互逻辑。
3. 删除本地 `checkAnswer()` 判题判断。
4. 提交答案时调用 `api.submitQuizAnswer()`。
5. 以提交响应填充：
   - 是否正确
   - 标准答案
   - explanation
   - 掌握度变化

不需要改：

- 题号导航
- 正确率展示方式
- Tabs 布局

## 3.6 错题本统一规则

`GET /api/quiz/wrong` 返回后端聚合结果。

统一规则：

- 同一题重复做错，`count +1`
- 同一题做对后，不删除错题记录，但保留最近一次正确状态
- `variant_quizzes` 只返回题目 ID 列表，不在这个接口内展开完整题面

## 3.7 掌握度更新统一规则

第一阶段统一采用简化规则，避免后面不断返工：

- 单题答对：`+0.05`
- 单题答错：`-0.03`
- 范围钳制：`0.0 ~ 1.0`
- 多选题完全正确才算对
- 填空题第一阶段采用严格匹配，前后 trim 后比较

这套规则先写死，不在第一阶段做复杂加权和衰减。

## 4. 链路二：Tutor / Generate 统一规范

这两条链路放在一起统一，因为它们共享最大的风险点：流式协议。

## 4.1 当前隐患

当前前端存在两个问题：

### 问题 A：页面是假实时

- `generate` 用 `setInterval` 假装进度条
- `tutor` 用整条消息响应，UI 上只显示加载状态

### 问题 B：现有 `useStreaming.ts` 不是真正的 SSE 适配器

它只是读 `fetch().body` 的字节流并直接拼接文本，没有：

- 事件边界解析
- `data:` 行解析
- 多事件类型分发
- `done/error` 终态管理

如果后端直接按文档返回 SSE，当前前端是接不住的。

## 4.2 统一决策

### 决策 A

`tutor` 与 `generate` 统一采用 SSE 文本事件流。

### 决策 B

前端新增统一 SSE hook，不再使用当前 `useStreaming.ts` 直接拼 chunk。

### 决策 C

后端事件必须统一 envelope 格式，禁止不同接口随意返回不同 shape。

### 决策 D

`generate` 页面保留当前布局，但进度完全以后端事件为准。

### 决策 E

`tutor` 页面改为“边收边渲染”消息，而不是等整条返回。

## 4.3 统一 SSE 事件格式

所有流式接口统一使用：

```text
data: {"type":"...","payload":{...}}

```

注意：

- 每个事件一行 `data:`
- 事件之间用空行分隔
- 不再使用无 envelope 的裸字段

统一事件模型：

```json
{
  "type": "text",
  "payload": {
    "content": "闭包"
  }
}
```

## 4.4 Tutor 事件协议

`POST /api/tutor/chat`

允许的事件类型：

- `message_start`
- `text`
- `sources`
- `warning`
- `done`
- `error`

示例：

```text
data: {"type":"message_start","payload":{"message_id":"m_001","role":"assistant"}}

data: {"type":"text","payload":{"content":"闭包"}}

data: {"type":"text","payload":{"content":"（Closure）是指..."}}

data: {"type":"sources","payload":{"knowledgePoints":["3.2 闭包"]}}

data: {"type":"done","payload":{"message_id":"m_001"}}

```

统一规则：

- 页面收到 `message_start` 后创建一条 assistant 空消息
- 收到 `text` 时追加到该消息
- 收到 `sources` 时挂到当前消息元数据
- 收到 `done` 时结束当前消息
- 收到 `error` 时结束并显示降级提示

## 4.5 Generate 事件协议

`POST /api/resources/generate`

允许的事件类型：

- `job_start`
- `progress`
- `preview`
- `done`
- `error`

示例：

```text
data: {"type":"job_start","payload":{"resourceId":"res_20260418_001","title":"Python 函数讲解","resourceType":"document"}}

data: {"type":"progress","payload":{"stage":"画像分析","percent":20}}

data: {"type":"progress","payload":{"stage":"路径匹配","percent":40}}

data: {"type":"preview","payload":{"content":"# Python 函数\n\n"}}

data: {"type":"progress","payload":{"stage":"内容审核","percent":90}}

data: {"type":"done","payload":{"resourceId":"res_20260418_001","status":"completed"}}

```

统一规则：

- `job_start` 决定资源卡片的创建
- `progress` 决定进度条和阶段文案
- `preview` 用于实时预览 markdown 内容，允许多次发送并累积
- `done` 代表本次生成完成
- `error` 代表失败并给出可展示错误

## 4.6 前端统一事件类型建议

前端新增：

```ts
export type SseEvent =
  | { type: 'message_start'; payload: { messageId: string; role: 'assistant' } }
  | { type: 'text'; payload: { content: string } }
  | { type: 'sources'; payload: { knowledgePoints: string[] } }
  | { type: 'warning'; payload: { message: string } }
  | { type: 'job_start'; payload: { resourceId: string; title: string; resourceType: string } }
  | { type: 'progress'; payload: { stage: string; percent: number } }
  | { type: 'preview'; payload: { content: string } }
  | { type: 'done'; payload: Record<string, unknown> }
  | { type: 'error'; payload: { message: string; code?: string } }
```

## 4.7 资源生成状态统一规则

后端状态枚举统一为：

- `generating`
- `completed`
- `failed`
- `review_failed`
- `fallback`

前端兼容规则统一为：

- `generating` -> 生成中
- `completed` -> 完成
- `failed` -> 失败
- `review_failed` -> 按失败展示，但需要展示审核未通过原因
- `fallback` -> 按完成展示，但附加“低可信/模板生成”标识

## 4.8 页面改造边界

### Tutor 页

允许改：

- 发送后先插入用户消息
- 创建 assistant 空消息
- 边收边拼接 assistant 内容
- 记录消息来源 metadata

不改：

- 整体布局
- 气泡样式
- Markdown 渲染方式

### Generate 页

允许改：

- 用真实事件替代本地 `setInterval`
- 用 `preview` 事件更新右侧预览
- 用 `done` 事件决定最终状态

不改：

- 六宫格资源类型 UI
- 左侧资源列表布局
- 右侧预览面板结构

## 4.9 降级规则统一

### Tutor

- 检索不足：返回 `warning` 事件，再继续给通用解释
- 审核拦截：直接返回 `error` 事件，前端落一条固定低可信提示
- 超时：返回 `error`，文案统一为“抱歉，请稍后重试”

### Generate

- 审核失败：返回 `done`，状态置为 `review_failed`
- 生成超时：返回 `error`，同时资源状态写 `failed`
- 生成兜底内容：返回 `done`，状态置为 `fallback`

## 5. 链路三：Onboarding 统一规范

## 5.1 当前隐患

当前前端 onboarding 是 5 步卡片问卷。

但后端设计文档定义的是：

- `POST /api/profile/initiate`
- `POST /api/profile/chat`
- 3 轮对话式建档

这不是“小差异”，而是两套完全不同的产品流。

如果不先统一，后面一定返工，因为：

1. 页面结构不同
2. 状态模型不同
3. API 形态不同
4. 数据来源不同
5. 画像抽取逻辑不同

## 5.2 统一决策

第一阶段保留当前前端的“5 步卡片问卷”页面，不改造成聊天 UI。

这是本文件最重要的决定之一。

原因：

- 当前问卷页面完成度高，且已有测试覆盖
- 如果先强推对话式建档，会立刻引发前端大改
- 比赛阶段先跑通真实建档比“形式一定是对话式”更重要

因此，后端画像建档方案做如下统一：

### 决策 A

第一阶段新增“问卷式建档接口”，作为主入口。

### 决策 B

文档中的对话式建档接口先保留为二期能力，不作为当前联调主链路。

### 决策 C

当前 `/onboarding` 页面只对接问卷式建档 API。

## 5.3 新的主接口定义

新增：

### `POST /api/profile/onboarding`

请求：

```json
{
  "goal": ["期末提分"],
  "knowledge_level": "有一些基础",
  "weak_points": ["数据结构"],
  "learning_preference": ["实践型"],
  "daily_time_label": "1-2小时"
}
```

响应：

```json
{
  "success": true,
  "data": {
    "user_id": "single_user",
    "profile_version": 1,
    "next_route": "/"
  }
}
```

## 5.4 Onboarding 字段映射统一

当前页面问卷项与画像字段的映射统一如下：

### Step 1 学习目标

前端选项 -> 后端 `goal`

### Step 2 编程基础

前端选项 -> 后端 `knowledge_level`

### Step 3 薄弱环节

前端选项 -> 后端 `weak_points`

### Step 4 学习偏好

前端选项 -> 后端 `learning_preference`

### Step 5 学习时间

前端选项 -> 后端 `daily_time`

时间映射写死为：

- `30分钟以内` -> `30`
- `30-60分钟` -> `45`
- `1-2小时` -> `90`
- `2小时以上` -> `150`

## 5.5 未在问卷中直接采集的画像字段

当前问卷没有直接覆盖：

- `name`
- `major`
- `grade`
- `cognitive_style`
- `practical_ability`
- `current_stage`

统一规则：

- `name` / `major` / `grade` 第一阶段留空或从 auth 假数据预填
- `cognitive_style` 第一阶段由后端根据偏好规则推断
- `practical_ability` 第一阶段根据 `knowledge_level` 映射默认值
- `current_stage` 第一阶段由路径规划接口首次生成时写回

推断规则统一如下：

### `cognitive_style`

- 若包含 `视觉型` -> `视觉归纳型`
- 若包含 `实践型` -> `动手探索型`
- 若包含 `阅读型` -> `文本理解型`
- 若包含 `听觉型` -> `讲解吸收型`
- 多选时优先级：
  - `实践型`
  - `视觉型`
  - `阅读型`
  - `听觉型`

### `practical_ability`

- `零基础` -> `尚未形成实操能力`
- `有一些基础` -> `可完成简单练习`
- `基础较好` -> `能独立完成小项目`
- `有编程经验` -> `能独立完成综合项目`

## 5.6 页面改造边界

当前 onboarding 页面允许做的改动只有：

1. 将本地 selections 提交到 `api.submitOnboarding()`
2. 增加提交 loading
3. 增加提交失败提示
4. 成功后跳转首页

不允许：

- 重写成聊天式建档界面
- 在页面里硬编码画像推断规则
- 在页面里写复杂数据转换

## 5.7 与旧对话式建档接口的关系

旧接口：

- `POST /api/profile/initiate`
- `POST /api/profile/chat`

统一处理：

- 第一阶段不删除
- 但不作为主链路实现优先级
- 如果时间允许，作为二期实验性接口保留

文档执行顺序上，问卷式 onboarding 优先于对话式 onboarding。

## 6. 三条链路共用的隐藏隐患与统一处理

## 6.1 字段命名不一致

统一处理：

- 后端 snake_case
- 前端 camelCase
- 全部在 API 适配层映射

## 6.2 枚举值零散

统一写死枚举，不再靠示例推断。

### quiz.type

- `single`
- `multiple`
- `fill_blank`

### tutor.mode

- `knowledge_qa`
- `step_hint`
- `follow_up`

### resource.type

- `document`
- `ppt`
- `mindmap`
- `quiz`
- `reading`
- `code`

### resource.status

- `generating`
- `completed`
- `failed`
- `review_failed`
- `fallback`

### path.status

- `completed`
- `current`
- `pending`

## 6.3 时间与日期字段不一致

统一规则：

- 后端所有时间字段返回 ISO 8601 字符串
- 前端显示格式化在 UI 层处理
- 不再在 mock 中混用 `2026-04-15` 和 `2026-04-15T10:30:00`

## 6.4 错误展示不一致

统一规则：

- 普通接口失败：`success: false, error: "..."`
- SSE 失败：发送 `error` 事件
- 页面只展示用户可读文案，不展示内部异常栈

## 6.5 测试失效风险

这三条链路改造后，以下 Playwright 用例大概率会受影响：

- onboarding 流程完成后跳首页
- practice 提交后显示反馈
- tutor 发送消息后的消息展示
- generate 点击生成后的进度和结果

统一要求：

- 在真实 API 接入后，更新对应测试断言
- 不用再依赖 mock 的固定文案和本地计时器

---

## 7. 实施顺序

按返工风险从高到低，统一后的开发顺序应为：

1. 先改 `quiz` 契约
2. 再补统一 SSE hook
3. 再接 `tutor`
4. 再接 `generate`
5. 最后接 `onboarding` 问卷提交

原因：

- `quiz` 是字段和业务规则冲突最大的一块
- SSE hook 是 `tutor/generate` 的共同基础
- `onboarding` 主流程虽重要，但第一阶段选择“保留现有问卷 UI”后，返工风险已经大幅下降

---

## 8. 最终执行结论

从本文件开始，三条链路的最终统一方案为：

- `quiz`：服务端判题，题面不暴露答案
- `tutor/generate`：统一 SSE 事件协议，前端统一解析
- `onboarding`：第一阶段保留问卷式 UI，新增问卷提交 API，旧对话式接口降为二期

这三条一旦锁定，后续后端开发就不会再因为协议方向摇摆而反复返工。
