## 1. 文档目的

《学而思》是一套面向高校课程学习场景的个性化学习智能体系统。系统围绕赛题要求，重点支撑以下核心能力：对话式学习画像构建、多智能体协同生成至少 5 类个性化资源、个性化学习路径规划与资源推送、智能辅导、学习效果评估，以及防幻觉、内容安全和多模态结果展示。

本数据设计文档的目标，不是单纯说明“数据存在哪里”，而是系统性回答以下问题：

1. 学生画像、学习行为、资源结果、评估结果如何被统一建模。

2. 多智能体在资源生成、路径规划、学习评估中如何完成数据交接。

3. 前台学习端、学习效果分析页、管理端如何共享同一套可信数据底座。

4. 系统如何在保证结构清晰、可解释、可追踪的前提下，支撑课程资源生成、网页渲染和后续扩展。

***

## 2. 数据设计目标

围绕“因材施教”的系统目标，学而思的数据设计遵循五项原则：

### 2.1 以学生状态为中心

系统所有功能最终都围绕“学生当前是谁、会什么、缺什么、下一步学什么”展开。因此，数据设计必须以学生画像与学习状态为核心。

### 2.2 以学习过程为依据

系统不能只记录最终结果，还必须记录学习过程中发生了什么，包括资源浏览、任务完成、练习作答、错题分布、阶段变化等。只有过程数据充分，学习效果评估才具有可信性。

### 2.3 以资源留痕为基础

系统生成的讲义、PPT、PDF、视频、练习包和学习报告，既是学习内容，也是评估依据和管理对象。因此，每一类资源都必须具备可追踪、可预览、可重建的元数据描述。

### 2.4 以视图模型支撑渲染

系统内部状态数据不直接暴露给前端。前端页面，尤其是学习效果分析网页和管理端页面，统一读取“视图模型数据”，从而实现页面结构清晰、渲染稳定、逻辑解耦。

### 2.5 以结构化与可扩展为导向

数据结构既要满足当前系统落地，也要为后续的数据库化、多用户治理和后台运营扩展保留空间。

***

## 3. 数据总体架构

学而思采用“**状态数据层 + 过程事件层 + 资源资产层 + 评估视图层 + 管理索引层**”五层数据架构。

### 3.1 状态数据层

用于描述学生当前的稳定状态与可计算状态，包括：

* 学生画像

* 知识点掌握度

* 当前学习阶段

* 当前任务进度

* 当前推荐依据

它服务于路径规划、资源生成、推荐和答疑。

### 3.2 过程事件层

用于记录学习过程中发生的关键行为，包括：

* 对话建档事件

* 学习资源使用事件

* 练习提交事件

* 阶段任务完成事件

* 管理员操作事件

它服务于评估、审计、追踪和回溯。

### 3.3 资源资产层

用于保存系统生成的各类实际资源文件，包括：

* 讲义文档

* PPT 及其预览文件

* PDF

* 视频

* 学习报告网页

* 思维导图图片

它服务于资源预览、历史查看、再次引用和管理端操作。

### 3.4 评估视图层

用于保存面向展示和渲染的数据模型，包括：

* 学习效果分析页视图模型

* 学习报告总结数据

* 知识掌握图表数据

* 薄弱点与建议列表

它服务于网页渲染和可视化分析。

### 3.5 管理索引层

用于统一组织各类对象的入口信息，包括：

* 学生总索引

* 资源总索引

* 任务总索引

* 管理操作日志

它服务于管理端的用户操作、资源检索、任务追踪和异常治理。

## 4. 核心数据对象设计

学而思的数据体系围绕 8 类核心对象展开。

## 4.1 学生对象 User

表示平台中的学习主体，是所有状态和资源的归属中心。

核心字段包括：

* `user_id`

* `name`

* `major`

* `grade`

* `course_id`

* `created_at`

* `status`

## 4.2 课程对象 Course

表示系统当前支撑的课程知识边界，是知识库和路径规划的基础。

核心字段包括：

* `course_id`

* `course_name`

* `chapters`

* `knowledge_points`

* `difficulty_levels`

## 4.3 学习画像对象 Profile

表示学生当前的个性化学习状态，是学而思的核心状态模型。

建议至少包含以下字段：

* 基础身份信息

* 学习目标

* 知识基础

* 知识短板

* 学习偏好

* 认知风格

* 学习节奏

* 实操能力

* 当前学习阶段

* 最近更新原因

## 4.4 掌握度对象 Mastery

表示学生对各知识点的掌握情况，是路径规划、练习生成和评估分析的基础。

核心字段包括：

* `knowledge_point_id`

* `knowledge_point_name`

* `mastery_score`

* `last_updated`

* `source`

## 4.5 学习任务对象 Task

表示系统为学生生成的阶段任务、节点任务和资源任务。

核心字段包括：

* `task_id`

* `user_id`

* `task_type`

* `title`

* `status`

* `start_time`

* `end_time`

* `related_resource_ids`

## 4.6 资源对象 Resource

表示系统生成的学习内容，是多智能体协同的核心输出。

核心字段包括：

* `resource_id`

* `user_id`

* `resource_type`

* `title`

* `knowledge_points`

* `created_at`

* `source_agent`

* `files`

* `status`

## 4.7 评估对象 Evaluation

表示系统对学生学习效果的阶段性分析结果。

核心字段包括：

* 总学习时长

* 资源完成率

* 任务完成率

* 习题正确率

* 薄弱点排行

* 掌握度变化

* 阶段建议

## 4.8 管理操作对象 AdminAction

表示管理端对学生、资源、任务等对象进行的操作记录。

核心字段包括：

* `action_id`

* `operator`

* `target_type`

* `target_id`

* `action_type`

* `before`

* `after`

* `created_at`

## 5. 存储组织方案

为保证结构清晰、读取直接、资源可追踪，学而思采用“**用户目录化 + 资源子目录化 + 索引集中化**”的数据组织方式。

### 5.1 顶层目录设计

```plain&#x20;text
backend/
  data/
    users/
      {user_id}/
        profile.json
        mastery.json
        history.json
        learning_events.jsonl
        quiz_records.json
        resource_usage.json
        task_progress.json
        evaluation_snapshot.json
        report_view_model.json
        resources/
          resources_index.json
          {resource_dir}/
            meta.json
            ...
    indexes/
      users_index.json
      resources_global_index.json
      tasks_index.json
      admin_action_log.jsonl
```

该结构的设计逻辑是：

* 用户级状态数据放在用户目录下，便于按学生聚合查看。

* 资源级文件单独成目录，便于保留原文件、预览文件和元信息。

* 平台级索引统一放在 `indexes` 目录下，便于管理端集中读取。

## 6. 学生状态数据设计

## 6.1 `profile.json`：学生画像主文件

该文件用于保存学生当前最新的学习画像，是系统所有个性化能力的统一状态入口。

建议结构如下：

```json
{
  "user_id": "u_001",
  "course_id": "python_basic",
  "profile": {
    "major": "计算机科学与技术",
    "grade": "大二",
    "goal": ["期末提分", "补基础"],
    "knowledge_level": "入门",
    "weak_points": ["函数", "面向对象"],
    "learning_preference": ["代码案例", "短讲解"],
    "cognitive_style": "先案例后原理",
    "daily_time": 60,
    "practical_ability": "可读代码，独立编写较弱",
    "current_stage": "基础巩固"
  },
  "version": 12,
  "updated_at": "2026-04-15T10:30:00Z"
}
```

其作用包括：

* 为学习路径规划提供输入

* 为资源生成提供个性化参数

* 为资源推荐提供偏好和短板依据

* 为评估模块提供初始状态基准

## 6.2 `mastery.json`：知识掌握度文件

该文件用于保存学生对课程知识点的掌握情况。

建议结构如下：

```json
{
  "user_id": "u_001",
  "course_id": "python_basic",
  "knowledge_points": [
    {"id": "kp_01", "name": "变量与数据类型", "score": 0.82},
    {"id": "kp_02", "name": "循环语句", "score": 0.76},
    {"id": "kp_03", "name": "函数", "score": 0.41},
    {"id": "kp_04", "name": "面向对象", "score": 0.28}
  ],
  "updated_at": "2026-04-15T10:35:00Z"
}
```

其作用包括：

* 为路径节点排序提供依据

* 为练习题难度控制提供依据

* 为学习报告中的图表展示提供数据

## 6.3 `history.json`：状态变更摘要文件

该文件用于保存画像和学习阶段的重要变更摘要。

其记录内容包括：

* 哪次对话更新了学习目标

* 哪次学习后判定薄弱点变化

* 哪次评估后更新了学习阶段

它的价值在于提升可解释性，使系统可以说明“为什么做出当前推荐”。

## 7. 学习过程数据设计

## 7.1 `learning_events.jsonl`：学习事件流水

该文件用于追加记录用户在学习过程中的关键行为，一行一个事件。

示例：

```json
{"time":"2026-04-15T10:00:00","event":"open_resource","resource_id":"res_001","resource_type":"note","knowledge_point":"函数"}
{"time":"2026-04-15T10:12:00","event":"finish_resource","resource_id":"res_001","duration":720}
{"time":"2026-04-15T10:20:00","event":"submit_quiz","quiz_id":"quiz_001","score":60}
```

其作用包括：

* 计算学习时长

* 计算资源完成率

* 生成活跃趋势

* 分析学习行为模式

## 7.2 `quiz_records.json`：练习与测验结果文件

该文件用于保存题目作答和练习结果。

建议记录：

* 题目 ID

* 考点

* 难度

* 是否正确

* 用时

* 错误类型

* 所属任务

该文件用于：

* 计算正确率

* 识别高频错误知识点

* 驱动错题分析与变式题生成

* 支撑学习效果评估

## 7.3 `resource_usage.json`：资源使用情况文件

该文件用于保存学生对不同资源的使用情况。

建议记录：

* 资源类型

* 资源 ID

* 点击次数

* 浏览时长

* 完成率

* 收藏/再次查看情况

其作用包括：

* 修正学习偏好

* 评估哪类资源更有效

* 为推荐系统提供反馈依据

## 7.4 `task_progress.json`：任务进度文件

该文件用于保存学习路径中各任务节点的完成情况。

建议记录：

* 当前阶段

* 当前任务

* 已完成任务列表

* 未完成任务列表

* 阶段完成率

* 最近推进时间

其作用包括：

* 支撑学习路径页面展示

* 支撑阶段性学习报告

* 支撑管理员查看学生推进状态

## 8. 资源数据设计

系统生成的资源不仅要可存，还要可找、可预览、可管理。因此，资源数据采用“**资源目录 + 元数据文件 + 资源索引**”的组合设计。

## 8.1 单资源目录设计

每个资源生成后，单独建立一个资源目录，例如：

```plain&#x20;text
resources/
  20260415_103000_slides_001/
    meta.json
    slides.pptx
    slides_preview.pdf
```

## 8.2 `meta.json`：资源元数据文件

建议结构如下：

```json
{
  "resource_id": "res_20260415_001",
  "user_id": "u_001",
  "resource_type": "slides",
  "title": "Python函数专题PPT",
  "course_id": "python_basic",
  "knowledge_points": ["函数"],
  "created_at": "2026-04-15T10:30:00",
  "source_agent": "resource_agent",
  "files": {
    "origin": "slides.pptx",
    "preview": "slides_preview.pdf"
  },
  "status": "ready"
}
```

该文件回答的是：

* 这个资源属于谁

* 是什么类型

* 基于什么知识点生成

* 从哪个智能体流程产生

* 用户如何预览和下载

## 8.3 `resources_index.json`：用户资源总索引

该文件用于保存某个学生所有历史资源的摘要信息，前台历史资源页与管理端用户资源页直接读取此文件。

建议字段包括：

* `resource_id`

* `title`

* `resource_type`

* `created_at`

* `dir`

* `preview_path`

* `tags`

* `status`

## 8.4 资源文件类型规范

为保证渲染效果和资源留存，系统建议同时保留“原件 + 预览件”：

* 讲义：`md/html`

* PPT：`pptx + pdf 预览`

* 文档：`pdf`

* 视频：`mp4`

* 报告：`html`

* 思维导图：`png/svg`

这样既保留源文件，也保证页面直接预览能力。

## 9. 学习效果评估数据设计

学习效果评估不是一张静态报告，而是一条完整的数据链。其核心目标是将“学习行为数据”和“系统判断结果”有效衔接，并最终形成网页渲染所需的视图数据。

## 9.1 `evaluation_snapshot.json`：评估快照文件

该文件保存系统对学生某一时点学习效果的结构化评估结果。

建议包含：

* 总学习时长

* 任务完成率

* 资源完成率

* 习题正确率

* 章节掌握度摘要

* 薄弱点排名

* 最近提升知识点

* 当前阶段结论

* 下一步建议

该文件服务于：

* 后续评估回溯

* 管理端查看评估结果

* 学习报告摘要生成

## 9.2 `report_view_model.json`：报告视图模型文件

该文件专门为网页渲染设计，是学习效果分析页的直接数据源。

建议结构如下：

```json
{
  "summary_cards": {
    "total_learning_time": "6.5小时",
    "task_completion_rate": "78%",
    "quiz_accuracy": "64%",
    "current_stage": "基础巩固"
  },
  "charts": {
    "mastery_bar": [
      {"name": "变量与类型", "value": 82},
      {"name": "循环语句", "value": 76},
      {"name": "函数", "value": 41},
      {"name": "面向对象", "value": 28}
    ],
    "trend_line": [
      {"date": "04-10", "value": 42},
      {"date": "04-11", "value": 46},
      {"date": "04-12", "value": 51}
    ]
  },
  "weak_points": ["函数", "面向对象", "调试能力"],

    "优先学习函数章节的基础讲义与例题",
    "补充类与对象图解资源",
    "完成一套函数专题练习"
  ]
}
```

其设计意义在于：

* 让前端无需理解复杂业务计算

* 让学习效果网页可以直接渲染卡片、柱状图、趋势图和建议模块

* 让管理端与学习端共享统一展示结果

## 11. 多智能体数据交接设计

赛题强调多智能体协同，因此数据设计必须体现清晰的数据交接链路，而不是孤立地存几份文件。

### 11.1 画像构建链路

学生对话输入
→ 对话解析 Agent 抽取特征
→ 画像 Agent 生成 `profile.json`
→ 路径规划 Agent 读取画像生成学习阶段与任务
→ 更新 `task_progress.json`

### 11.2 资源生成链路

画像与知识库上下文
→ 规划 Agent 确定资源范围
→ 资源生成 Agent 输出资源文件
→ 审核 Agent 校验内容与安全
→ 写入资源目录与 `meta.json`
→ 更新 `resources_index.json` 与 `resources_global_index.json`

### 11.3 学习评估链路

学习行为事件、资源使用记录、练习记录
→ 评估 Agent 读取 `learning_events.jsonl`、`quiz_records.json`、`resource_usage.json`
→ 更新 `mastery.json`
→ 生成 `evaluation_snapshot.json`
→ 汇总为 `report_view_model.json`
→ 前端渲染学习效果分析页

### 11.4 管理操作链路

管理员操作学生或资源
→ 更新对应状态文件或资源索引
→ 写入 `admin_action_log.jsonl`
→ 如涉及重生成，创建新任务并更新 `tasks_index.json`

## 12. 数据一致性与更新机制

为保证多模块协同下的数据正确性，学而思定义以下更新规则。

### 12.1 单一主文件原则

每类核心状态只保留一个主文件：

* 学习画像主文件：`profile.json`

* 掌握度主文件：`mastery.json`

* 评估主文件：`evaluation_snapshot.json`

避免多个文件同时承担同一状态来源，减少歧义。

### 12.2 事件追加优先原则

学习行为、管理行为和过程流水统一采用追加式记录，不覆盖原数据，以保证过程可追踪。

### 12.3 索引同步更新原则

任何新资源、新任务、新学生写入后，都必须同步更新对应索引文件，否则前端和管理端无法准确检索。

### 12.4 状态机约束原则

任务状态统一使用：

* `pending`

* `running`

* `reviewing`

* `ready`

* `failed`

资源状态与任务状态保持一致口径，避免页面展示混乱。

### 12.5 版本号原则

画像和评估文件建议带 `version` 字段，便于后续变更追踪和回滚。

## 13. 数据与页面渲染的关系设计

由于系统需要直接呈现 PPT、PDF、视频和学习效果网页，因此数据设计必须天然支持渲染。

### 13.1 文档类资源渲染

* 讲义优先渲染 `html/md`

* PDF 直接作为页面预览源

* PPT 同时保留原始文件和 PDF 预览件

### 13.2 视频类资源渲染

* 视频统一输出为可直接播放的 `mp4`

* 预留封面图 `cover.jpg` 作为资源列表展示入口

### 13.3 学习效果页渲染

* 页面不直接读取过程事件文件

* 页面只读取 `report_view_model.json`

* 由前端按模块渲染统计卡片、掌握度图表、薄弱点分析和建议模块

### 13.4 管理端渲染

* 用户管理页读取 `users_index.json`

* 资源管理页读取 `resources_global_index.json`

* 学生详情页再回查其个人目录下的状态文件和资源索引

这套设计确保系统界面与底层数据结构解耦，提升页面开发效率和展示稳定性。

***

## 14. 安全性、可信性与可追踪性设计

赛题要求系统具备防幻觉和内容安全过滤机制，数据设计应为此提供支撑。

### 14.1 内容可信支撑

资源元数据中保留：

* 来源知识点

* 生成时间

* 生成智能体

* 审核状态

便于说明资源的生成依据和校验过程。

### 14.2 行为审计支撑

通过学习事件流水和管理操作日志，系统可以说明：

* 某项评估依据何种学习行为得出

* 某项管理调整是谁在何时做出的

### 14.3 数据恢复支撑

通过主文件 + 事件流水 + 索引文件的分层设计，即使某一层文件损坏，也可通过目录重建或日志追溯恢复主要状态。

***

## 15. 设计总结

学而思的数据设计并非孤立的数据存档方案，而是围绕“画像—路径—资源—学习—评估—管理”完整闭环构建的一套系统化数据底座。

其核心价值体现在四点：

第一，**以学生画像为中心统一个性化状态**，确保系统真正理解学生差异。
第二，**以学习过程事件为依据支撑评估与推荐**，确保学习效果分析有真实数据来源。
第三，**以资源元数据和索引体系支撑多模态资源留存与管理**，确保生成结果可预览、可追踪、可复用。
第四，**以视图模型支撑网页渲染和管理端展示**，确保系统对评委呈现出完整、清晰、可信的智能学习平台形态。

这套数据设计既满足赛题对个性化学习、多智能体协同、学习评估和系统可用性的要求，也为后续平台规模扩展、数据库升级和后台治理提供了清晰的演进基础。
