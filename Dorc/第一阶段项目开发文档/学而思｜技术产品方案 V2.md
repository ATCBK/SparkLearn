## 1. 文档目的

本文档是在初版技术产品方案、数据设计文档、动态数据设计文档的基础上，结合赛题要求与团队讨论结果产出的定稿版技术方案。本文档聚焦四个问题：

第一，技术选型是否明确且可落地。

第二，课程知识库如何构建并接入系统。

第三，多智能体、资源生成、路径规划等核心模块如何用具体框架实现。

第四，系统如何被测试、部署并满足比赛合规要求。

本文档作为后续开发的唯一依据，所有开发工作必须以此为准。

## 2. 技术选型

### 2.1 后端框架

后端采用 **FastAPI**（Python）。

选择理由：FastAPI 原生支持流式输出，赛题要求的"流式输出"可直接通过 `StreamingResponse` 实现；FastAPI 的 async 机制天然适配多 Agent 并行调用、RAG 检索和大模型推理等异步 I/O 密集型场景；FastAPI 自动生成 Swagger UI 接口文档，比赛演示时可直接展示 API 接口；团队已有 FastAPI 开发经验，无需额外学习成本。

### 2.2 前端框架

前端采用 **React + Next.js**。

选择理由：React 生态的训练数据量远大于 Vue，AI 工具生成 React 代码的准确率和完整度更高；Next.js 的路由、SSR、API 路由开箱即用；Ant Design、shadcn/ui 等成熟组件库可直接调用；对 AI Native 开发方式最友好。

### 2.3 大模型

大模型采用 **科大讯飞星火 API**。

赛题明确要求"开发过程中使用的其他 AI 辅助工具，需选用科大讯飞相关工具"。系统中所有涉及大模型推理的环节，包括对话式建档、资源生成、辅导答疑、评估分析，均以讯飞星火为底座。

### 2.4 Agent 框架

Agent 编排采用 **CrewAI**。

选择理由：CrewAI 的 Agent 定义方式是"角色 + 目标 + 背景描述"，天然契合画像 Agent、资源 Agent、路径 Agent 的分工方式；Agent 之间的协作链路是声明式的，直接告诉框架谁先谁后，开发效率高；CrewAI 是 Python 原生框架，与 FastAPI 集成顺畅；社区活跃，文档完善，对 AI Native 开发友好。

不选择 LangGraph 的原因：LangGraph 学习曲线陡，代码量大，调试困难，比赛时间有限。

### 2.5 向量数据库

知识库检索采用 **ChromaDB**。

课程知识库约 1500 条知识点条目，ChromaDB 在此规模下本地运行零压力、零配置。ChromaDB 纯 Python 实现，安装简单，无需额外服务。采用"关键词精确匹配 + ChromaDB 语义检索"组合方案，兼顾精确性和语义理解能力。

### 2.6 数据存储

采用 **SQLite + JSON/JSONL 混合** 存储方案。

学生画像、掌握度等需要频繁查询和更新的结构化数据存入 SQLite，解决纯 JSON 文件的并发写入冲突问题。学习事件流水采用 JSONL 文件，纯追加写入无并发冲突。资源元数据和评估视图模型采用 JSON 文件，结构简单，前端可直接读取。

### 2.7 资源生成

内容类资源（课程讲解文档、练习题、拓展阅读、代码案例）由讯飞星火直接生成，不要求美观，内容正确即可。

展示类资源（PPT、视频/动画）调用外部专业平台 API 生成，需要美观展示。PPT 生成平台待调研，具体方案见《外部资源生成 API 待招清单》。思维导图采用前端 Mermaid/Markmap 渲染，后端只输出 Markdown 格式数据。

### 2.8 语音

语音采用 **讯飞 ASR + TTS**，双向支持。

ASR 语音识别用于首次建档对话和智能辅导答疑中的语音输入。TTS 语音合成用于系统回答/讲解内容的语音播报。支持语音的页面包括：首次建档对话页、智能辅导答疑页、学习效果分析报告页。

### 2.9 技术选型总览

| 层级 | 选型 | 说明 |
|------|------|------|
| 后端框架 | FastAPI (Python) | 流式输出、异步 I/O、Swagger 文档 |
| 前端框架 | React + Next.js | AI Native 友好、组件库丰富 |
| 大模型 | 讯飞星火 API | 赛题要求科大讯飞工具 |
| Agent 框架 | CrewAI | 角色分工、声明式编排 |
| 向量数据库 | ChromaDB | 本地零配置、1500 条规模 |
| 数据存储 | SQLite + JSON/JSONL | 结构化 + 追加写入 + 视图模型 |
| 知识切片 | LangChain TextSplitter | 仅切片工具，不用整套框架 |
| PPT 生成 | 外部 API（待招） | 需要美观展示 |
| 视频/动画 | 外部 API（待招） | 需要美观展示 |
| 思维导图 | 前端 Mermaid/Markmap | 后端输出 Markdown |
| 语音 | 讯飞 ASR + TTS | 语音输入 + 语音播报 |
| 部署 | 本地运行 | 后端 uvicorn，前端 npm run dev |

## 3. 课程知识库构建

### 3.1 课程选择

系统以《Python 程序设计》课程为切入点，满足赛题"以某一具体专业课程为切入点"的要求。

### 3.2 知识库来源

系统知识库由四份文件构成：

第一份是《python 从入门到实践.pdf》，位于 KnowledgeRepo 目录下，是课程教材原始文件，需要做 PDF 切片处理后作为补充知识源入库。

第二份是《Python 基础知识库.xlsx》，是已切片的结构化知识点，包含基础语法体系等内容，可直接入库。

第三份是《Python 进阶知识库.xlsx》，是已切片的结构化知识点，包含进阶主题等内容，可直接入库。

第四份是《Python 面试知识库.xlsx》，是已切片的结构化知识点，包含面试高频知识点等内容，可直接入库。

三个 xlsx 文件共约 1500 条知识点条目，是系统知识库的主体数据。

### 3.3 表格数据结构

三个 xlsx 文件统一采用七列结构：

第一列是一级分类，表示大章节归属，例如"基础语法体系"。

第二列是二级分类，表示子模块归属，例如"核心概念 - 变量"。

第三列是具体知识点条目编号，是知识点的唯一标识，例如"1.1.1"。

第四列是具体知识点条目，是知识点的名称，例如"变量定义规则"。

第五列是清洗后内容，包含完整讲解和示例说明，可直接用于 RAG 检索和 Agent 上下文输入。

第六列是数据来源，标注该知识点的出处。

第七列是检索标签，用于关键词匹配和标签过滤。

### 3.4 检索方案

采用"关键词精确匹配 + ChromaDB 语义检索"组合方案。

入库阶段：将三个 xlsx 文件中的知识点内容做 embedding，存入 ChromaDB 向量数据库。同时保留关键词和标签索引用于精确匹配。

检索阶段：学生输入问题后，先做关键词精确匹配（匹配章节名、知识点名、标签），命中则直接返回。未命中时走 ChromaDB 语义检索，通过向量相似度找到语义相近的知识点。最后根据学生画像进行标签过滤，控制难度和课程阶段。

补充召回：PDF 教材切片内容作为第三层补充召回来源，在前两层召回不足时补充上下文。

### 3.5 RAG 检索链路

系统 RAG 检索遵循三段式召回链路：

第一层是关键词检索，用于精确定位章节、知识点名和术语标签。

第二层是语义检索，通过 ChromaDB 向量相似度理解学生自然语言问题。

第三层是标签过滤，根据学生画像控制难度等级、课程阶段和资源类型。

最终返回给 Agent 的不是原始文档，而是结构化知识上下文包，包括关联章节、核心知识点、示例片段、难度等级和可信度分数。

当检索结果低相关、低置信或命中不足时，系统不直接自由发挥，而是提示学生补充信息，或降级为通用解释并由审核 Agent 标记为低可信内容。

## 4. 多智能体架构设计

### 4.1 Agent 角色定义

系统配置 7 个核心 Agent，通过 CrewAI 框架编排协作。

**画像 Agent**：负责从对话中抽取学生特征、更新画像、生成学生标签。它关注画像是否需要修正、薄弱点是否变化、目标和偏好是否发生迁移。

**路径规划 Agent**：负责计算阶段目标、章节顺序和节点优先级。它关注当前应该优先补哪部分、是推进还是回退、下一阶段节点如何排列。

**资源生成 Agent**：负责生成课程讲解文档、知识点思维导图、拓展阅读材料和代码实操案例。它关注生成何种资源、资源难度如何设定、内容形式应偏文字还是代码案例。

**练习与案例 Agent**：负责生成分层练习题、错题变式题和实验任务。它关注题目难度控制、错题变式生成和实验任务设计。

**评估 Agent**：负责掌握度评估、薄弱点发现和学习报告生成。它关注当前学习是否有效、哪些知识点有提升、哪些短板仍需关注、下一阶段建议是什么。

**审核 Agent**：负责事实校验、安全过滤、格式标准化和低置信拦截。所有 Agent 的输出都需经过审核 Agent 校验后才交付给前端展示。

**答疑 Agent**：负责知识点问答、分步提示和追问解释。它关注学生的具体问题、当前学习上下文、应采用精简提示还是分步讲解。

> 注意：原方案中的"任务理解 Agent"取消，意图识别功能集成到 CrewAI 的 Task 定义和编排器逻辑中。

### 4.2 编排规则

资源生成类任务采用串行与并行混合的编排方式。

首先由画像 Agent 串行执行，明确学生当前状态。随后路径规划 Agent 串行执行，给出目标知识范围。接着资源生成 Agent 与练习案例 Agent 并行执行，同时生成内容。最后由审核 Agent 串行执行，统一校验输出质量。

辅导答疑类任务由答疑 Agent 独立完成，从知识库检索上下文后调用讯飞星火生成回答，经审核 Agent 校验后交付。

评估类任务由评估 Agent 读取学习事件、练习记录和资源使用情况，生成评估结论和学习报告。

### 4.3 Agent 核心流程

每个 Agent 的执行遵循统一三步流程：

第一步，从 AgentContext 中读取当前任务所需的学生数据和知识库上下文。

第二步，调用讯飞星火 API 完成推理和内容生成。

第三步，将结果结构化输出，返回 AgentResult 给编排器。

### 4.4 容错机制

系统设计四类容错策略：

单 Agent 超时时，启用降级模板或重试一次。

知识库检索为空时，提示学生补充信息或切换通用模式。

Agent 输出不合规时，审核 Agent 回退要求重生成。

多 Agent 结果冲突时，优先保留高可信结果并标注来源。

## 5. 数据存储设计

### 5.1 混合存储策略

学而思采用混合存储方案，不同类型的数据使用最适合的存储方式。

学生画像和掌握度等需要频繁查询和更新的结构化数据存入 SQLite。学习事件流水采用 JSONL 文件，纯追加写入无并发冲突。资源元数据和索引采用 JSON 文件，结构简单直接。评估视图模型采用 JSON 文件，预计算好展示数据。知识库内容同时使用 ChromaDB（向量检索）和 JSON（结构化查询）。

### 5.2 SQLite 表设计

**students 表**：存储学生基础信息，字段包括自增主键 id、用户标识 user_id（唯一）、姓名 name、专业 major、年级 grade、课程标识 course_id、创建时间 created_at 和更新时间 updated_at。

**profiles 表**：存储学生画像，字段包括自增主键 id、外键 user_id、学习目标 goal（JSON 数组）、知识基础 knowledge_level、知识短板 weak_points（JSON 数组）、学习偏好 learning_preference（JSON 数组）、认知风格 cognitive_style、每日学习时长 daily_time（分钟）、实操能力 practical_ability、当前学习阶段 current_stage、版本号 version 和更新时间 updated_at。

**mastery_records 表**：存储知识点掌握度，字段包括自增主键 id、外键 user_id、知识点 ID knowledge_point_id、知识点名称 knowledge_point_name、掌握度分数 mastery_score（0 到 1 的浮点数）和最后更新时间 last_updated。

### 5.3 文件存储目录结构

系统文件按以下目录组织：

backend 目录下的 data 目录包含 users 子目录和 indexes 子目录。users 目录下按用户划分，当前单学生演示阶段固定使用 single_user 目录，包含 learning_events.jsonl（学习事件流水）、quiz_records.json（练习结果记录）、resource_usage.json（资源使用情况）、task_progress.json（任务进度）、evaluation_snapshot.json（评估快照）和 report_view_model.json（报告视图模型）。resources 子目录下包含 resources_index.json（资源总索引）和各资源独立目录，每个资源目录包含 meta.json 元数据和对应的资源文件。

indexes 目录存放平台级索引文件，包括 resources_global_index.json（资源总索引）、tasks_index.json（任务总索引）和 admin_action_log.jsonl（管理操作日志）。

data 目录下的 db 子目录存放 SQLite 数据库文件 sparklearn.db。

### 5.4 单学生模式

当前阶段只支持单学生演示，不实现用户登录和切换功能。数据库中固定一个 user_id。多学生扩展作为后续优化点。

## 6. 页面清单

### 6.1 学生端页面

系统学生端包含 8 个页面。

**首次建档对话页**：通过 3 轮对话式采集学生画像信息，支持流式输出和语音输入。第一轮采集课程、目标和当前基础。第二轮采集薄弱点、习惯和偏好。第三轮采集时间安排和实操能力。建档结束后由画像 Agent 生成结构化画像。

**首页学习总览页**：展示今日学习任务、当前学习阶段、画像标签、最近完成资源、待完成任务和学习时长统计。这是学生登录后看到的第一个页面，需要快速了解当前学习状态。

**学习路径页**：可视化展示阶段节点和当前进度，包含节点名称、所属章节、学习目标、推荐资源、推荐练习、掌握状态和下一步动作。路径根据评估结果动态调整。

**资源生成与资源详情页**：支持学生请求生成资源，展示资源列表和资源详情。资源类型包括课程讲解文档、PPT、思维导图、练习题、拓展阅读材料和代码实操案例。支持资源预览和下载。

**练习与错题页**：支持题目作答和结果展示，展示错题列表和变式题推荐。题目由练习与案例 Agent 根据学生画像和薄弱点生成。

**学习效果分析报告页**：展示统计卡片（总学习时长、任务完成率、习题正确率）、掌握度柱状图、趋势图、薄弱点列表和下一步建议。页面数据来源是 report_view_model.json，不由前端直接读取原始事件文件。支持语音播报报告摘要。

**资源推送中心页**：展示今日推荐、阶段推荐、补强推荐和推送记录。推送由推荐 Agent 根据画像信号、路径信号、行为信号和评估信号综合决策。

**视频/动画播放页**：提供视频播放器和动画预览能力，展示知识点关联信息，支持学习笔记功能。视频和动画由外部 API 生成。

### 6.2 管理端页面

系统管理端包含 2 个页面。

**学生管理页**：展示学生列表，支持状态筛选和详情查看，提供操作入口。页面数据来源是 users_index.json 和学生个人目录下的状态文件。

**学情汇总大屏**：展示学生概况统计、知识掌握度分布、薄弱点排行、资源使用统计和学习趋势。大屏采用全屏布局，数据实时刷新，用于教师快速了解整体学习情况。

### 6.3 答疑页面

**智能辅导答疑页**：支持三种问答模式（知识点问答、分步提示、追问解释），支持语音输入和语音播报，回答内容采用 Markdown 渲染。答疑由答疑 Agent 独立完成，从知识库检索上下文后调用讯飞星火生成回答。

## 7. 资源生成策略

### 7.1 资源生成方式分类

系统资源按生成方式分为三类。

第一类是内容类资源，由讯飞星火直接生成，不要求美观，内容正确即可。包括课程讲解文档（生成 Markdown 格式）、分层练习题、拓展阅读材料和代码实操案例。

第二类是展示类资源，需要调用外部专业平台 API 生成，要求美观展示。包括 PPT（待调研具体平台，见《外部资源生成 API 待招清单》）和视频/动画（待调研具体平台）。

第三类是前端渲染类资源，后端只输出数据格式，由前端渲染展示。包括思维导图（后端输出 Markdown，前端使用 Mermaid 或 Markmap 渲染）。

### 7.2 外部 API 待招清单

需要后续调研并集成的外部资源生成平台，单独形成《外部资源生成 API 待招清单》文档。待招清单应包含以下内容：

PPT 生成平台调研，包括 Gamma、AiPPT、美图 AI PPT 等平台的 API 接入方式、是否支持 API 调用、使用价格和调用限制。

视频/动画生成平台调研，包括多模态视频生成工具的能力、接入方式和限制。

思维导图导出工具调研，包括是否支持 API 导出为图片格式。

## 8. 语音功能设计

### 8.1 讯飞语音集成

系统集成讯飞语音的双向能力。

ASR 语音识别用于将学生的语音输入转成文字，使用场景包括首次建档对话中的语音提问和智能辅导答疑中的语音提问。

TTS 语音合成用于将系统生成的文字内容转成语音播报，使用场景包括系统回答的语音播报、讲解内容的语音播报和学习效果分析报告的语音播报摘要。

### 8.2 支持语音的页面

首次建档对话页支持语音输入，学生可以用语音回答建档问题。

智能辅导答疑页支持语音输入和语音播报，形成完整的语音问答体验。

学习效果分析报告页支持语音播报，学生可以听取报告摘要。

## 9. 前端设计规范

### 9.1 设计风格

系统采用 Airtable 风格设计系统，具体规范详见 Design.md。设计风格的核心特征是白底搭配深海军蓝文字，以 Airtable 蓝色作为交互强调色，整体呈现专业、干净、企业级的视觉效果。

### 9.2 核心设计令牌

主文字色为深海军蓝 #181d26，用于所有正文和标题。

主强调色为 Airtable 蓝 #1b61c9，用于按钮、链接和交互高亮。

背景色为白色 #ffffff，用于页面和卡片背景。

浅底色为 #f8fafc，用于侧边栏和次级区域背景。

边框色为 #e0e2e6，用于卡片和输入框边框。

成功绿为 #006400，用于完成状态和正确提示。

弱文字色为 rgba(4,14,32,0.69)，用于辅助说明文字。

按钮圆角 12px，卡片圆角 16px 到 24px，间距基准 8px，阴影采用蓝色色调多层阴影。

### 9.3 字体

正文字体采用 Haas，回退到系统字体（-apple-system, system-ui, Segoe UI, Roboto, Inter）。展示字体采用 Haas Groot Disp，回退到 Haas。正文字号 16px 到 18px，正文字距 0.08px 到 0.28px。

## 10. 部署与运行

### 10.1 运行环境

系统部署在本地环境中运行。后端使用 Python uvicorn 启动 FastAPI 服务，前端使用 npm run dev 启动 Next.js 开发服务器。

### 10.2 知识库构建

系统提供知识库初始化脚本 init_knowledge_base.py，用于将三个 xlsx 知识库文件和 PDF 教材切片内容导入 ChromaDB 向量数据库。初始化完成后，系统即可开始提供知识检索服务。

## 11. 与初版方案的变更总结

后端框架从未明确变更为 FastAPI，原因是团队确认使用 Python 技术栈。

前端框架从未明确变更为 React + Next.js，原因是 AI Native 开发方式对 React 生态的支持最好。

Agent 框架从提及 LangChain 变更为 CrewAI，原因是 CrewAI 上手简单、角色定义清晰、与 FastAPI 集成顺畅。

大模型从未明确变更为讯飞星火，原因是赛题要求科大讯飞相关工具。

向量数据库从提及多种方案变更为 ChromaDB，原因是 1500 条规模本地零配置即可运行。

数据存储从纯 JSON 文件变更为 SQLite 加 JSON 和 JSONL 混合方案，原因是解决 JSON 文件的并发写入问题。

Agent 数量从 7 个（含任务理解 Agent）调整为 7 个（取消任务理解 Agent，增加答疑 Agent），原因是意图识别功能可集成到编排器中，而赛题要求的智能辅导功能需要专门的答疑 Agent。

页面数量从 6 个扩展为 11 个，原因是补全赛题要求的资源推送、视频播放、管理端、学情大屏和答疑页面。

知识库设计从缺失变更为本文档新增，原因是知识库是系统输入端，缺失则无法开始开发。

设计风格从未明确变更为 Airtable 风格，具体规范详见 Design.md。

部署方式从未明确变更为本地运行，原因是比赛演示场景需求。

学生模式从未明确变更为单学生演示，原因是先做核心功能，多学生扩展作为后续优化点。
