## 1. 结论摘要

当前仓库处于一个很明确的阶段：

- 前端已经是可运行的完整产品壳，主要页面基本齐全。
- 后端尚未开始落地，仓库中目前不存在实际的 `backend/` 代码目录。
- 产品、数据、后端、UI/UX 文档已经比较完整，文档密度很高。
- 前端当前完全依赖 `mock.ts` 供数，真实 API 尚未接入。
- 前端现有的“类型定义”已经天然形成了第一版联调契约，但它与后端设计文档之间存在多处不一致。

一句话判断：这是一个“设计和前端先行，后端待从 0 到 1 补齐”的项目，后续开发重点不是补页面，而是把后端、知识库和真实流式交互补上，并处理好“文档契约”和“前端现实”之间的差异。

## 5. 前端数据契约现状

### 5.1 当前唯一真实契约源

当前前端侧最实际的契约源不是后端文档，而是：

- `frontend/src/lib/api/types.ts`

它定义了页面目前真正依赖的数据形状。后续真实 API 接入时，至少要保证：

- 前端页面还能继续吃到这里定义的结构
- 如果后端返回的是另一套字段名，应在 `src/lib/api/real.ts` 中做转换，不要把转换逻辑散到页面

### 5.2 当前 API 层组织方式

`frontend/src/lib/api/index.ts` 当前把所有方法都指向 `mock.ts`：

- `getTodayTasks`
- `completeTask`
- `getDashboardStats`
- `getRecentResources`
- `getMasteryData`
- `getProfile`
- `getResources`
- `generateResource`
- `sendMessage`
- `getChatHistory`
- `getQuizQuestions`
- `getReport`
- `getRecommendations`
- `getLearningPath`
- `getVideos`
- `getDailyQuote`
- `getContributionData`

这意味着接入真实后端时，推荐策略很清楚：

1. 新建 `frontend/src/lib/api/real.ts`
2. 在 `index.ts` 中逐步把方法从 `mock` 切到 `real`
3. 页面层保持不动，最多只改少量不合理字段依赖

## 6. 前端页面真实依赖的接口能力

### 6.1 首页 `/`

依赖：

- 今日任务
- 统计概览
- 最近资源
- 学生画像
- 学习打卡热力图
- 掌握度数据

特点：

- 页面会并行拉取 6 组数据
- 任务完成状态目前是前端本地切换，没有调用 `completeTask`
- 热力图已经需要一整年的日粒度数据

后端含义：

- 首页不是一个“大而全聚合接口”，而是多个小接口并行
- 后端要优先保证这些基础读接口稳定

### 6.2 学习路径 `/path`

依赖：

- 当前阶段
- 阶段列表
- 嵌套知识树

特点：

- 允许多层 children
- 节点状态严格依赖 `completed | current | pending`

### 6.3 资源中心 `/resources`

依赖：

- 资源列表全量拉取
- 前端本地搜索
- 前端本地筛选

特点：

- 当前页面没有调用服务端搜索和分页
- 空状态会引导到 `/generate`

后端含义：

- 即便后端设计文档支持 `limit/offset/type`，前端第一阶段也能先吃全量
- 后续如果资源量增长，再切成服务端搜索/分页

### 6.4 练习 `/practice`

依赖：

- 拉取题目列表
- 前端本地判题
- 右侧准确率前端实时计算

这是当前最重要的契约冲突点之一：

- 当前 `QuizQuestion` 类型包含 `correctAnswer`
- 页面直接用 `correctAnswer` 在前端判题
- 但后端设计文档中，`GET /api/quiz` 返回题目时不应暴露正确答案

结论：

- 真实后端接入前，这个页面必须重构
- 推荐改为：
  - `GET /api/quiz` 只返回题面
  - `POST /api/quiz/submit` 返回判题结果
  - 页面在提交后更新 `results`

### 6.5 资源生成 `/generate`

依赖：

- 资源类型
- 文本 prompt
- 资源列表
- 生成进度
- 生成完成后的内容预览

当前状态：

- `api.generateResource()` 只返回一个初始资源对象
- 后续进度是页面内 `setInterval` 人工模拟

这与后端设计文档差异很大：

- 文档设计为 SSE 流式进度
- 当前前端并没有真正消费 SSE

结论：

- 这个页面后续需要接入真实流式协议
- 建议把进度流逻辑从页面内模拟迁到 `real.ts` 或一个专用 hook

### 6.6 智能辅导 `/tutor`

依赖：

- 初始历史消息
- 发送消息
- 助手回答

当前状态：

- 页面是“伪流式”
- 点击发送后等待 `Promise<Message>` 返回整条消息
- `useStreaming.ts` 存在，但页面没有实际使用

这与设计文档也存在明显落差：

- 文档是 SSE 流式返回
- 还包含 `mode`、`sources`、审核降级逻辑

结论：

- `/tutor` 需要从“整条消息响应”升级为“流式增量渲染”
- 现有 `useStreaming.ts` 不能直接用于 SSE，因为它只是拼接原始 chunk，没有事件级解析

### 6.7 报告 `/report`

依赖：

- 报告整体对象
- 顶部统计
- 日学习时长
- 时间分布
- 薄弱点
- AI 总结

当前状态：

- 页面上的“本周/本月”切换仅改变本地 UI 状态
- 并没有传入任何 period 参数重新请求数据

结论：

- 第一期后端可只实现“默认返回本周”
- 如果要支持本月，应补真实 query 参数或单独接口

### 6.8 个人信息 `/profile`

依赖：

- 读取画像
- 展示目标、偏好、认知风格、每日时长、实操能力

当前状态：

- “保存修改”按钮没有实际提交逻辑

结论：

- 后续应接上 `PUT /api/profile`
- 但要先明确前端编辑态如何提交数组字段

### 6.9 视频 `/video`

依赖：

- 视频列表
- 当前视频信息
- 实际 `url`

当前状态：

- 播放区域仍是占位 UI，没有真实播放器
- mock 中 `url` 指向 `/demo-video.mp4`，但仓库里没有该资源

结论：

- 这是一个半成品页面
- 后续至少需要：
  - 真实可播放 URL
  - 统一视频资源元数据
  - 决定视频是资源的一种，还是独立内容实体

### 6.10 登录 `/auth` 与建档 `/onboarding`

当前状态：

- `auth` 是纯前端假登录，提交后直接跳 `/onboarding`
- `onboarding` 是 5 步本地收集，不调用任何后端

这与后端设计文档中的“对话式建档”并不一致。

结论：

- 现有前端 onboarding 更像“静态问卷版”
- 后端文档里的画像建档是“session + 多轮对话 + SSE”
- 后续必须二选一：
  - 保留当前卡片式问卷，实现问卷建档 API
  - 或把前端 onboarding 改成对话式建档

## 7. 当前最关键的契约差异

这是后续开发中最需要优先处理的部分。

### 7.1 字段命名风格不一致

前端当前大量使用 camelCase：

- `createdAt`
- `weakPoints`
- `learningPreference`
- `knowledgeLevel`
- `dailyTime`

后端设计文档大量使用 snake_case：

- `created_at`
- `weak_points`
- `learning_preference`
- `knowledge_level`
- `daily_time`

建议规则：

- 后端内部与数据库层保持 snake_case
- 前端页面继续使用 camelCase
- 统一在 `frontend/src/lib/api/real.ts` 做映射

### 7.2 用户标识字段不一致

前端 `StudentProfile` 使用：

- `id`

后端设计文档使用：

- `user_id`

建议规则：

- 后端响应保留 `user_id`
- 前端适配层转换为 `id`

### 7.3 画像字段不一致

前端类型：

- `goals`

后端设计文档：

- `goal`

建议规则：

- 后端存储可保留 `goal`
- 前端适配层统一转成 `goals`

### 7.4 资源状态枚举不一致

前端当前仅识别：

- `generating`
- `completed`
- `failed`

后端文档还出现：

- `review_failed`
- `fallback`

建议规则：

- 前端第一阶段增加兼容：
  - `review_failed` 视为 `failed`
  - `fallback` 可视为 `completed` 但附带低可信标记

### 7.5 题目判题逻辑冲突

当前前端把 `correctAnswer` 暴露给浏览器。

这不符合真实考试/练习后端设计。

建议规则：

- 真实模式下题目接口不返回答案
- 练习页面改为服务端判题
- mock 模式可继续保留 `correctAnswer` 方便开发

### 7.6 流式协议尚未打通

后端文档设计：

- 画像建档 SSE
- 资源生成 SSE
- 答疑 SSE

前端现状：

- 没有任何一个页面真正完成 SSE 解析
- `useStreaming.ts` 只是“读字节流并拼接”

建议规则：

- 后续统一采用 SSE 文本事件格式
- 封装一个真正的 SSE parser hook，不在页面里手写 reader 解析

## 11. 我后续开发会遵循的规则

这是本次探索后形成的明确执行规则。

### 11.1 总体规则

- 以后端设计文档为主设计依据。
- 以前端 `api/types.ts` 为当前联调现实依据。
- 当两者冲突时，不直接改页面散点逻辑，优先通过 API 适配层消解。

### 11.2 前后端字段规则

- 后端内部保持 snake_case。
- 前端页面保持 camelCase。
- 所有字段转换只放在前端 `real.ts` 或共享转换层。

### 11.3 页面改动规则

- 非必要不重写页面。
- 只在以下场景允许页面层改造：
  - 题目判题从前端改为后端
  - 流式页面接入 SSE
  - onboarding 方案选型发生变化

### 11.4 API 接入规则

- 不让页面直接写真实 `fetch`。
- 所有真实请求统一经由 `frontend/src/lib/api/real.ts`。
- `index.ts` 负责 mock/real 的切换或逐项替换。

### 11.5 流式协议规则

- 统一走 SSE。
- 前端提供统一解析 hook。
- 页面只消费“事件对象”，不直接处理原始 chunk。

### 11.6 单用户规则

- 第一阶段默认按文档采用 `single_user` 跑通全链路。
- 在未引入真实认证前，不提前做复杂多用户体系。
- 但代码组织上要保留未来扩展 `user_id` 的入口。

## 14. 最终判断

这个项目的“难点”不在页面数量，而在三件事：
