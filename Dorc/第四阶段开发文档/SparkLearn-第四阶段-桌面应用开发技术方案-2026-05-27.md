# SparkLearn 第四阶段桌面应用开发技术方案

日期：2026-05-27
阶段：第四阶段
目标：以最快开发速度完成 SparkLearn 桌面化，保留现有前后端能力，并将本机 `nanobot` 纳入桌面应用统一托管。

## 1. 目标与范围

### 1.1 阶段目标

1. 提供可安装、可启动、可稳定运行的 Windows 桌面应用。
2. 复用现有 `frontend` 与 `backend`，避免重写业务逻辑。
3. 将 `nanobot` 以本机子进程方式纳入桌面应用生命周期管理。
4. 解决“飞书/网页容器无法直连本机 localhost”导致的配置与调用阻塞。

### 1.2 本阶段范围（In Scope）

1. Electron 桌面壳工程搭建。
2. Electron 主进程托管 `backend(8000)` 与 `nanobot(8900)`。
3. 桌面启动、健康检查、异常提示、退出回收。
4. 本地日志落盘与故障排查入口。

### 1.3 非本阶段范围（Out of Scope）

1. 全平台安装包（先做 Windows）。
2. 云端多租户配置中心完整建设（仅预留接口）。
3. 深度离线模型能力（仍依赖上游模型服务）。

## 2. 现状与约束

### 2.1 当前可复用资产

1. `backend` 已具备完整 API 体系。
2. `frontend` 已具备完整业务页面与交互。
3. `nanobot` 已可本机运行并通过 OpenAI-compatible API 对接。

### 2.2 已验证约束

1. 浏览器容器（如飞书内网页）不能可靠访问用户本机 `127.0.0.1`。
2. `nanobot` 必须本机部署才能满足当前方案落地。
3. Windows 下端口占用与孤儿进程是高频问题，必须由桌面壳统一进程治理。

## 3. 技术选型

### 3.1 选型结论

1. 桌面壳：`Electron`（优先开发速度）。
2. 前端：保留 Next.js，MVP 阶段可直接接入现有开发/生产服务。
3. 后端：保留 FastAPI + uvicorn。
4. nanobot：保留本机进程模式，由 Electron 统一托管。

### 3.2 选型理由

1. Electron 上手快，生态成熟，Windows 打包链路成熟。
2. 现有 Web 前端可直接复用，改造成本最低。
3. 易于实现进程管理、日志采集、健康探针与自动恢复。

## 4. 总体架构

```text
+----------------------- Electron App ------------------------+
| Main Process                                                |
|  - ProcessManager                                           |
|  - HealthChecker                                            |
|  - ConfigLoader                                             |
|  - LogManager                                               |
|       |                          |                          |
|       | spawn                    | spawn                    |
|       v                          v                          |
|  backend: uvicorn :8000     nanobot serve :8900            |
|       |                          |                          |
|       +----------- localhost API ----------+                |
|                                           v                 |
|                         Renderer (WebView / Frontend UI)    |
+-------------------------------------------------------------+
```

关键原则：

1. 前端只连本机 `backend`，不直接连 `nanobot`。
2. `backend` 统一调用 `nanobot`，对前端保持稳定接口。
3. Electron 对本机服务“可启动、可观测、可回收”。

## 5. 进程与端口管理方案

### 5.1 启动时序

1. Electron 启动。
2. 检查端口占用（`8000/8900`）。
3. 启动 `nanobot` 进程。
4. 健康检查 `http://127.0.0.1:8900/health`。
5. 启动 `backend` 进程。
6. 健康检查 `http://127.0.0.1:8000/health`。
7. 打开主窗口加载前端页面。

### 5.2 退出时序

1. Electron 触发退出。
2. 先停止 `backend`，再停止 `nanobot`。
3. 写入退出日志并释放句柄。

### 5.3 异常策略

1. 启动失败：弹窗显示具体失败点（端口占用/配置缺失/健康检查超时）。
2. 运行中进程退出：自动重试 1 次，失败后提示用户并给出日志路径。
3. 端口冲突：显示冲突 PID 与处理建议。

## 6. 配置管理方案

### 6.1 固定关键配置

1. `NANOBOT_PET_ENABLED`（按产品策略控制）。
2. `NANOBOT_API_BASE_URL=http://127.0.0.1:8900`
3. `NANOBOT_CONFIG_PATH` 指向项目/应用内专用配置，禁止依赖用户目录历史配置。
4. `NANOBOT_WORKSPACE` 指向应用受控路径。

### 6.2 配置来源优先级

1. Electron 侧用户配置（桌面设置页）。
2. 本地配置文件（应用目录）。
3. `.env` 默认值。

## 7. 安全与边界

1. 所有服务仅监听 `127.0.0.1`。
2. Electron 到 backend 请求增加本地会话 token（后续实现）。
3. 禁止前端直接暴露敏感模型 key。
4. 日志默认脱敏（key/token 打码）。

## 8. 开发里程碑（MVP）

### M1：桌面壳基础（1-2 天）

1. 创建 `desktop/` 工程。
2. 可启动 Electron 窗口并加载前端地址。

### M2：进程托管（2-3 天）

1. 主进程托管 `nanobot` + `backend`。
2. 健康检查与启动失败提示。
3. 退出回收与孤儿进程防护。

### M3：稳定性与日志（1-2 天）

1. 标准日志目录与日志查看入口。
2. 端口冲突提示与一键重试。

### M4：打包发布（2 天）

1. Windows 安装包（仅内部测试渠道）。
2. 安装后一键启动验证。

## 9. 目录规划

建议新增：

1. `desktop/main/`：Electron 主进程逻辑
2. `desktop/preload/`：安全桥接
3. `desktop/renderer/`：可选壳内辅助页面
4. `desktop/scripts/`：启动与清理脚本
5. `desktop/config/`：桌面端配置模板

## 10. 验收标准

1. 双击桌面应用后，60 秒内完成可用启动。
2. `backend` 与 `nanobot` 健康检查通过率 > 95%（本机环境）。
3. 关闭应用后无残留 `uvicorn/nanobot` 进程。
4. 前端核心页面可正常刷新与任务发起。

## 11. 风险与应对

1. 风险：端口冲突频发。  
应对：启动前检测端口并提供 PID 级提示。

2. 风险：用户环境 Python/依赖不一致。  
应对：MVP 先给环境检测器，后续引入运行时打包。

3. 风险：上游模型余额或可用性不稳定。  
应对：在 UI 明确区分“接入故障”和“上游余额/供应商故障”。
