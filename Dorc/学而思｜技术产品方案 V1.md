## 1. 文档定位

本文档用于指导“学而思”系统的方案设计与开发实现，服务对象是产品、算法、工程和比赛文档撰写成员。文档目标不是做市场汇报，而是回答 4 个问题：

第一，赛题要求是否全部被系统覆盖。

&#x20;第二，系统整体是怎么搭起来的。

&#x20;第三，多智能体、知识库、资源生成、路径规划是怎么跑起来的。

&#x20;第四，系统如何被测试、部署并满足比赛合规要求。

***

## 1. 项目概述与赛题映射

### 1.1 产品定义

学而思是一套面向高校学生的**个性化学习多智能体系统**。系统以课程知识库为底座，通过对话式建档构建学生画像，再由多智能体协作生成个性化学习资源、规划学习路径、推送学习内容，并通过答疑和评估形成持续优化闭环。这个定义与赛题“构建高等教育个性化学习资源体系、开发智能学习智能体系统、满足学生个性化与多模态学习需求”的目标一致。

### 1.2 技术目标

本系统的技术目标分为五项：

* &#x20;用自然语言对话替代表单，自动构建并动态更新学生画像。&#x20;

* &#x20;以多智能体协同机制完成至少 5 类个性化资源生成。&#x20;

* &#x20;基于画像、知识库和学习状态生成动态学习路径。&#x20;

* &#x20;提供智能辅导与学习评估，构成学习闭环。&#x20;

* &#x20;满足防幻觉、安全过滤、流式输出、进度追踪与稳定运行等非功能要求。&#x20;

## 2. 系统总体技术架构

### 2.1 总体架构分层

系统采用 8 层架构：

* **前端交互层**：负责聊天、路径、资源卡片、练习、报告等界面呈现。&#x20;

* **应用服务层**：负责会话管理、任务创建、用户状态读取、结果聚合。&#x20;

* **Agent 编排层**：负责识别任务类型、调度 Agent、管理串并行流程。&#x20;

* **模型能力层**：负责大模型推理、结构化抽取、内容生成、答疑解释。&#x20;

* **知识库与 RAG 层**：负责课程资料入库、检索召回、知识片段供给。&#x20;

* **画像与学习状态层**：负责学生画像、学习进度、错题、偏好、掌握度存储。&#x20;

* **安全与可信层**：负责事实校验、敏感内容过滤、低置信度兜底。&#x20;

* **日志与评估层**：负责链路日志、性能指标、质量评估、测试追踪。&#x20;

### 2.2 核心数据流

系统核心数据流是：

学生输入问题或学习目标

&#x20;→ 会话服务识别任务

&#x20;→ 编排器读取学生画像与学习状态

&#x20;→ 检索知识库相关片段

&#x20;→ 调度对应 Agent 执行

&#x20;→ 审核 Agent 进行校验与过滤

&#x20;→ 前端流式展示结果

&#x20;→ 用户行为与反馈回流

&#x20;→ 更新画像、路径和推荐策略

### 2.3 核心时序

系统至少要有两张图写进正式文档：

第一张是“**资源生成时序图**”：用户请求生成资源时，画像 Agent、规划 Agent、资源 Agent、审核 Agent 如何协同。
&#x20;第二张是“**学习闭环时序图**”：学生学习、做题、答疑、评估、路径调整如何形成循环。

## 3. 课程知识库构建与 RAG 设计

赛题要求参赛团队自行构造至少一门完整高校专业课程的初始知识库/文档集作为系统输入，因此知识库不是附属能力，而是系统底座。

### 3.1 知识库输入范围

以《Python 程序设计》为例，知识源包括：

* &#x20;课程大纲&#x20;

* &#x20;教材章节内容&#x20;

* &#x20;授课 PPT&#x20;

* &#x20;章节练习与答案&#x20;

* &#x20;上机实验说明&#x20;

* &#x20;代码示例&#x20;

* &#x20;项目案例&#x20;

* &#x20;拓展阅读资料&#x20;

### 3.2 数据处理流程

知识库处理链路分 5 步：

* &#x20;文档清洗：去重、去噪、统一格式。&#x20;

* &#x20;结构切分：按章节、知识点、题型、实验项目切分。&#x20;

* &#x20;标签增强：补充难度、前置知识、资源类型、适用阶段标签。&#x20;

* &#x20;建立索引：同时建立关键词索引与语义向量索引。&#x20;

* &#x20;建立知识关系：形成章节树、知识点依赖图和题目考点映射表。&#x20;

### 3.3 检索增强策略

RAG 采用三段式召回：

第一层是关键词检索，用于精确定位章节、题型和术语。

&#x20;第二层是语义检索，用于理解学生自然语言问题。

&#x20;第三层是标签过滤，用于根据画像控制难度、课程阶段和资源类型。

最终返回给 Agent 的不是原始文档，而是**结构化知识上下文包**，包括：

* &#x20;关联章节&#x20;

* &#x20;核心知识点&#x20;

* &#x20;示例片段&#x20;

* &#x20;难度等级&#x20;

* &#x20;推荐资源类型&#x20;

* &#x20;可信度分数&#x20;

### 3.4 检索失败兜底

当检索结果低相关、低置信或命中不足时，系统不直接自由发挥，而是采用三种兜底：

* &#x20;提示学生补充信息。&#x20;

* &#x20;降级为通用解释，不给出高确定性结论。&#x20;

* &#x20;触发审核 Agent 标记为“低可信内容”。&#x20;

这部分直接服务赛题要求的“防幻觉”机制。

## 4. 对话式学习画像系统设计

赛题要求通过自然语言对话抽取特征，构建不少于 6 个维度的动态学生画像，并支持画像随学随新。

### 4.1 画像维度设计

学而思建议采用 8 维画像：

* &#x20;基础身份：专业、年级、课程。&#x20;

* &#x20;学习目标：考试通过、成绩提升、项目实战、竞赛准备。&#x20;

* &#x20;知识基础：零基础、入门、中等、进阶。&#x20;

* &#x20;知识短板：薄弱章节、常错知识点、易错题型。&#x20;

* &#x20;学习偏好：文字、图解、视频、代码案例。&#x20;

* &#x20;认知风格：先理论后练习、先案例后总结。&#x20;

* &#x20;学习节奏：每日时长、冲刺型或长期型。&#x20;

* &#x20;实操能力：代码理解、调试能力、项目独立完成度。&#x20;

### 4.2 建档方式

首次建档不采用表单，而是采用 3 轮对话式采集：

第一轮采课程、目标和当前基础。

&#x20;第二轮采薄弱点、习惯和偏好。

&#x20;第三轮采时间安排和实操能力。

建档结束后，画像 Agent 生成结构化画像对象，并同步在前端展示可视化标签。

### 4.3 动态更新机制

画像不是静态字段，而是持续更新状态。更新触发包括：

* &#x20;学生修改目标&#x20;

* &#x20;章节学习完成&#x20;

* &#x20;题目错误率变化&#x20;

* &#x20;资源偏好点击变化&#x20;

* &#x20;答疑高频问题变化&#x20;

每次更新后，画像系统向路径规划系统和推荐系统广播新状态。

## 5. 多智能体协同与调度设计

赛题强调系统必须体现“多智能体架构设计”，并由不同角色的智能体协作完成资源生成。

### 5.1 Agent 角色定义

学而思建议配置 7 个核心 Agent：

* **画像 Agent**：负责抽取画像、更新画像、生成学生标签。&#x20;

* **任务理解 Agent**：判断用户意图是建档、学路径、生成资源、做题还是答疑。&#x20;

* **路径规划 Agent**：负责计算阶段目标、章节顺序、节点优先级。&#x20;

* **资源生成 Agent**：负责生成讲义、导图、阅读材料、PPT 结构稿。&#x20;

* **练习与案例 Agent**：负责习题、错题变式、代码案例、实验任务。&#x20;

* **评估 Agent**：负责掌握度评估、薄弱点发现、学习报告生成。&#x20;

* **审核 Agent**：负责事实校验、安全过滤、格式标准化与低置信拦截。&#x20;

### 5.2 编排器职责

主控编排器不是简单路由器，而是整套系统的执行中枢。它负责：

* &#x20;识别当前任务类型&#x20;

* &#x20;决定调用哪些 Agent&#x20;

* &#x20;控制串行还是并行&#x20;

* &#x20;分配上下文包&#x20;

* &#x20;合并中间结果&#x20;

* &#x20;管理任务状态&#x20;

* &#x20;处理重试与降级&#x20;

### 5.3 协同规则

资源生成类任务建议采用“串行 + 并行混合”：

* &#x20;先由画像 Agent 和任务理解 Agent 串行，明确学生状态与任务意图。&#x20;

* &#x20;再由路径规划 Agent 给出目标知识范围。&#x20;

* &#x20;随后资源生成 Agent 与练习案例 Agent 并行生成内容。&#x20;

* &#x20;最后由审核 Agent 统一校验输出。&#x20;

### 5.4 容错机制

技术文档里必须写清 4 个容错：

* &#x20;单 Agent 超时：启用降级模板或重试一次。&#x20;

* &#x20;检索为空：提示补充信息或切换通用模式。&#x20;

* &#x20;输出不合规：审核 Agent 回退重生成。&#x20;

* &#x20;多 Agent 结果冲突：优先保留高可信结果并标注来源。&#x20;

## 6. 个性化资源生成系统设计

赛题要求由不同角色 Agent 协作完成至少 5 种类型的个性化资源生成。

### 6.1 首期资源清单

建议学而思首期做 6 类资源：

* &#x20;课程讲解文档&#x20;

* &#x20;知识点思维导图&#x20;

* &#x20;分层练习题&#x20;

* &#x20;错题解析与变式题&#x20;

* &#x20;拓展阅读材料&#x20;

* &#x20;代码实操案例&#x20;

这样能覆盖“理论理解—练习巩固—实操落地”三段链路。

### 6.2 每类资源统一生成模板

每类资源都按统一模板写到技术文档里：

* &#x20;资源名称&#x20;

* &#x20;生成目标&#x20;

* &#x20;依赖画像字段&#x20;

* &#x20;依赖知识库数据&#x20;

* &#x20;调用 Agent 链路&#x20;

* &#x20;输出格式&#x20;

* &#x20;前端展示方式&#x20;

* &#x20;质量检查项&#x20;

例如“代码实操案例”的模板可以写成：

* &#x20;目标：让学生在具体场景下掌握章节知识的应用。&#x20;

* &#x20;画像依赖：知识基础、短板、实操能力。&#x20;

* &#x20;知识依赖：章节知识点、示例代码、实验要求。&#x20;

* &#x20;输出：问题背景、任务描述、代码示例、逐行注释、运行结果、常见错误。&#x20;

* &#x20;校验：是否可运行、是否覆盖指定知识点、是否符合难度等级。&#x20;

### 6.3 个性化控制参数

资源生成必须由画像驱动，而不是一套模板给所有人。控制参数至少包括：

* &#x20;难度等级&#x20;

* &#x20;讲解深度&#x20;

* &#x20;资源形式&#x20;

* &#x20;章节范围&#x20;

* &#x20;题目数量&#x20;

* &#x20;是否附代码示例&#x20;

* &#x20;是否生成图解版&#x20;

### 6.4 质量约束

资源输出必须经过 4 重约束：

* &#x20;结构完整。&#x20;

* &#x20;内容与课程知识库一致。&#x20;

* &#x20;难度符合画像。&#x20;

* &#x20;无敏感与事实错误。&#x20;

## 7. 个性化学习路径与资源推荐设计

赛题要求整合系统生成的个性化资源，结合学生专业、学习进度、知识掌握情况及学习偏好，生成科学动态路径，并基于画像进行精准推送。

### 7.1 路径规划逻辑

路径规划以“知识图谱 + 画像 + 掌握度”三要素驱动。

路径生成原则是：

先补前置基础，再进章节核心，再做练习巩固，最后完成实操任务。

&#x20;对于基础薄弱学生，优先走“解释—例题—基础练习”。

&#x20;对于进阶学生，优先走“总结—案例—综合任务”。

### 7.2 路径节点设计

每个路径节点包含：

* &#x20;节点名称&#x20;

* &#x20;所属章节&#x20;

* &#x20;学习目标&#x20;

* &#x20;推荐资源&#x20;

* &#x20;推荐练习&#x20;

* &#x20;掌握状态&#x20;

* &#x20;下一步动作&#x20;

### 7.3 动态调整

动态调整至少考虑 5 类触发条件：

* &#x20;连续答错同一知识点&#x20;

* &#x20;当前节点停留时间过长&#x20;

* &#x20;学习任务连续完成率高&#x20;

* &#x20;学习偏好明显变化&#x20;

* &#x20;阶段目标完成&#x20;

### 7.4 推荐策略

推荐系统同时考虑 4 个信号：

* &#x20;画像信号&#x20;

* &#x20;路径信号&#x20;

* &#x20;行为信号&#x20;

* &#x20;评估信号&#x20;

推荐位建议分为：

首页今日学习任务、章节补强推荐、错题补救推荐、阶段学习包推荐。

## 8. 智能辅导与学习评估设计

这两块属于可选加分项，但如果你们要冲高分，建议纳入方案。赛题明确把智能辅导和学习效果评估列为加分点，并强调评估结果要能反向调整推送与学习计划。

### 8.1 智能辅导

智能辅导要支持三种模式：

* &#x20;知识点问答&#x20;

* &#x20;题目不会做时的分步提示&#x20;

* &#x20;基于当前资源上下文的追问解释&#x20;

回答输出可支持：

* &#x20;精简提示版&#x20;

* &#x20;分步讲解版&#x20;

* &#x20;图解说明版&#x20;

* &#x20;案例演示版&#x20;

### 8.2 学习评估

评估系统至少跟踪：

* &#x20;学习时长&#x20;

* &#x20;资源完成率&#x20;

* &#x20;习题正确率&#x20;

* &#x20;错题重复率&#x20;

* &#x20;章节掌握度&#x20;

* &#x20;实操任务完成度&#x20;

### 8.3 评估结果输出

学习报告建议输出：

* &#x20;学习进度&#x20;

* &#x20;掌握度雷达图&#x20;

* &#x20;薄弱知识点&#x20;

* &#x20;常错题型&#x20;

* &#x20;下一阶段建议&#x20;

### 8.4 闭环优化

评估结果会反向作用于三个模块：

* &#x20;更新画像短板权重&#x20;

* &#x20;调整路径顺序&#x20;

* &#x20;调整推荐资源形式和难度&#x20;

## 9. 前端交互与非功能实现

赛题明确要求界面符合现代 AI 产品交互规范，包括流式输出、Markdown 渲染、多模态卡片化展示，同时要求防幻觉、内容安全过滤、合理响应时间与生成进度追踪。

### 9.1 前端页面最小闭环

为了满足比赛演示，前端至少保留 6 个页面：

* &#x20;首次建档对话页&#x20;

* &#x20;首页学习总览页&#x20;

* &#x20;学习路径页&#x20;

* &#x20;资源生成与资源详情页&#x20;

* &#x20;练习与错题页&#x20;

* &#x20;学习报告页&#x20;

* 9.2 交互机制

必须具备三种交互机制：

* &#x20;流式输出：边生成边展示，缩短等待感。&#x20;

* &#x20;结构化渲染：Markdown + 卡片化展示资源。&#x20;

* &#x20;长任务进度展示：显示“检索中、规划中、生成中、审核中”。&#x20;

### 9.3 防幻觉实现

防幻觉不要写成口号，要写成机制：

* &#x20;所有高风险内容优先走 RAG。&#x20;

* &#x20;输出受检索片段约束。&#x20;

* &#x20;低置信度结果自动提示。&#x20;

* &#x20;审核 Agent 对关键事实二次校验。&#x20;

### 9.4 内容安全机制

内容安全至少包含：

* &#x20;敏感词与违规内容过滤&#x20;

* &#x20;学术内容错误拦截&#x20;

* &#x20;输出格式规范检查&#x20;

* &#x20;不适宜内容拒答与重定向&#x20;

### 9.5 性能目标

正式文档里建议写成指标：

* &#x20;首 token 响应时间可控&#x20;

* &#x20;普通问答秒级返回&#x20;

* &#x20;长资源生成有状态反馈&#x20;

* &#x20;复杂任务支持进度追踪而非白屏等待&#x20;

## 10. 测试、部署与合规说明

赛题明确要求提交系统开发说明书、测试说明书、可运行源码、数据集、模型部署配置文件，并要求说明 AI Coding 工具使用情况；若使用开源项目和 AI 工具，也要标注名称、来源和协议要求。赛题还特别指出，开发过程中使用的其他 AI 辅助工具应选用科大讯飞相关工具。

### 10.1 测试方案

测试至少分 6 类：

* &#x20;功能测试：画像、资源生成、路径规划、推荐是否可用。&#x20;

* &#x20;协同测试：多 Agent 是否按时序正确协作。&#x20;
