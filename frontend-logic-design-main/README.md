# Frontend Logic Design

> 你的界面"感觉不对"但说不清哪里不对？这个 Skill 帮你系统性地找出来并修好它。

## 它解决什么问题

前端开发中最常被忽视的不是视觉层（好不好看），而是**信息架构层**（合不合理）：

- 为什么有的按钮能点，有的不能？
- 为什么同样是查看详情，有时弹侧栏，有时跳新页面？
- 为什么看板上有个"快速添加"输入框，但添加出来的东西毫无意义？
- 为什么用户找不到某个功能的入口？

这些问题的根因不是 CSS 写错了，而是**信息分类和交互层级缺少统一规则**。

## 它怎么工作

基于三大经典理论，提供一套 5 阶段诊断流程：

```
MECE 原则         → 保证信息分类不重不漏
渐进式披露        → 保证信息按深度递进展示
施耐德曼法则      → 概览优先，缩放筛选，按需详情
```

### 核心模型：四层信息深度

```
L0 概览  →  "什么情况？"      →  卡片/摘要      →  直接可见
L1 聚焦  →  "这条是什么？"    →  侧栏抽屉       →  点击条目
L2 管理  →  "让我找/改"       →  独立页面        →  点击"查看全部"
L3 执行  →  "帮我做这件事"    →  对话/向导       →  点击行动按钮
```

**核心规则**：同一页面内，所有同类组件在同一深度层的行为必须一致。

### 5 阶段诊断

1. **识别问题类型** — 分类不互斥？不穷尽？深度不一致？层级错位？入口缺失？
2. **信息深度审计** — 逐个元素标注属于 L0/L1/L2/L3 哪一层
3. **MECE 检查** — 互斥性 + 穷尽性验证
4. **一致性矩阵** — 表格化对比所有组件的交互行为是否统一
5. **生成修正方案** — 统一下钻规则 + 入口暗示 + 容器选择

## 使用方式

安装后，在 Claude Code 中对你的前端项目说：

```
/frontend-logic-design
```

或者直接描述你的困惑：

```
"我们的看板页面交互逻辑不一致，有的卡片能点进去看详情，有的不能"
"这个页面的信息层级设计不合理，用户不知道该去哪找信息"
"帮我审查一下这个模块的信息架构是否符合 MECE 原则"
```

## 真实案例

详见 [case-study.md](skills/frontend-logic-design/references/case-study.md) — 一次完整的看板交互一致性改造，从问题发现到修复验收的全过程。

### Before

- 7 个看板卡片，各自决定交互行为
- 有的能点击看详情，有的不能
- "快速添加任务"在看板上无意义
- "AI 执行"弹出空洞的表单
- 没有"查看全部"入口
- 用户困惑："为什么这里不能点？"

### After

- 统一的四层下钻规则，用户只需学一次
- 所有列表型卡片条目点击 → 侧栏（L1）
- 所有列表型卡片底部"查看全部 →" → 管理页面（L2）
- AI 执行 → 意图确认 → 跳转对话页面（L3）
- 审批按钮就地完成（L0 内操作）

## 适用场景

- 看板/仪表盘的卡片布局和交互设计
- 管理后台的导航结构和页面层级
- 任何"用户觉得不对劲但说不清"的前端设计问题
- 新功能上线前的信息架构审查
- 重构/改版时的一致性检查

## 不适用

- 纯视觉层问题（颜色、字体、间距）→ 用 design-review / polish
- 性能问题（加载慢、卡顿）→ 用 optimize
- 无障碍问题（读屏器、对比度）→ 用 audit
